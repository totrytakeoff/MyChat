## 4. WSS 性能分析

### 4.1 RTT 延迟（所有正常测试汇总）

| 场景 | RTT 均值 | p50 | p95 | p99 |
|------|---------|-----|-----|-----|
| 全部正常测试 | 48-60ms | 45-58ms | 72-89ms | 80-114ms |

> RTT 延迟非常稳定，与并发数和消息频率几乎无关。跨公网 RTT 基线约 30-50ms，  
> 网关处理开销仅增加 15-20ms，说明消息处理路径没有明显阻塞。

### 4.2 消息吞吐

| 场景 | 发送吞吐 | 接收吞吐 | 有效载荷 |
|------|---------|---------|---------|
| 100u × 1s | 98 msg/s | 197 msg/s | 20.7 KB/s |
| 200u × 1s | 197 msg/s | 393 msg/s | 38.9 KB/s |
| 200u × 500ms | 392 msg/s | 784 msg/s | 77.6 KB/s |

### 4.3 连接建链瓶颈

连接建立速度（SSL 握手）是主要瓶颈：
- 20 连接/秒速率下，200 用户连接建立耗时约 8-9 秒（均值）
- p95 连接时间高达 40 秒，说明 SSL 握手存在严重排队
- **发送端 4 个 io 线程的 SSL 客户端握手已到极限**

| 阶段 | VUs | 请求数 | 失败率 | 延迟中位数 | 说明 |

## 7. 崩溃分析

### 7.1 崩溃场景

- **触发条件**: 200 用户 × 250ms 间隔 WSS 压测
- **现象**: 网关进程意外退出，日志末尾显示：
  ```
  terminate called after throwing an instance of 'std::system_error'
    what(): Resource temporarily unavailable
  ```
- **恢复**: 需手动重启 `./bin/gateway_server`
- **可稳定复现**: 是

### 7.2 可能原因

1. **文件描述符耗尽**: ulimit -n 65535 已设置，但某处仍可能打满
2. **线程池耗尽**: 4 线程固定线程池在大量并发 SSL 读写下可能全部阻塞
3. **上游服务过载**: 后端 user/message 服务无法处理 770 msg/s 的 RPC 调用
4. **内存分配失败**: 某处动态分配在并发下 OOM

## 8. 总结

### 8.1 性能上限（实测）

| 指标 | 安全值 | 极限值 | 崩溃点 |
|------|--------|--------|--------|
| WSS 并发连接 | 200 | ~200 | - |
| WSS 消息吞吐 | 200 msg/s | 392 msg/s | 770 msg/s |
| WSS RTT | 50-60ms | 48ms | - |
| HTTP 吞吐 | 10 req/s | 15 req/s | - |
| HTTP 并发 VUs | 50 | 100 | 500 |

### 8.2 瓶颈定位

1. **SSL 握手速度** - 单点瓶颈，20 连接/秒为上限
2. **HTTP accept 队列** - 高并发下溢出，新连接被拒绝
3. **上游 RPC 调用** - 疑似同步阻塞，导致网关线程被占满
4. **服务器资源** - 远未充分利用（CPU <1%, 内存 <1GB）

### 8.3 优化建议

1. **SSL 会话复用**: 启用 TLS Session Resumption 减少握手开销
2. **HTTP 连接池化**: 使用 HTTP keep-alive 减少连接建立
3. **上游调用异步化**: 后端 RPC 改为异步非阻塞模式
4. **增加网关线程数**: 当前 4 线程可能不足以支撑高并发
5. **添加背压机制**: 当负载过高时主动拒绝或限流，避免崩溃
6. **修复崩溃**: 排查 200u × 250ms 下的 `std::system_error` 根本原因
