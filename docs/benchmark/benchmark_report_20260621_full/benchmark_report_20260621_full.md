## 6. 本轮核心问题

### 问题 1: WSS 连接建立能力不足

表现:

- 50 连接稳定。
- 100 连接开始部分失败。
- 200 连接大面积失败。
- Connect p95 长期卡在 15s。

可能原因:

1. TLS/WebSocket handshake 并发处理能力不足。
2. 发压端和云端跨公网建连长尾明显。
3. gateway accept 后 session 初始化/认证/注册链路存在排队。
4. 之前残留连接没有释放，影响后续测试。

### 问题 2: WSS 连接关闭清理不完整

表现:

- 发压机已经没有 `bench_ws` 进程。
- 云端 gateway 仍有 `98 ESTAB` 和 `21 CLOSE-WAIT`。
- 日志中出现多次:

```text
WebSocket read failed: stream truncated
WebSocket write failed: Operation canceled
```

判断:

- `CLOSE-WAIT` 通常表示对端已经关闭，但服务端应用层没有完全 close fd。
- `ESTAB` 残留可能是长连接没有心跳/idle timeout，或服务端没有及时感知对端异常退出。

这是下一轮最应该修的 gateway 侧问题。

### 问题 3: HTTP 高并发接入仍弱

表现:

- HTTP 总吞吐约 15.6 req/s。
- 失败率 27.18%。
- `/health` p95 接近 60s。

判断:

- `/health` 都超时，说明不是业务查询慢。
- 更像 HTTP accept/线程模型/连接排队/httplib 并发处理能力的问题。

## 7. 人话结论

这次完整压测说明:

1. gateway 不再一压就崩，这是好消息。
2. 已经连上的 WSS 消息链路延迟并不高，这也是好消息。
3. 但当前系统还不能说稳定支持 100~200 WebSocket 在线连接。
4. 最大问题已经从“消息处理崩溃”变成“连接建不上、连接关不干净”。
5. HTTP 也还没到稳定可压的状态，超过百级 VU 后大量超时。

当前项目状态应该定义为:

```text
功能 MVP 基本可用；
单轮中低并发消息链路稳定；
长时间连续压测和高并发接入仍未稳定。
```

---

## 8. 下一步建议

优先级从高到低:

1. 修 gateway WebSocket session 关闭清理:
   - read/write error 后确保从 connection manager 移除 session
   - close socket/fd
   - 清理 uid/session 映射
   - 避免 CLOSE-WAIT 堆积

2. 增加 WSS idle timeout / heartbeat timeout:
   - 客户端异常断开后不能无限 ESTAB
   - 长时间无读写应主动关闭

3. benchmark 工具继续完善:
   - 输出连接失败原因: tcp/ssl/ws/timeout
   - 每阶段结束后检查远端残留连接数
   - `run_all.py` 不应把远端日志误写到本机路径

4. 单独做连接建立基准:
   - `connect-rate=1/s, 5/s, 10/s, 20/s`
   - 分别跑 50/100/200 连接
   - 每轮前重启 gateway 或确保连接清零

5. HTTP 接入优化:
   - 继续确认 httplib 线程模型和 backlog 生效范围
   - `/health` 单接口独立压测
   - 检查 HTTP 连接 keep-alive、accept、工作线程数量
