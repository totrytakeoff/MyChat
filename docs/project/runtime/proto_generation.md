# Proto/gRPC 生成链路约束

本文档记录当前 MyChat 的 protobuf/gRPC 生成方式。后续修改 `common/proto/*.proto` 时，必须遵守这里的约束。

## 核心约束

- proto 源文件统一放在 `common/proto/`。
- checked-in 生成文件也统一放在 `common/proto/`。
- 不要直接调用系统 `protoc`。
- 不要手写或手改 `*.pb.*`、`*.grpc.pb.*`。
- 生成入口统一使用 CMake target：`generate_proto`。

原因是系统 `protoc` 可能和 vcpkg protobuf headers/runtime 不匹配。例如当前机器上系统 `protoc` 是 `34.1`，而项目 vcpkg protobuf 是 `5.29.3`。一旦混用，生成文件会在编译期出现 protobuf gencode/runtime 不兼容问题。

## 当前 CMake 行为

根 `CMakeLists.txt` 会优先从 active vcpkg installed tree 中选择：

- `tools/protobuf/protoc`
- `tools/grpc/grpc_cpp_plugin`

优先级以当前构建目录和 `VCPKG_INSTALLED_DIR` 为准，而不是盲目使用系统 PATH 中的工具。
如果 CMake 找不到 vcpkg 管理的 `protoc`，会直接 configure 失败；如果 `Protobuf_PROTOC_EXECUTABLE` 不在 `MYCHAT_ACTIVE_VCPKG_PREFIX` 下，也会失败。也就是说 proto 生成链路现在是硬约束，不再依赖开发者自觉避开系统 `protoc`。

`common/CMakeLists.txt` 中的 `im_proto` 使用顶层定义的 generated source 列表，并把这些源标记为 `GENERATED`。这样即使 `common/proto/*.pb.cc` 暂时不存在，CMake 也可以先完成 configure/generate，再由 `generate_proto` 产出。

## 标准命令

以 remote-push-odb 构建目录为例：

```bash
cmake -S . -B build/remote-push-odb
cmake --build build/remote-push-odb --target generate_proto -j2
cmake --build build/remote-push-odb --target gateway_server -j2
```

本轮已验证：

```bash
cmake --build build/remote-push-odb --target generate_proto -j2
cmake --build build/remote-push-odb --target gateway_server -j2
cmake --build build/remote-push-odb --target test_gateway_message_ws -j2
```

`ctest --test-dir build/remote-push-odb -R test_gateway_message_ws` 当前没有发现已注册测试条目；可执行目标已构建通过。

## 常见故障

如果看到类似错误：

```text
This file was generated by a newer version of protoc
PROTOBUF_NAMESPACE_ID does not name a type
PROTOBUF_NAMESPACE_OPEN does not name a type
```

优先检查：

1. 是否有人用系统 `protoc` 生成过 `common/proto/*.pb.*`。
2. CMake configure 日志中的 `Using vcpkg protoc:` 是否指向当前 active vcpkg installed tree。
3. 是否执行过 `cmake --build <build-dir> --target generate_proto`。

正确处理方式是重新 configure 并执行 `generate_proto`，不要手工修生成文件。
