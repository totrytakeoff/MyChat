## Problems To Fix Next

1. Generation is split: common protobuf files are generated by root
   `generate_proto`, while codec gRPC stubs live under `services/codec` and
   are not regenerated by CMake.
2. `codec_service.pb.*` exists under `common/proto` but is not part of
   `im::proto`.
3. `services/codec` duplicates `base.pb.*`, which risks duplicate descriptor
   and version drift when linked with `im::proto`.
4. gRPC is not declared in `vcpkg.json`, even though the local machine has it
   installed already. A fresh machine may not be able to enable codec service.
5. CMake does not expose the selected gRPC C++ plugin path as a first-class
   generation variable.
6. No deterministic command exists to regenerate both protobuf and gRPC outputs
   from one canonical source tree.

## Decision

Use `common/proto` as the single canonical proto source tree.

The next implementation task should first make generation deterministic and
testable, without implementing service-to-service RPC behavior yet.

Do not start standalone Push Service work until this generation chain is clean
enough that future service contracts can be regenerated without manual file
copying.

## Task 009 Scope

Goal:

```text
Create a deterministic CMake-driven protobuf/gRPC generation chain that uses
common/proto as the only proto source directory and eliminates stale duplicated
codec generated files from the active build path.
```

Recommended implementation sequence:

1. Add a `grpc` vcpkg feature or dependency plan for `MYCHAT_BUILD_CODEC_SERVICE`.
   Keep default builds unchanged.
2. Add CMake discovery for `gRPC::grpc_cpp_plugin` or the vcpkg
   `grpc_cpp_plugin` executable.
3. Replace root `generate_proto` with explicit generated output lists or split
   targets:
   - `generate_common_proto`
   - `generate_codec_grpc`
   - optional aggregate `generate_proto`
4. Generate `codec_service.grpc.pb.*` from `common/proto/codec_service.proto`.
5. Stop compiling duplicated `services/codec/base.pb.*`.
6. Decide the generated output location before implementation:
   - preferred: keep all generated proto/gRPC outputs in `common/proto` for now;
   - acceptable alternative: generated tree under `generated/proto`, but this is
     a larger include-path and checked-in artifact change.
7. Make `im_codec_service` consume generated files from the selected canonical
   output path.
8. Add a minimal build verification path for `MYCHAT_BUILD_CODEC_SERVICE=ON`.

## Acceptance Criteria

- `cmake --build <build> --target generate_proto` regenerates all active
  protobuf and codec gRPC generated files from `common/proto`.
- No active target compiles duplicated `services/codec/base.pb.*`.
- `MYCHAT_BUILD_CODEC_SERVICE=ON` builds `im_codec_service` from canonical
  generated files.
- Existing ODB-enabled baseline remains 14/14 passing.
- Existing no-ODB baseline remains 2/2 passing.
- Default `MYCHAT_BUILD_CODEC_SERVICE=OFF` behavior remains unchanged.
- No standalone Push Service behavior is implemented in this task.

## Out Of Scope

- Implementing a running CodecService server.
- Introducing Gateway-to-service gRPC calls.
- Refactoring Friend/Group/Message business logic.
- Changing WebSocket protobuf request/response contracts.
- Changing `ProtobufCodec` wire format.
- Regenerating ODB files.
- Starting standalone Push Service implementation.

## Risks

- Regenerating `.pb.*` with the shell `protoc` 34.1 would produce files
  incompatible with the vcpkg Protobuf 5.29.3 runtime used by CMake. Always use
  the CMake-selected `Protobuf_PROTOC_EXECUTABLE`.
- Moving generated outputs out of `common/proto` would touch many include paths.
  Do that only if the plan explicitly chooses a generated output directory.
- Linking both `common/proto/base.pb.*` and `services/codec/base.pb.*` into the
  same binary can create duplicated protobuf descriptors. The cleanup should
  remove the duplicate from active targets.
