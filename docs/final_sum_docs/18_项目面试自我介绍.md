# MyChat 项目面试自我介绍

日期：2026-06-23

## 1. 2 分钟版本（推荐）

适用于面试开场“请简单介绍一下你做的项目”。

```text
我做了一个 C++ 分布式即时通讯后端项目，叫 MyChat。

它的核心架构是 Gateway 统一接入 HTTP 和 WebSocket，后端按 User、Message、
Friend、Group、Push 拆分服务边界。Gateway 不直接处理业务逻辑，而是负责协议接入、
鉴权、连接管理、cmd 路由和 packet 转发。HTTP 和 WebSocket 请求都会先进入
MessageParser，转换为统一的 UnifiedMessage，再通过 MessageProcessor 进入
GatewayCommandHandlerRegistry，最后由 GatewayRuntimeRegistry 选择本地 service
packet dispatcher 或远程 gRPC ForwardPacket。

功能上，当前项目已经实现注册登录、个人资料、好友申请、单聊、群聊、在线推送和离线补偿。
消息链路采用先持久化再推送：Message 或 Group 服务先把消息落到 PostgreSQL，
持久化成功后返回 ack，Gateway 再把服务侧返回的 push_event 交给 Push 服务做在线投递。
Push 是 best-effort，失败不回滚消息，用户后续可以通过历史消息或离线拉取补偿。

比较有特点的是 Gateway 重构后的 packet boundary。Gateway 只看 envelope 和 cmd_id，
不解析业务 payload；真正的业务 protobuf 解析在各个服务自己的 packet dispatcher 中完成。
这样 HTTP、WebSocket、本地调用和远程 gRPC 都能走同一套请求模型，新增服务时主要是注册
cmd_id 到服务 endpoint，而不是再写一套 controller 和 client facade。

存储上，Redis 负责 token、session 等临时状态，PostgreSQL/ODB 负责用户、消息、好友、
群组等持久数据。我还做了一个 React Web 客户端，用两个浏览器窗口真实验证加好友、单聊、
群聊和实时推送。最近也完成了一轮压测和网关链路修复：重构后 HTTP 500 VU ramp
失败率为 0，p95 约 3.75ms；WebSocket 200 用户每 250ms 一条消息时吞吐约
714 msg/s，RTT p95 约 35.91ms。
```

## 4. 分模块展开脚本

### 4.1 为什么做 Gateway

```text
Gateway 解决的是客户端接入复杂度。Web 或未来 Qt/Electron 客户端只需要连接 Gateway，
不需要知道 User、Message、Friend、Group、Push 分别部署在哪里。Gateway 统一处理
token、HTTP/WS 协议、WebSocket session 和 cmd 路由，后端服务只暴露统一 packet 边界。
```

### 4.2 当前消息处理链路

```text
HTTP / WebSocket
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> GatewayCommandHandlerRegistry
-> GatewayRuntimeRegistry
-> local service packet dispatcher 或 remote ForwardPacket gRPC
-> service 解析业务 payload 并执行业务逻辑
-> 返回 ServicePacketResult
-> Gateway 统一响应或交给 Push 做在线投递
```

这个链路的重点是：Gateway 不解析业务 payload，不构造业务 DTO，不做好友、群成员、
消息状态这类业务判断。

### 4.3 消息链路怎么保证不丢

```text
MyChat 不承诺绝对的“消息永不丢失”，但 MVP 范围内做了可靠性优先的取舍：

1. 发送消息先进入 Message/Group 服务持久化。
2. 持久化成功后才返回发送 ack。
3. 在线 Push 失败不回滚消息。
4. 接收方可以通过历史消息或离线消息拉取补偿。
```

### 4.4 Push 回调的具体原理

```text
Gateway 持有 WebSocket 连接，Push 服务在远程进程中不能直接访问 Gateway 内存。
因此 Push 通过 GatewayPushDeliveryService 回调 Gateway：
ListUserSessions 查询用户在线连接，SendSessionPayload 向指定 session 发送 payload，
MarkMessageDelivered 在成功投递后标记 delivered。
```

### 4.5 local/remote 如何切换

```text
当前不要再用“local/remote client facade”解释主链路。更准确的说法是：
GatewayRuntimeRegistry 按服务配置和 cmd 路由选择 endpoint。
local 模式下 endpoint 调用服务进程内的 packet dispatcher；
remote 模式下 endpoint 调用对应服务的 ForwardPacket gRPC。
外部 HTTP/WS 契约不变，变化只发生在 Gateway 到服务之间的 runtime endpoint。
```

### 4.6 当前不足和后续规划

```text
当前是完整可演示的 MVP，但不是生产级 IM：
- 没有富媒体、撤回、编辑、全文搜索。
- 没有多 Gateway 实例下的全局连接路由。
- 没有生产级对象存储、部署编排和监控告警。
- 群管理还缺管理员、踢人、禁言、审核等完整能力。

下一步优先级是：补齐面试讲解文档、继续容量压测、完善多机器部署验证、
设计多 Gateway 连接路由和消息序列号机制。
```

## 5. 面试表达原则

- 先讲业务目标，再讲架构设计。
- 先讲 Gateway 统一 packet 主链路，再讲具体服务。
- 不再把旧 controller/client facade 当作当前架构亮点。
- 主动说明 MVP 边界和生产级扩展方向。
- 有压测数据时只说当前基准，不夸大成生产级容量。
