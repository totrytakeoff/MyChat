# MyChat 项目面试自我介绍

日期：2026-06-17

## 1. 2 分钟版本（推荐）

适用于面试开场"请简单介绍一下你做的项目"。

```text
我最近做了一个 C++ 分布式即时通讯后端项目，叫 MyChat。

它的核心架构是 Gateway 作为统一接入层，同时处理 HTTP 和 WebSocket。
Gateway 后面是五个按业务拆分的服务：User、Message、Friend、Group 和 Push。
Gateway 对外屏蔽了后端部署方式——它可以通过 local facade 直接调用进程内的服务，
也可以通过 remote facade 调用 gRPC 远程服务，切换只改配置。

功能上，当前项目已经完整实现了注册登录、好友管理、单聊、群聊和在线推送。
消息链路的设计是先持久化再推送——服务端把消息落库后返回 ack，然后通过 Push 服务
做在线投递。Push 是 best-effort，失败后可以通过离线拉取补偿。

比较特别的是 Push 服务的架构：它拆成了 PushNotifier、PushRuntime 和 FanoutPolicy
三层。远程模式下 Push 处于独立进程，但 WebSocket 连接归 Gateway 持有，
所以 Push 必须通过 Gateway callback 来查找用户 session、发送消息和标记已投递。

存储上，Redis 负责 token 和 session 等临时状态，PostgreSQL 负责用户、消息、
好友、群组等持久数据。我还做了一个 React Web 客户端，可以用两个浏览器窗口
模拟两个用户，真实走完加好友、聊天、群聊的完整链路。

最近我也补了一轮压测和性能修复。HTTP 侧 500 VU 阶梯压测下 46230 次请求失败率为 0，
p95 约 5.81ms；WebSocket 侧 200 连接建连失败为 0，200 用户每 500ms 一条消息时
RTT p95 约 21.92ms。当前瓶颈出现在 200 用户每 250ms 一条消息的极限场景，
说明消息处理、持久化或 push 链路还需要进一步分段观测。
```

## 4. 分模块展开脚本

如果面试官在 2 分钟简介后追问细节，不要从头再讲一遍，而是按下面的模块展开：

### 4.1 为什么做 Gateway

```text
Gateway 解决的第一个问题是客户端接入的统一性。如果没有 Gateway，
Web 客户端需要知道 User、Message、Friend、Group、Push 五个服务地址，
还要处理 token、session、WebSocket 连接管理。Gateway 把这些问题收拢了。
```

### 4.2 消息链路怎么保证不丢

```text
MyChat 不承诺"消息绝不丢失"这种绝对的语义，但在 MVP 范围内做到了：

1. WebSocket 发送时服务端先持久化到 PostgreSQL，持久化成功后返回 ack。
2. 服务端收到消息时不会因为推送失败而丢弃消息。
3. 在线推送失败后，用户可以通过历史消息查询或离线消息拉取补偿。

这是先持久化再推送的基本含义。
```

### 4.3 Push 回调的具体原理

```text
Gateway 持有 WebSocket 连接，Push 服务在远程时不能直接访问 Gateway 进程内 session。
所以 Push 通过 gRPC 调用 GatewayPushDeliveryService 的三个方法：
ListUserSessions（查用户有哪些连接）、SendSessionPayload（向指定 session 发消息）、
MarkMessageDelivered（标记已投递）。
这三步本质上是把"在线投递"决策和执行分开——PushRuntime 决定要发给谁，
Gateway 负责实际发送。
```

### 4.4 local/remote 切换怎么做到的

```text
每个服务定义一个 Client 接口（比如 UserClient），然后分别做本地和远程实现。
LocalUserClient 直接调进程内 UserService，RemoteUserClient 通过 gRPC 调用。
Gateway 启动时根据配置 user.mode = local/remote 决定注入哪个实现。
Controller 只依赖 UserClient 接口，不感知具体实现。
```

### 4.5 当前项目的不足和后续规划

```text
当前是完整可用的 MVP，但离生产级还有一段距离：
- 没有富媒体消息和消息撤回编辑。
- 没有多 Gateway 实例下的全局连接路由。
- 没有生产级部署编排；当前已有基础压测，但还缺完整容量规划和监控告警。
- 群管理只有创建和加入，没有踢人、禁言、群主转让。

下一步优先级是：消息序列号增强顺序语义、多 Gateway 连接路由、头像对象存储、
WebSocket 极限场景分段观测和容量规划。
```

## 5. 面试表达原则（总结）

- 先说业务目标，再说架构设计。
- 先说核心链路，再说技术细节。
- 先说已经实现，再说后续优化。
- 遇到不足不要回避，说明当前 MVP 边界和下一步方案。
- "微服务"措辞要能衔接后续话术，例如"当前通过静态配置文件管理服务端点，后续可引入 etcd/Consul 实现服务发现"。
