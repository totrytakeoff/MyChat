# MyChat 简历项目描述与亮点

日期：2026-06-17

## 1. 简历项目描述（推荐版本）

### 中文版

```text
MyChat - 基于C++20的分布式即时通讯后端系统  2025.06 - 至今

项目介绍：基于 C++20 的分布式 IM 后端，Gateway + 微服务架构。
Gateway 统一接入 HTTP/WebSocket，后端按 User/Message/Friend/Group/Push
拆分为独立微服务，通过 local/remote facade 切换本地调用与 gRPC 远程调用。
已实现注册登录、好友、单聊、群聊及实时推送，配套 React 客户端端到端验证。

技术栈：C++20 Boost.Asio/Beast Protobuf gRPC PostgreSQL/ODB Redis JWT CMake vcpkg Docker GoogleTest

技术亮点：
• Gateway + local/remote facade：5 个服务共享统一接入层，外部协议不随部署方式变化
• Protobuf 帧协议（varint + CRC32），信封解码仅解析路由头；手写 C++20 协程与调度器
• JWT 双 Token（6 端独立策略），Redis 实现分布式 Token 吊销与自动轮换
• Redis 多端 WebSocket 会话（同平台挤号、跨平台在线）；Push 通过 gRPC callback 投递
• 多 io_context 线程池 Round-Robin 分发；泛型 ConnectionPool<T> 超时借出与溢出创建
```

### 英文版

```text
MyChat - C++20 Distributed Instant Messaging Backend  2025.06 - Present

A C++20 distributed IM backend with Gateway + microservice architecture.
Gateway unifies HTTP/WebSocket access, backend split into User/Message/
Friend/Group/Push microservices, local/remote facade for switching between
in-process calls and gRPC. Covers registration, friends, direct/group chat,
and real-time push with a React client for E2E validation.

Tech Stack: C++20 Boost.Asio/Beast Protobuf gRPC PostgreSQL/ODB Redis JWT CMake vcpkg Docker GoogleTest

Highlights:
• Gateway + local/remote facade: unified access layer, external protocol unchanged regardless of deployment mode
• Protobuf binary frame protocol (varint + CRC32), envelope decoding; hand-written C++20 coroutine Task<T> and scheduler
• JWT dual Token (6 platforms), Redis-based distributed token revocation and rotation
• Redis multi-device session management (same-platform kick, cross-platform online); Push via gRPC callback for remote session delivery
• Multi io_context thread pool with atomic Round-Robin; generic ConnectionPool<T> with timeout borrow and overflow creation
```

## 2. 简历 Bullet Points（可选用）

推荐 4-6 条，每条聚焦一个独立的技术亮点：

### 2.1 Gateway 统一接入

```text
• 设计 Gateway 作为 IM 系统统一接入层，同时承载 HTTP 和 WebSocket，集中处理 token 鉴权、
  连接管理和服务编排，对外部客户端屏蔽后端微服务部署细节。
```

### 2.2 local/remote 服务边界

```text
• 采用 Client Facade 模式实现 User/Message/Friend/Group/Push 五个微服务的 local/remote 双模式：
  local 直接调用进程内服务，remote 通过 gRPC 调用独立进程，外部 HTTP/WS 契约不随部署方式变化。
```

### 2.3 消息可靠性与在线推送

```text
• 消息链路采用"先持久化再推送"策略，服务端持久化成功即向客户端返回 ack，
  在线推送作为 best-effort 加速路径，推送失败不回滚消息，通过离线拉取和历史查询补偿。
```

### 2.4 Push 分布式回调

```text
• 将 Push 拆为 PushNotifier/PushRuntime/FanoutPolicy 三层，
  远程模式下 Push 服务通过 GatewayPushDeliveryService 回调 Gateway 完成 WebSocket session 查询、
  payload 发送和 delivered 标记，解决 WebSocket 连接归属权与流程解耦的矛盾。
```

### 2.5 存储分层设计

```text
• Redis 承载 refresh token、撤销状态和 session 信息等临时高频数据（利用 TTL 自动过期），
  PostgreSQL + ODB 承载用户、消息、好友、群组等持久业务数据，
  两类存储生命周期不同，职责分离明确。
```

### 2.6 Web 端到端验证

```text
• 配套 React + TypeScript Web 客户端，双账号多窗口验证注册登录、好友申请、单聊实时消息、
  群聊 fanout 推送、离线消息拉取等完整 IM 业务流程，后端能力均通过真实 UI 操作验证。
```

## 3. 项目亮点分层

### 3.1 架构设计亮点

| 亮点 | 说明 | 面试价值 |
| --- | --- | --- |
| Gateway 统一接入 | HTTP + WebSocket 共用接入层，集中鉴权和连接管理 | 高频考点，体现协议分层思维 |
| local/remote facade | 同接口支持本地调用和 gRPC 远程调用，部署方式可切换 | 体现工程演进思维，不是一上来就拆分布式 |
| 服务按业务拆分 | User/Message/Friend/Group/Push 边界清晰 | 体现领域驱动设计意识 |
| Push 回调机制 | 远程 Push 通过 Gateway callback 操作 WebSocket session | 分布式边界经典取舍，适合展开深聊 |

### 3.2 消息系统设计亮点

| 亮点 | 说明 | 面试价值 |
| --- | --- | --- |
| 先持久化再推送 | 发送成功 = 服务端落库，推送是 best-effort | 消息可靠性的核心取舍 |
| 离线消息补偿 | 在线推送失败后拉取离线消息兜底 | 体现容错设计 |
| ack 不等于已读 | 发送 ack 只表示服务端收到，阅读状态需单独标记 | 语义准确 |
| FanoutPolicy 抽象 | AllSessions / PlatformFilter / NewestSession 三种策略 | 体现策略模式应用 |

### 3.3 分布式与工程化亮点

| 亮点 | 说明 | 面试价值 |
| --- | --- | --- |
| gRPC 服务边界 | 5 个服务通过 proto 定义契约，独立进程可部署 | 分布式基础能力 |
| Proto 统一消息结构 | WebSocket 二进制消息和 gRPC 接口共用 proto | 体现接口先行的工程习惯 |
| Redis + PostgreSQL 分治 | 临时状态与持久数据分开存储 | 存储选型的基本判断力 |
| 单机 remote-all 脚本 | 本地一台机器启动 6 个进程模拟分布式部署 | 工程验证能力 |

## 4. 荣誉与数据（如果需要量化）

以下数据是当前 MVP 的参考值，不是生产压测结果：

```text
服务规模：Gateway + 5 个后端服务（User/Message/Friend/Group/Push）
核心接口：~25 个 HTTP 接口 + 3 个 WebSocket 命令 + 20 个 gRPC RPC
数据表：6 张 PostgreSQL 表（user/message/friend/group/group_member/group_message）
存储：Redis token 状态 + PostgreSQL 持久数据
构建依赖：C++20 / Boost / Protobuf / gRPC / hiredis / httplib / jwt-cpp / ODB
验证：React Web 客户端 + 单元测试 + 双账号多窗口手工测试
压测：HTTP 500 VU ramp，46230 请求，失败率 0%，p95 5.81ms
实时链路：200 WSS 连接失败 0；200 用户 / 500ms 消息 RTT p95 21.92ms
```

性能数据使用口径：

- 这些数据来自 2026-06-22 云服务器完整复测，不是生产 SLA。
- 可以写“完成压力测试和性能瓶颈定位”，不要写“支持百万连接”。
- 可以写“200 在线用户、0.5-1s 消息间隔下 WebSocket RTT p95 约 12-22ms”。
- 需要主动说明极限区：200 用户 / 250ms 间隔时 RTT p95 上升到 2628.05ms，说明消息处理链路存在吞吐拐点。

## 5. 不适合简历写的内容

不要写、或者面试中不要主动提：

- 不要笼统写"高并发"——可以写具体压测数据，不要泛泛而谈。
- 不要写"生产级"——当前是完整 MVP，不是生产部署。
- 不要夸大 Web 客户端——它的定位是验证工具，不是产品级前端。
- 不要写"支持万人群"——没有做群规模压测。

## 6. 简历中如何面试开场

简历上的项目描述只是第一眼印象，面试时通常会让你"简单介绍一下这个项目"。

这时建议不要复述简历文字，而是换一个结构：

```text
1. 一句话：MyChat 是什么（分布式 IM 后端）
2. 架构框架：Gateway 接入 + 服务边界 + 存储
3. 一条链路：比如注册登录怎么做
4. 一个亮点：比如为什么 Push 要回调 Gateway
5. 项目状态：完整 MVP，正在持续工程化
```
