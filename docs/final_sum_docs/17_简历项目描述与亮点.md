# 简历项目描述与亮点

日期：2026-06-23

## 1. 推荐简历描述

```text
MyChat - 基于 C++20 的分布式即时通讯后端系统

基于 C++20 实现的分布式 IM 后端，Gateway 统一接入 HTTP/WebSocket，
内部通过 MessageParser、MessageProcessor 和 command handler registry
按 cmd_id 分发统一 packet。后端按 User/Message/Friend/Group/Push 拆分服务边界，
支持本地 in-process 运行和 gRPC ForwardPacket 远程调用，覆盖注册登录、好友、
单聊、群聊、在线推送和离线补偿，并配套 React Web 客户端完成端到端验证。
```

技术栈：

```text
C++20 / Boost.Asio / Boost.Beast / Protobuf / gRPC / PostgreSQL / ODB /
Redis / JWT / OpenSSL / CMake / GoogleTest / React / TypeScript
```

## 2. 推荐 Bullet Points

- 设计 Gateway 统一接入层，同时承载 HTTP 和 WebSocket，通过 `MessageParser -> UnifiedMessage -> MessageProcessor -> GatewayCommandHandlerRegistry` 将不同协议收敛为统一命令分发模型。
- 重构 Gateway 与业务服务边界，Gateway 只负责鉴权、路由、连接管理和 packet 转发，业务 payload 解析下沉到各 service packet dispatcher，避免接入层耦合业务逻辑。
- 基于 gRPC `ForwardPacket` 实现 User/Message/Friend/Group 服务远程边界，同一套 Gateway handler 支持 local service 和 remote service adapter 两种运行模式。
- 消息链路采用“先持久化、再 ack、再 best-effort push”策略，在线推送失败不回滚消息，通过历史查询和离线拉取补偿。
- 设计 PushNotifier/PushRuntime/FanoutPolicy，并通过 GatewayPushDeliveryService callback 解决远程 Push 无法直接访问 Gateway WebSocket session 的分布式边界问题。
- 使用 Redis 管理 token/session 等临时状态，PostgreSQL/ODB 持久化用户、消息、好友、群组等业务数据，并配套 React Web 客户端验证双账号好友、单聊、群聊和在线推送。

## 3. 可量化数据

重构后压测基准：

```text
HTTP ramp：46296 请求，平均 306.86 req/s，失败率 0.00%，p95 3.75ms
WSS 建连：conn-200 全成功，Connect p95 14ms
WSS 消息：200 users / 250ms，714.4 msg/s，RTT p95 35.91ms，错误 0
```

使用口径：

- 可以写“完成 Gateway 重构后完整压测和性能对比分析”。
- 不要写“百万连接”或“生产级高并发”。
- 明确这是 MVP 稳定性基准，不是商业 SLA。

## 4. 项目亮点分层

架构亮点：

- Gateway 统一 packet 架构。
- local/remote 运行模式共享 packet contract。
- gRPC 服务边界和 remote-all 多进程验证。
- Push remote callback 解决连接归属问题。

消息亮点：

- 先持久化再 push。
- ack / delivered / read 语义区分。
- online push + offline/history 补偿。
- FanoutPolicy 多端策略抽象。

工程亮点：

- C++20 多模块工程。
- Protobuf/gRPC 生成链路。
- PostgreSQL/ODB 持久化。
- Redis token/session 状态。
- GoogleTest/CTest 和 benchmark 压测。
- Web 客户端端到端验证。

## 5. 不建议写的内容

- 不要把旧 `HTTP controller + Local/RemoteClient` 写成当前架构。
- 不要写生产级、百万连接、万人群。
- 不要夸大富媒体、群权限、服务发现、监控告警等未完成能力。
- 不要把 Web 客户端包装成商业产品，它的定位是后端验证客户端。
