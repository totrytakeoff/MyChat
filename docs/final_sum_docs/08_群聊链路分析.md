# MyChat 群聊链路分析

日期：2026-06-16

## 1. 文档目标

这份文档分析 MyChat 群聊相关能力：

- 建群、搜群、加群、退群、群资料、群成员。
- 群消息发送和历史查询。
- 群消息在线 fanout。
- Group Service 与 GroupMessageService 的职责边界。
- 当前 MVP 和生产级群管理之间的差距。

一句话概括：

```text
MyChat 的群聊 MVP 把群资料、群成员、群消息分开建模，
通过 Group Service 管群和成员，通过 GroupMessageService 管群消息，
Gateway 负责鉴权和 HTTP 接入，Push 负责群消息在线 fanout。
```

## 2. 模块职责

| 模块 | 职责 |
| --- | --- |
| Web Client | 搜群、群资料页、加入/退出、成员展示、群聊消息 |
| Gateway Group HTTP | 群资料、成员、加入/退出等 HTTP 接口 |
| Gateway Group Message HTTP | 群消息发送和历史接口 |
| GroupClient facade | 屏蔽 local/remote Group Service 差异 |
| Group Service | 群资料、群成员、成员权限校验 |
| GroupMessageService | 群消息持久化和历史查询 |
| GroupRepository | 群资料和群成员 ODB 访问 |
| GroupMessageRepository | 群消息 ODB 访问 |
| User Service | 校验用户存在、解析成员昵称 |
| PushNotifier | 对群成员做在线 fanout |
| PostgreSQL | 群资料、群成员、群消息持久化 |

## 4. 建群链路

接口：

```text
POST /api/v1/groups
```

请求核心字段：

```json
{
  "name": "测试群"
}
```

服务端链路：

```text
Web Client
-> Gateway GroupHttpController::handle_create_group
-> verify_access_token
-> name 参数校验
-> creator_uid = token_user.user_id
-> GroupClient::create_group
-> GroupService::create_group
-> 校验群名非空
-> 校验创建者 uid 非空
-> User Service 校验创建者存在
-> 创建 Group
-> 创建 GroupMember OWNER
-> GroupRepository::create_group_with_owner
-> PostgreSQL/ODB transaction
-> 返回 group_id
```

关键点：

- 创建者来自 token uid。
- 建群时同时创建群主成员关系。
- owner 不能后续直接退群，除非转让群主，这属于后续扩展。

## 5. 搜群与加群链路

### 5.1 搜群

接口：

```text
GET /api/v1/groups/search?keyword=<keyword>
```

链路：

```text
Gateway 验证 token
-> GroupClient::search_groups
-> GroupService::search_groups
-> GroupRepository::search_groups
-> 计算 member_count
-> 返回群资料列表
```

用户体验上，应先展示搜索结果和群资料，再让用户点击加入。

### 5.2 加群

接口：

```text
POST /api/v1/groups/join
```

请求核心字段：

```json
{
  "group_id": 1
}
```

链路：

```text
Gateway verify_access_token
-> user_uid = token_user.user_id
-> GroupClient::join_group(group_id, user_uid)
-> GroupService::join_group
-> 校验群存在
-> 校验用户存在
-> 校验不是已有成员
-> GroupRepository::add_member
-> 返回 Joined group
```

失败场景：

| 场景 | 错误 |
| --- | --- |
| group_id 为空 | 400 |
| 群不存在 | GROUP_NOT_FOUND |
| 用户不存在 | USER_NOT_FOUND |
| 已经是成员 | ALREADY_MEMBER |

## 6. 群资料和成员链路

### 6.1 群资料

接口：

```text
GET /api/v1/groups/info?group_id=<id>
```

返回：

- group。
- joined。
- members。

当前 `joined` 的判定来自：

```text
list_members(group_id, current_uid) 是否为空
```

### 6.2 群成员

接口：

```text
GET /api/v1/groups/members?group_id=<id>
```

链路：

```text
Gateway verify_access_token
-> GroupClient::list_members(group_id, caller_uid)
-> GroupService::list_members
-> 校验群存在
-> 校验 caller 是成员
-> 查询所有成员
-> resolve_nickname
-> 返回成员列表
```

关键点：

```text
只有群成员可以查看群成员列表。
```

## 7. 退群链路

接口：

```text
POST /api/v1/groups/leave
```

链路：

```text
Gateway verify_access_token
-> user_uid = token_user.user_id
-> GroupClient::leave_group(group_id, user_uid)
-> GroupService::leave_group
-> 校验群存在
-> 校验当前用户是成员
-> 如果是 OWNER，拒绝
-> GroupRepository::remove_member
```

当前规则：

- 非成员不能退群。
- 群主不能直接退出，需要先转让所有权。
- 转让群主能力是后续扩展。

## 8. 群消息发送链路

接口：

```text
POST /api/v1/groups/messages/send
```

请求核心字段：

```json
{
  "group_id": 1,
  "content": "hello group"
}
```

服务端链路：

```text
Gateway GroupMessageHttpController::handle_send_message
-> verify_access_token
-> 校验 group_id/content
-> group_exists(group_id)
-> list_members(group_id, sender_uid)
-> 如果成员列表为空，拒绝
-> GroupClient::send_message
-> GroupMessageService::send_message
-> 校验群存在
-> 校验发送者是成员
-> GroupMessageRepository::create
-> PostgreSQL/ODB persist
-> Gateway 遍历成员
-> 对除发送者外的成员 PushNotifier::notify_user
-> 返回 msg_id
```

关键点：

- 发送者来自 token uid。
- 群消息先持久化，再 fanout。
- 默认不向发送者重复推送。
- 当前群消息通过 HTTP 发送，WebSocket 主要用于接收 push。

## 9. 群消息历史链路

接口：

```text
GET /api/v1/groups/messages/history?group_id=<id>&before=<ts>&limit=<n>
```

链路：

```text
Gateway verify_access_token
-> 校验 group_id
-> group_exists
-> list_members(group_id, current_uid)
-> 非成员拒绝
-> GroupClient::get_history(group_id, current_uid, before_time, limit)
-> GroupMessageService::get_history
-> GroupMessageRepository::find_by_group
-> resolve_nickname(sender_uid)
-> 返回 messages
```

当前语义：

- 只有成员可以查看群消息历史。
- 使用 `before` 和 `limit` 做分页。
- 返回 sender_uid、sender_nickname、content、created_at。

## 10. 群消息 fanout

群消息 fanout 发生在 Gateway：

```text
members = group_client_->list_members(group_id, sender_uid)
for member in members:
  if member.user_uid != sender_uid:
    PushContext {
      sender_uid,
      conversation_type = "group",
      conversation_id = group_id
    }
    push_notifier_->notify_user(member.user_uid, msg_id, content, context)
```

当前是简单 fanout：

- 遍历成员。
- 逐个通知 Push。
- Push 再根据用户在线 session 选择实际投递目标。

适合 MVP 和小群验证。

大群生产优化方向：

- 异步 fanout 队列。
- 分批投递。
- 在线成员过滤提前下沉。
- 群消息扩散策略按群规模选择。
- 推拉结合。

## 11. local/remote Group Service

### 11.1 local 模式

```text
Gateway GroupHttpController / GroupMessageHttpController
-> LocalGroupClient
-> GroupService / GroupMessageService in-process
-> Repository
```

### 11.2 remote 模式

```text
Gateway
-> RemoteGroupClient
-> im.group.GroupService gRPC
-> group_server
-> GroupGrpcService
-> GroupService / GroupMessageService
-> Repository
```

GroupClient 把群资料、群成员和群消息都统一封装在同一个 Gateway facade 后面。

## 12. 当前优势

- 群资料、成员、消息分离建模。
- 建群时自动建立 owner 成员关系。
- 成员列表和群消息历史都校验成员身份。
- 群消息先持久化，再在线 fanout。
- 支持 Group Service 远程 gRPC 边界。
- Web 客户端可以验证搜群、加群、群资料、群成员、群聊。

## 13. 当前不足和后续优化

当前不足：

- 没有入群申请审核，当前是直接加入。
- 没有邀请入群。
- 没有踢人、禁言、管理员。
- 群公告、群头像、群备注、通知设置等 proto/UI 有规划，但后端 MVP 未完整产品化。
- 群消息没有会话级 seq。
- 大群 fanout 还不是生产级。
- owner 不能退出但还没有转让群主流程。

后续优化：

- 群主/管理员权限体系。
- 入群申请和邀请。
- 踢人、禁言、公告。
- 群设置持久化。
- 大群 fanout 异步化。
- 群消息 seq 和增量同步。
- 群成员变更通知。

## 14. 面试问答准备

### 14.1 群和群成员为什么分开建模

回答思路：

```text
群资料描述群本身，群成员描述用户和群之间的关系，两者生命周期不同。
成员还有 role、joined_at 等关系属性，所以不能只把成员塞到群资料字段里。
```

### 14.2 为什么发送群消息前要校验成员身份

回答思路：

```text
群消息只能由群成员发送。这个校验必须在服务端做，不能依赖前端是否显示发送入口。
```

### 14.3 群消息为什么不推送给发送者

回答思路：

```text
发送者已经通过 HTTP 返回拿到发送结果，重复 Push 给自己会造成前端重复处理。
当前默认只 fanout 给除发送者外的成员。
```

### 14.4 大群 fanout 怎么优化

回答思路：

```text
当前 MVP 是遍历成员逐个 Push，适合小群验证。
生产大群需要异步队列、分批投递、在线成员过滤、推拉结合，不能让 Gateway 同步阻塞在大群 fanout 上。
```

## 15. 一句话总结

```text
MyChat 的群聊链路已经完成建群、搜群、加群、退群、成员、群消息和群在线 fanout 的 MVP 闭环；
核心设计是群资料、群成员、群消息分离建模，并在服务端统一做成员身份校验。
```

### B. 建群按代码读

Gateway：

```text
GroupHttpController::handle_create_group
-> verify_access_token
-> json::parse
-> name 校验
-> CreateGroupRequest request
   name = body.name
   creator_uid = user_info.user_id
-> group_client_->create_group(request)
```

Service：

```text
GroupService::create_group
-> name 非空
-> creator_uid 非空
-> user_service_->get_profile_by_uid(creator_uid) 校验创建者存在
-> Group g(name, creator_uid, t)
-> GroupMember owner(0, creator_uid, OWNER, t)
-> repo_->create_group_with_owner(g, owner)
```

关键代码点：

- 创建者来自 token uid。
- 建群和创建 OWNER 成员关系在同一个 repository 操作中完成。
- 群主身份不是前端标记，而是成员表 role。

### C. 加群/退群按代码读

加群：

```text
GroupHttpController::handle_join_group
-> verify_access_token
-> group_id 校验
-> group_client_->join_group(group_id, user_info.user_id, now_ms())
-> GroupService::join_group
-> repo_->find_group_by_id
-> user_service_->get_profile_by_uid(user_uid)
-> repo_->find_member(group_id, user_uid)
-> repo_->add_member(GroupMember(..., MEMBER, ...))
```

退群：

```text
GroupHttpController::handle_leave_group
-> verify_access_token
-> group_client_->leave_group(group_id, user_info.user_id)
-> GroupService::leave_group
-> repo_->find_group_by_id
-> repo_->find_member
-> member->role() == OWNER -> OWNER_CANNOT_LEAVE
-> repo_->remove_member
```

这两段代码体现：

- 成员关系由服务端维护。
- 群主不能直接退出。
- 重复加入会被 `ALREADY_MEMBER` 拦截。

### D. 群资料/成员按代码读

群资料：

```text
GroupHttpController::handle_group_info
-> verify_access_token
-> group_client_->get_group_info(group_id)
-> group_client_->list_members(group_id, user_info.user_id)
-> joined = !members.empty()
-> 返回 group + joined + members
```

成员列表：

```text
GroupService::list_members
-> repo_->find_group_by_id
-> repo_->find_member(group_id, caller_uid)
-> 非成员直接返回空
-> repo_->find_members(group_id)
-> resolve_nickname
```

代码边界：

- 当前用“成员列表是否为空”判断当前用户是否已加入。
- 非成员无法查看成员列表，返回空列表，再由 Gateway 映射成 joined=false 或 403。

### E. 群消息按代码读

Gateway：

```text
GroupMessageHttpController::handle_send_message
-> verify_access_token
-> group_id/content 校验
-> group_client_->group_exists(group_id)
-> members = group_client_->list_members(group_id, user_info.user_id)
-> members.empty() -> 403
-> group_client_->send_message(group_id, user_info.user_id, content, send_time)
-> 遍历 members，跳过发送者
-> push_notifier_->notify_user(member.user_uid, msg_id, content, context)
```

Service：

```text
GroupMessageService::send_message
-> group_id 非 0
-> sender_uid 非空
-> group_service_->group_exists
-> group_service_->list_members(group_id, sender_uid)
-> members.empty() -> FORBIDDEN
-> GroupMessage msg(group_id, sender_uid, content, now_ms)
-> repo_->create(msg)
```

关键代码点：

- Gateway 和 service 都做成员校验，属于 defense in depth。
- 群消息先落库，再 fanout。
- 群 fanout 跳过发送者。

### F. remote Group 按代码读

```text
RemoteGroupClient::create_group/join_group/leave_group/...
-> 构造 im.group.*Request
-> header.from_uid = 当前用户
-> apply_deadline
-> stub->*
-> 检查 grpc status
-> 检查 BaseResponse
-> 转 GroupInfoDTO / MemberInfoDTO / GroupMessageDTO
```

`group.proto` 中有不少未来字段：

- avatar。
- notice。
- mute。
- invite/kick/apply。
- admin/owner transfer。

当前 MVP 不要把这些 proto 字段等同于完整后端能力。文档和面试中应明确“已实现 RPC”与“预留协议结构”的区别。
