# 群聊链路分析

日期：2026-06-23

## 1. 文档目标

这份文档按当前统一 packet 架构分析 MyChat 群聊能力：

- 建群、搜群、加群、退群、群资料、群成员。
- 群消息发送和历史查询。
- 群消息在线 fanout。
- GroupService 与 GroupMessageService 的职责边界。
- 当前 MVP 和生产级群管理之间的差距。

当前权威链路：

```text
HTTP
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> group_command_handlers.cpp
-> GroupPacketRequest
-> local GroupPacketDispatcher 或 remote GroupService::ForwardPacket
-> GroupPacketResponse
-> optional push_events
```

一句话概括：

```text
MyChat 群聊 MVP 把群资料、群成员、群消息分开建模；
Gateway 只做鉴权、可信 uid 注入、packet 转发和 push_events 转交，
群资料解析、成员校验、群消息 fanout 成员计算都在 Group service packet dispatcher 中完成。
```

## 2. 模块职责

| 模块 | 当前职责 |
| --- | --- |
| Web Client | 搜群、群资料页、加入/退出、成员展示、群聊消息 |
| Gateway | HTTP catch-all、route -> cmd_id、token 校验、packet 转发、push_events 转交 |
| Group command handler | 构造 `GroupPacketRequest`，选择 local dispatcher 或 remote adapter |
| GroupPacketDispatcher | 解析 JSON payload，调用 GroupService/GroupMessageService，构造响应和 push_events |
| GroupService | 群资料、群成员、成员权限校验 |
| GroupMessageService | 群消息持久化和历史查询 |
| GroupRepository | 群资料、群成员 ODB 访问 |
| GroupMessageRepository | 群消息 ODB 访问 |
| UserService | 校验用户存在、解析成员昵称 |
| PushNotifier | 消费 group service 返回的 push_events，逐成员在线投递 |
| PostgreSQL | 群资料、群成员、群消息持久化 |

## 4. 建群链路

请求：

```text
POST /api/v1/groups

{
  "name": "测试群"
}
```

当前链路：

```text
HTTP -> MessageParser -> CMD_CREATE_GROUP
-> group_command_handlers.cpp
-> verify_http_user
-> packet.header.from_uid = token uid
-> GroupPacketDispatcher::handle_create_group
-> 解析 name
-> CreateGroupRequest.creator_uid = packet.header.from_uid
-> GroupService::create_group
-> 校验群名非空
-> 校验创建者存在
-> 创建 Group
-> 创建 GroupMember OWNER
-> GroupRepository::create_group_with_owner
```

关键点：

- 创建者来自 token uid。
- Gateway 不解析 `name`。
- 建群时同时创建群主成员关系。
- 群主不能直接退群，转让群主属于后续扩展。

## 5. 搜群与加群

搜群：

```text
GET /api/v1/groups/search?keyword=<keyword>
-> CMD_SEARCH_GROUP
-> GroupPacketDispatcher::handle_search_group
-> GroupService::search_groups
-> 返回 groups + joined 状态
```

加群：

```text
POST /api/v1/groups/join

{
  "group_id": 1
}

-> CMD_APPLY_JOIN_GROUP
-> GroupPacketDispatcher::handle_join_group
-> GroupService::join_group(group_id, token_uid)
```

失败场景：

| 场景 | 错误 |
| --- | --- |
| group_id 为空 | 400 |
| 群不存在 | GROUP_NOT_FOUND |
| 用户不存在 | USER_NOT_FOUND |
| 已经是成员 | ALREADY_MEMBER |

## 6. 群资料、我的群、成员列表

群资料：

```text
GET /api/v1/groups/info?group_id=<id>
-> CMD_GET_GROUP_INFO
-> GroupPacketDispatcher::handle_group_info
-> GroupService::get_group_info
-> GroupService::list_members
-> 返回 group + joined + members
```

我的群：

```text
GET /api/v1/groups
-> CMD_GET_GROUP_LIST
-> GroupPacketDispatcher::handle_group_list
-> GroupService::list_my_groups(token_uid)
```

群成员：

```text
GET /api/v1/groups/members?group_id=<id>
-> CMD_GET_GROUP_MEMBERS
-> GroupPacketDispatcher::handle_group_members
-> GroupService::list_members(group_id, token_uid)
```

当前规则：

- 查看成员需要调用者是群成员。
- 群资料会返回 joined 状态。
- 成员昵称由 service 侧解析。

## 7. 退群链路

请求：

```text
POST /api/v1/groups/leave

{
  "group_id": 1
}
```

链路：

```text
CMD_QUIT_GROUP
-> GroupPacketDispatcher::handle_leave_group
-> GroupService::leave_group(group_id, token_uid)
-> 校验群存在
-> 校验当前用户是成员
-> 如果是 OWNER，拒绝
-> GroupRepository::remove_member
```

当前规则：

- 非成员不能退群。
- 群主不能直接退出。
- 转让群主是后续扩展。

## 8. 群消息发送

请求：

```text
POST /api/v1/groups/messages/send

{
  "group_id": 1,
  "content": "hello group"
}
```

当前链路：

```text
CMD_SEND_GROUP_MESSAGE
-> group_command_handlers.cpp
-> verify_http_user
-> packet.header.from_uid = token uid
-> GroupPacketDispatcher::handle_send_group_message
-> 解析 group_id/content
-> GroupService::list_members(group_id, sender_uid)
-> 校验 sender 是群成员
-> GroupMessageService::send_message
-> GroupMessageRepository::create
-> fill_push_events(members, sender_uid, group_id, msg_id, content)
-> GroupPacketResponse(push_events)
-> Gateway notify_group_push_events
-> PushNotifier::notify_user(each receiver)
```

关键边界：

- Gateway 不解析 group_id/content。
- Gateway 不计算群成员 fanout。
- GroupPacketDispatcher 负责生成 `push_events`。
- Gateway 只遍历 service 返回的 `push_events` 并转交 PushNotifier。
- 默认不向发送者重复推送。

## 9. 群消息历史

```text
GET /api/v1/groups/messages/history?group_id=<id>&before_time=<ms>&limit=<n>
-> CMD_GET_GROUP_MESSAGES
-> GroupPacketDispatcher::handle_group_history
-> GroupService::list_members(group_id, token_uid)
-> 校验调用者是成员
-> GroupMessageService::get_history
```

当前语义：

- 只有群成员可以查群消息历史。
- `before/before_time/since` 和 `limit` 由 GroupPacketDispatcher 解析。
- Gateway 不解析这些业务查询字段。

## 10. Local / Remote 差异

local 模式：

```text
group_command_handlers.cpp
-> GroupPacketDispatcher dispatcher(group_service, group_message_service)
-> dispatcher.handle(packet)
```

remote 模式：

```text
group_command_handlers.cpp
-> remote_group_adapter->forward(packet)
-> GroupService::ForwardPacket
-> group_server
-> GroupPacketDispatcher::handle(packet)
```

local 和 remote 共享同一 packet 契约。remote 模式下群资料、群成员、群消息和 push_events 都由远端 group service 生成后返回 Gateway。

## 13. 面试追问

### Q: 群消息 fanout 为什么不在 Gateway 计算？

群成员关系是 Group service 的业务数据。Gateway 如果自己查询成员并计算 fanout，就会重新耦合群业务规则。当前做法是 Group service 返回 `push_events`，Gateway 只负责投递转交。

### Q: 为什么群消息不走 WebSocket 发送？

当前 MVP 群消息主要通过 HTTP 验证后端链路和 push fanout。WebSocket 群发送可以作为后续扩展，但底层仍应复用 `GroupPacketRequest/GroupPacketDispatcher`，不能让 Gateway 解析群消息业务字段。

### Q: 当前群聊和生产级群聊差距在哪？

生产级还需要群权限体系、管理员、入群审核、禁言、踢人、群公告、群消息 sequence、大群 fanout 优化、跨 Gateway 路由和更完整的回执体系。
