# 好友关系链路分析

日期：2026-06-23

## 1. 文档目标

这份文档按当前统一 packet 架构分析 MyChat 好友系统：

- 用户搜索和好友申请如何衔接。
- 好友申请、接受、拒绝、好友列表、待处理申请的链路。
- Friend service 如何校验自加好友、重复申请、目标用户存在性。
- local service 与 remote `ForwardPacket` 如何保持同一语义。
- 当前 MVP 与生产级社交关系系统的边界。

当前权威链路：

```text
HTTP
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> friend_command_handlers.cpp
-> FriendPacketRequest
-> local FriendPacketDispatcher 或 remote FriendService::ForwardPacket
-> FriendPacketResponse
```

一句话概括：

```text
MyChat 好友系统以“申请记录 + 状态流转”为核心模型；
Gateway 只负责鉴权、可信 uid 注入和 packet 转发，
好友业务字段解析和状态机在 Friend service packet dispatcher / FriendService 中完成。
```

## 2. 模块职责

| 模块 | 当前职责 |
| --- | --- |
| Web Client | 搜索用户、展示资料卡、发起好友申请、处理待处理申请、展示好友列表 |
| Gateway | HTTP catch-all、route -> cmd_id、access token 校验、packet 转发 |
| Friend command handler | 构造 `FriendPacketRequest`，选择 local dispatcher 或 remote adapter |
| FriendPacketDispatcher | 解析 JSON payload，调用 FriendService，构造 JSON response |
| FriendService | 好友申请状态机、关系校验、好友列表、待处理申请 |
| FriendRepository | PostgreSQL/ODB 好友关系表读写 |
| UserService | FriendService 用于校验目标用户存在、解析昵称 |
| PostgreSQL | 好友申请和关系状态持久化 |

## 3. 对外接口

用户搜索由 User 链路提供：

```text
GET /api/v1/users/search?keyword=<keyword>
```

好友接口：

```text
POST /api/v1/friends/request
POST /api/v1/friends/respond
GET  /api/v1/friends
GET  /api/v1/friends/pending
```

Friend packet gRPC 边界：

```proto
rpc ForwardPacket(FriendPacketRequest) returns (FriendPacketResponse);
```

## 4. 搜索用户到好友申请

正常 IM 体验应该是：

```text
搜索账号/昵称
-> 展示用户资料卡
-> 用户确认后点击添加好友
-> 发送好友申请
```

搜索链路：

```text
GET /api/v1/users/search
-> CMD_SEARCH_USER
-> user_command_handlers.cpp
-> UserPacketDispatcher::handle_search_user
-> UserService::search_profiles
-> 返回用户资料列表
```

申请链路：

```text
POST /api/v1/friends/request
-> CMD_ADD_FRIEND
-> friend_command_handlers.cpp
-> verify_http_user
-> packet.header.from_uid = token uid
-> FriendPacketDispatcher::handle_add_friend
-> 解析 target_uid
-> FriendService::send_request
-> FriendRepository::create(PENDING)
```

边界点：

- Gateway 不解析 `target_uid`。
- Gateway 不信任客户端传 requester。
- 申请人来自 token uid，写入 packet header。
- `target_uid` 在 FriendPacketDispatcher 中从 JSON payload 解析。

## 5. 好友申请状态机

当前状态：

```text
PENDING -> ACCEPTED
PENDING -> REJECTED
```

当前不允许：

- 非 pending 状态重复处理。
- 不是申请目标用户的人处理申请。
- 对同一用户重复发起关系。
- 添加自己为好友。

状态含义：

| 状态 | 含义 |
| --- | --- |
| PENDING | 申请已发出，等待目标用户处理 |
| ACCEPTED | 目标用户已接受，双方成为好友 |
| REJECTED | 目标用户已拒绝，申请结束 |

## 6. 处理好友申请

请求：

```text
POST /api/v1/friends/respond

{
  "friend_id": 123,
  "accept": true
}
```

当前链路：

```text
HTTP -> MessageParser -> CMD_HANDLE_FRIEND_REQUEST
-> friend_command_handlers.cpp
-> verify_http_user
-> packet.header.from_uid = token uid
-> FriendPacketDispatcher::handle_friend_response
-> 解析 friend_id / accept
-> FriendService::respond_to_request(friend_id, current_uid, accept)
-> FriendRepository::find_by_id
-> 校验 current_uid == target_uid
-> 校验 status == PENDING
-> update_status(ACCEPTED 或 REJECTED)
```

关键安全点：

```text
只有申请目标用户可以处理申请。
```

## 7. 好友列表与待处理申请

好友列表：

```text
GET /api/v1/friends
-> CMD_GET_FRIEND_LIST
-> FriendPacketDispatcher::handle_friend_list
-> FriendService::get_friends(packet.header.from_uid)
```

待处理申请：

```text
GET /api/v1/friends/pending
-> CMD_GET_FRIEND_REQUESTS
-> FriendPacketDispatcher::handle_pending_friends
-> FriendService::get_pending_requests(packet.header.from_uid)
```

返回字段当前包括：

- friend_id。
- friend_uid。
- nickname。
- status。
- created_at。

## 8. 关键业务规则

### 8.1 不能添加自己

```text
requester_uid == target_uid -> SELF_REQUEST
```

### 8.2 目标用户必须存在

FriendService 会通过 UserService 校验目标用户：

```text
user_service_->get_profile_by_uid(target_uid)
```

不存在时返回 `TARGET_NOT_FOUND`。

### 8.3 重复关系拒绝

FriendRepository 会检查两人之间是否已有关系：

```text
repo_->find_relationship(requester_uid, target_uid)
```

已有关系时返回 `ALREADY_EXISTS`。

### 8.4 只有目标用户能处理申请

```text
friend_record.target_uid == current_uid
```

否则返回 `FORBIDDEN`。

## 9. Local / Remote 差异

local 模式：

```text
friend_command_handlers.cpp
-> FriendPacketDispatcher dispatcher(friend_service)
-> dispatcher.handle(packet)
```

remote 模式：

```text
friend_command_handlers.cpp
-> remote_friend_adapter->forward(packet)
-> FriendService::ForwardPacket
-> friend_server
-> FriendPacketDispatcher::handle(packet)
```

两者共享同一个 `FriendPacketRequest/FriendPacketResponse` 契约，业务解析和状态机都复用 FriendPacketDispatcher/FriendService。

## 11. 面试追问

### Q: 为什么好友申请前要先搜索？

因为正常 IM 不应该让用户直接输入内部 uid。用户先通过账号/昵称搜索看到资料卡，再发起申请，体验更接近真实产品；后端上搜索走 User service，好友关系状态机走 Friend service，职责也更清晰。

### Q: Gateway 为什么不解析 target_uid？

`target_uid` 属于好友业务 payload。Gateway 只需要从 token 中拿到 requester uid 并写入 packet header，目标用户字段由 Friend service 解析和校验，避免 Gateway 混入好友业务逻辑。

### Q: 当前好友系统还缺什么？

缺好友备注、分组、黑名单完整接口、申请附言展示、删除好友、好友变更 push 通知、双向关系更细粒度审计等。这些属于后续扩展，不影响当前 MVP 主链路。
