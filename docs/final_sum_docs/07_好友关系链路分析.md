# MyChat 好友关系链路分析

日期：2026-06-16

## 1. 文档目标

这份文档分析 MyChat 好友系统的当前实现：

- 用户搜索与好友申请如何衔接。
- 好友申请、接受、拒绝、好友列表的完整链路。
- Friend Service 如何校验自加好友、重复申请、目标用户存在性。
- local/remote Friend Service 如何保持同一套业务语义。
- 当前 MVP 与生产级社交关系系统的边界。

一句话概括：

```text
MyChat 的好友系统以“申请记录 + 状态流转”为核心模型，用户搜索由 User Service 提供，
好友关系生命周期由 Friend Service 管理，Gateway 负责 token 身份和 HTTP 接入。
```

## 2. 模块职责

| 模块 | 职责 |
| --- | --- |
| Web Client | 搜索用户、展示用户资料卡、发起申请、处理待处理申请、展示好友列表 |
| Gateway User HTTP | 提供 `/api/v1/users/search` 用户搜索 |
| Gateway Friend HTTP | 提供好友申请、处理、列表接口 |
| MultiPlatformAuthManager | 校验 access token，解析当前 uid |
| FriendClient facade | 屏蔽 local/remote Friend Service 差异 |
| Friend Service | 好友申请状态机、关系校验、好友列表/待处理列表 |
| FriendRepository | 通过 ODB 操作好友关系表 |
| User Service | 校验目标用户是否存在，解析昵称 |
| PostgreSQL | 持久化好友申请和关系状态 |

## 3. 对外接口

### 3.1 用户搜索

```text
GET /api/v1/users/search?keyword=<keyword>
```

搜索由 User Service 提供，支持账号或昵称模糊搜索。

### 3.2 好友接口

```text
POST /api/v1/friends/request
POST /api/v1/friends/respond
GET  /api/v1/friends
GET  /api/v1/friends/pending
```

### 3.3 gRPC 边界

定义在 `common/proto/friend.proto`：

```text
im.friend_.FriendService.SendRequest
im.friend_.FriendService.RespondToRequest
im.friend_.FriendService.GetFriends
im.friend_.FriendService.GetPendingRequests
```

注意：proto package 使用 `im.friend_`，是为了避开 C++ 关键字 `friend`。

## 4. 用户搜索到好友申请链路

### 4.1 用户视角

正常 IM 体验应该是：

```text
搜索账号/昵称
-> 展示搜索结果和资料卡
-> 用户确认后点击添加好友
-> 发送好友申请
```

不是直接输入内部 uid 后立即申请。

### 4.2 服务端链路

搜索链路：

```text
Web Client
-> GET /api/v1/users/search
-> Gateway UserHttpController
-> verify_access_token
-> UserClient::search_profiles
-> User Service / Remote User gRPC
-> UserRepository::search_by_account_or_nickname
-> 返回用户资料列表
```

申请链路：

```text
Web Client
-> POST /api/v1/friends/request
-> Gateway FriendHttpController::handle_send_request
-> verify_access_token
-> requester_uid = token_user.user_id
-> target_uid = body.target_uid
-> FriendClient::send_request
-> FriendService::send_request
-> 校验目标用户存在
-> 校验不能添加自己
-> 校验不存在既有关系/申请
-> FriendRepository::create(PENDING)
-> 返回 201
```

## 5. 好友申请状态机

当前状态：

```text
PENDING  -> ACCEPTED
PENDING  -> REJECTED
```

当前不允许：

- 非 pending 状态重复处理。
- 不是目标用户的人处理申请。
- 对同一用户重复发起关系。

状态意义：

| 状态 | 含义 |
| --- | --- |
| PENDING | 申请已发出，等待目标用户处理 |
| ACCEPTED | 目标用户已接受，双方成为好友 |
| REJECTED | 目标用户已拒绝，申请结束 |

## 6. 处理好友申请链路

接口：

```text
POST /api/v1/friends/respond
```

请求核心字段：

```json
{
  "friend_id": 123,
  "accept": true
}
```

服务端链路：

```text
Gateway FriendHttpController::handle_respond_request
-> verify_access_token
-> friend_id 参数校验
-> FriendClient::respond_to_request(friend_id, current_uid, accept)
-> FriendService::respond_to_request
-> FriendRepository::find_by_id
-> 校验当前用户 == target_uid
-> 校验状态 == PENDING
-> update_status(ACCEPTED 或 REJECTED)
-> 返回结果
```

关键安全点：

```text
只有申请目标用户可以处理申请。
```

当前通过：

```text
f.target_uid() == uid
```

校验。

## 7. 好友列表与待处理列表

### 7.1 好友列表

接口：

```text
GET /api/v1/friends
```

链路：

```text
Gateway 验证 token
-> FriendClient::get_friends(current_uid)
-> FriendService::get_friends
-> FriendRepository::find_friends
-> resolve_nickname(other_uid)
-> 返回 FriendInfoDTO
```

`FriendInfoDTO` 当前包含：

- friend_id。
- friend_uid。
- nickname。
- status。
- created_at。

### 7.2 待处理申请

接口：

```text
GET /api/v1/friends/pending
```

语义：

```text
查询发给当前用户、状态为 PENDING 的好友申请。
```

Web 客户端据此展示“待处理申请”，用户可接受或拒绝。

## 8. 关键规则

### 8.1 不能添加自己

```text
requester_uid == target_uid -> SELF_REQUEST
```

### 8.2 目标用户必须存在

Friend Service 会调用 User Service：

```text
user_service_->get_profile_by_uid(target_uid)
```

目标不存在时返回：

```text
TARGET_NOT_FOUND
```

### 8.3 重复关系拒绝

FriendRepository 会检查两人之间是否已有关系：

```text
repo_->find_relationship(requester_uid, target_uid)
```

已有 pending/accepted/rejected 记录时，当前按已存在关系处理。

### 8.4 只能目标用户处理申请

```text
f.target_uid() != uid -> FORBIDDEN
```

### 8.5 非 pending 不能重复处理

```text
f.status() != PENDING -> NOT_PENDING
```

## 9. local/remote Friend Service

### 9.1 local 模式

```text
Gateway FriendHttpController
-> LocalFriendClient
-> FriendService in-process
-> FriendRepository
```

### 9.2 remote 模式

```text
Gateway FriendHttpController
-> RemoteFriendClient
-> im.friend_.FriendService gRPC
-> friend_server
-> FriendGrpcService
-> FriendService
-> FriendRepository
```

### 9.3 facade 的价值

Gateway HTTP controller 不需要直接处理 gRPC。

好处：

- local/remote 切换不影响 HTTP 接口。
- remote client 负责错误码转换。
- Friend Service 领域语义保持一致。
- 测试可以分别覆盖 controller、client、service。

## 10. 当前优势

- 好友关系不是前端状态，而是 PostgreSQL 持久化状态。
- 好友申请和好友关系使用明确状态机。
- 身份来自 token，不信任客户端传 requester_uid。
- 目标用户存在性由服务端校验。
- 支持远程 Friend gRPC 服务边界。

## 11. 当前不足和后续优化

当前不足：

- 没有好友备注。
- 没有好友分组。
- 没有删除好友。
- 没有黑名单和隐私规则。
- 重复 rejected 后是否允许重新申请，还可以细化。
- 好友申请附言字段在 proto 中存在，但当前 HTTP/service 侧使用还可扩展。
- 好友关系目前更像“申请记录状态”，后续可进一步拆成 friend_request 和 friend_relation 两类表。

后续优化：

- 增加好友备注和分组。
- 增加删除好友和双向关系解除。
- 增加黑名单与消息拦截。
- 增加隐私设置，例如是否允许被搜索。
- 增加申请附言和申请通知。
- 对 accepted 关系建立独立双向索引，优化好友列表查询。

## 12. 面试问答准备

### 12.1 为什么用户搜索不直接属于 Friend Service

回答思路：

```text
用户搜索查询的是用户资料，属于 User Service；Friend Service 负责关系生命周期。
这样 User Service 管人，Friend Service 管关系，职责更清晰。
```

### 12.2 如何避免用户伪造申请人

回答思路：

```text
Gateway 验证 access token 后，用 token 中的 uid 作为 requester_uid，
不信任客户端传入的申请人字段。
```

### 12.3 为什么好友申请要有 pending 状态

回答思路：

```text
添加好友不是单方立即生效的动作，需要目标用户确认。
pending 状态能表达“申请已发出但未处理”，也能防止重复申请和重复处理。
```

### 12.4 如何处理重复申请

回答思路：

```text
发送申请前会查两人之间是否已有关系或申请记录，存在则拒绝，避免同一对用户产生多条并发状态。
```

### 12.5 当前好友系统和生产系统差距在哪里

回答思路：

```text
当前完成了好友申请、处理和列表的 MVP 闭环；生产级还需要备注、分组、删除、黑名单、隐私规则和更细的状态建模。
```

## 13. 一句话总结

```text
MyChat 的好友系统通过 User Service 完成用户搜索，通过 Friend Service 管理申请状态机，
Gateway 负责 token 身份和 HTTP 接入，当前已经形成搜索、申请、处理、好友列表的完整 MVP 闭环。
```

### B. 发送好友申请按代码读

Gateway：

```text
FriendHttpController::handle_send_request
-> extract_bearer_token
-> auth_mgr_->verify_access_token(token, user_info)
-> json::parse(req.body)
-> target_uid = body.value("target_uid")
-> FriendRequest request
   requester_uid = user_info.user_id
   target_uid = target_uid
-> friend_client_->send_request(request)
```

Service：

```text
FriendService::send_request
-> 校验 requester_uid/target_uid 非空
-> requester_uid == target_uid -> SELF_REQUEST
-> user_service_->get_profile_by_uid(target_uid) 校验目标存在
-> repo_->find_relationship(requester_uid, target_uid) 查重复关系
-> Friend(..., PENDING, ...)
-> repo_->create(f)
```

关键代码点：

- 申请人不是客户端传的，而是 token uid。
- Friend Service 依赖 User Service 校验目标用户存在。
- 重复关系检查在 service 层完成。
- 当前创建的是一条 `PENDING` 关系/申请记录。

### C. 处理好友申请按代码读

Gateway：

```text
FriendHttpController::handle_respond_request
-> verify_access_token
-> friend_id = body.value("friend_id")
-> accept = body.value("accept")
-> friend_client_->respond_to_request(friend_id, user_info.user_id, accept)
```

Service：

```text
FriendService::respond_to_request
-> repo_->find_by_id(friend_id)
-> f.target_uid() != uid -> FORBIDDEN
-> f.status() != PENDING -> NOT_PENDING
-> new_status = ACCEPTED 或 REJECTED
-> repo_->update_status(friend_id, new_status)
```

这段代码体现两个关键规则：

- 只有目标用户能处理申请。
- pending 只能处理一次。

### D. 列表查询按代码读

好友列表：

```text
FriendHttpController::handle_list_friends
-> verify_access_token
-> friend_client_->get_friends(user_info.user_id)
-> FriendService::get_friends
-> repo_->find_friends(uid)
-> to_friend_info
-> resolve_nickname(other_uid)
```

待处理申请：

```text
FriendHttpController::handle_pending_requests
-> verify_access_token
-> friend_client_->get_pending_requests(user_info.user_id)
-> FriendService::get_pending_requests
-> repo_->find_pending(uid)
```

`to_friend_info` 中会根据当前用户判断对方 uid：

```text
other_uid = f.requester_uid() == uid ? f.target_uid() : f.requester_uid()
```

这解释了为什么同一条关系记录能从不同用户视角展示“对方”。

### E. remote Friend 按代码读

```text
RemoteFriendClient
-> 构造 im.friend_.AddFriendRequest / HandleFriendRequest
-> apply_deadline
-> stub->SendRequest / RespondToRequest
-> 检查 grpc status
-> 检查 BaseResponse.error_code
-> 转成 FriendResult / FriendInfoDTO
```

gRPC server：

```text
FriendGrpcService
-> 从 proto request 取 header.from_uid / to_uid / request_id
-> 调 FriendService
-> 把服务层错误映射为 BaseResponse
```

注意：`friend.proto` package 是 `im.friend_`，这是为避免 C++ 关键字冲突。

### F. 代码边界

- proto 中有 `FriendPermission`、黑名单相关消息，但当前 MVP HTTP/service 主路径尚未完整实现这些能力。
- 当前好友申请附言在 proto 中存在，但 HTTP/service 主链路主要围绕 target_uid 和状态流转。
- 当前关系模型更偏“一条记录表示申请及最终状态”，后续可以拆成申请表和好友关系表。
