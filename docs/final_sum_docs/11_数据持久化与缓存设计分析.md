# MyChat 数据持久化与缓存设计分析

日期：2026-06-16

## 1. 文档目标

这份文档分析 MyChat 的数据层设计：

- Redis 和 PostgreSQL 的职责边界。
- ODB 在 C++ 项目中的作用。
- 用户、消息、好友、群组分别如何持久化。
- token/session 等临时状态为什么放 Redis。
- migrations 和运行时准备脚本如何组织。
- 当前数据层距离生产级还有哪些差距。

一句话概括：

```text
MyChat 使用 PostgreSQL/ODB 保存可恢复的业务状态，用 Redis 保存高频、临时、可过期状态；
两者按数据生命周期和一致性要求分工，而不是混用。
```

## 2. 数据分类

MyChat 当前数据可以分成两类：

### 2.1 持久业务数据

特点：

- 必须可恢复。
- 需要长期保存。
- 需要查询历史。
- 需要关系建模。

当前包括：

- 用户账号与资料。
- 单聊消息。
- 好友申请和关系。
- 群资料。
- 群成员。
- 群消息。

存储：

```text
PostgreSQL + ODB
```

### 2.2 临时状态数据

特点：

- 高频读写。
- 生命周期短。
- 天然需要 TTL。
- 可由业务状态重新建立。

当前包括：

- refresh token 元数据。
- access token 撤销状态。
- 用户 token 索引。
- session/连接相关状态。
- Redis 连接池状态。

存储：

```text
Redis
```

## 3. PostgreSQL/ODB 设计

### 3.1 当前模型

ODB 模型位于：

```text
services/odb/user.hpp
services/odb/message.hpp
services/odb/friend.hpp
services/odb/group.hpp
services/odb/group_message.hpp
```

生成代码位于：

```text
services/odb/generated/
```

典型 repository：

```text
services/user/user_repository.*
services/message/message_repository.*
services/friend/friend_repository.*
services/group/group_repository.*
services/group/group_message_repository.*
```

### 3.2 为什么用 PostgreSQL

PostgreSQL 适合当前业务数据：

- 用户和关系天然是结构化数据。
- 消息历史需要可靠保存。
- 好友和群成员需要关系查询。
- 后续可扩展索引、事务、分页、统计。

### 3.3 为什么用 ODB

ODB 的价值：

- C++ 对象和数据库表映射。
- repository 可以围绕 C++ 对象写逻辑。
- 减少重复手写 SQL。
- 生成 schema 和对象访问代码。

代价：

- 工具链复杂。
- 生成物需要纳入构建。
- schema 演进仍要单独管理。
- 新人理解成本比手写 SQL 高。

面试表达：

```text
我用 ODB 是为了在 C++ 后端中保持对象模型和数据库表的映射，
但没有把数据库迁移完全交给 ODB 隐式处理，而是保留显式 migration 脚本。
```

## 4. 用户数据

User Service 持久化：

- uid。
- account。
- password_hash。
- nickname。
- avatar。
- gender。
- signature。
- create_time。
- last_login。
- online。

关键设计：

- 密码保存 PBKDF2-HMAC-SHA256 hash，不保存明文。
- uid 由服务端随机生成。
- account 唯一。
- avatar 当前是文本字段，可保存 URL 或 base64 data URL。

后续优化：

- 头像迁移到对象存储。
- 用户资料隐私设置。
- 登录安全日志。
- online 状态从持久字段进一步演进为在线状态服务。

## 5. 单聊消息数据

Message Service 持久化：

- msg_id。
- sender_uid。
- receiver_uid。
- content。
- msg_type。
- status。
- create_time。
- delivered_time。
- read_time。

核心查询：

```text
find_conversation(user_a, user_b, before_time, limit)
find_offline(receiver_uid, before_time, limit)
mark_delivered(msg_id, delivered_time)
mark_read(msg_id, read_time)
```

当前语义：

- 历史查询按双方 uid 查询双向消息。
- 离线查询查 `receiver_uid == 当前用户` 且 `status == SENT`。
- 拉取离线后标记 delivered。

后续优化：

- 会话表。
- 会话级 seq。
- client_msg_id 去重。
- 消息冷热分层。
- 全文搜索。

## 6. 好友关系数据

Friend Service 持久化：

- friend_id。
- requester_uid。
- target_uid。
- status。
- created_at。

当前状态：

- PENDING。
- ACCEPTED。
- REJECTED。

关键查询：

- 查两人之间是否已有关系。
- 查当前用户好友列表。
- 查当前用户待处理申请。
- 更新申请状态。

后续优化：

- friend_request 和 friend_relation 分表。
- 好友备注。
- 好友分组。
- 黑名单。
- 删除好友。

## 7. 群数据

Group Service 持久化三类数据：

### 7.1 群资料

- group_id。
- name。
- creator_uid。
- created_at。

### 7.2 群成员

- id。
- group_id。
- user_uid。
- role。
- joined_at。

### 7.3 群消息

- msg_id。
- group_id。
- sender_uid。
- content。
- created_at。

核心设计：

- 群资料和成员分离。
- 建群时创建 OWNER 成员关系。
- 群消息单独持久化。
- 群成员校验在服务端完成。

后续优化：

- 群公告、头像、设置持久化。
- 群成员禁言状态。
- 群管理员角色。
- 群消息 seq。
- 大群分表或分区。

## 8. Redis 设计

### 8.1 当前职责

Redis 当前主要服务认证和临时状态：

- refresh token 元数据。
- refresh token TTL。
- 用户 refresh token 集合。
- token 撤销状态。
- session/连接状态。

### 8.2 为什么 token 状态适合 Redis

原因：

- refresh token 天然有过期时间。
- 撤销状态只需要保留到 token 过期。
- 访问频繁，不适合每次查 PostgreSQL。
- Redis TTL 可以自动清理过期状态。

### 8.3 Redis 连接池

当前 Redis 封装基于 hiredis 和连接池。

关注点：

- pool size 可配置。
- wait timeout 可配置。
- 连接异常支持重连和一次重试。
- RAII 管理连接借还。

面试表达：

```text
Redis 不是用来替代数据库，而是用来承载 token/session 这类高频短生命周期状态。
连接池避免每次操作都创建 TCP 连接。
```

## 9. migration 和运行时准备

当前迁移基线：

```text
db/migrations/001_core_schema.sql
scripts/db/migrate_postgres.sh
```

运行时准备：

```text
scripts/dev/prepare_runtime.sh
```

它会：

- 尝试启动 Redis/PostgreSQL Docker 依赖。
- 等待容器健康。
- 执行 PostgreSQL migrations。

remote-all 启动脚本：

```text
scripts/dev/run_remote_services_stack.sh
```

会先调用 `prepare_runtime.sh`，再启动各服务。

当前策略：

```text
服务启动不自动迁移，由脚本显式准备运行时环境。
```

优点：

- 迁移职责清晰。
- 服务启动更可控。
- 多服务部署时避免多个服务同时抢迁移。

## 10. 数据一致性取舍

### 10.1 消息

当前核心原则：

```text
消息先持久化，再 Push。
```

因此：

- Push 失败不回滚。
- 离线拉取补偿。
- delivered/read 是后续状态更新。

### 10.2 好友

好友申请状态更新是单条关系状态流转。

当前不是复杂分布式事务。

### 10.3 群聊

群消息先持久化，再对成员 fanout。

群 fanout 不是事务的一部分。

### 10.4 token

access token 是 JWT，refresh token 在 Redis。

Redis 和 PostgreSQL 之间不做强事务绑定。

## 11. 当前优势

- Redis/PostgreSQL 职责清晰。
- 所有核心业务数据都可持久恢复。
- 临时 token 状态有 TTL。
- Repository 层隔离 ODB 访问。
- migration 显式执行。
- 支持 remote-all 多服务共享同一个数据库和 Redis。

## 12. 当前不足和后续优化

当前不足：

- migration 版本管理还比较基础。
- Repository 错误信息较粗。
- 消息查询索引和容量评估还需压测。
- Redis 高可用和故障降级未完整设计。
- 没有消息冷热分层。
- 没有统一数据访问 trace。
- ODB 工具链对环境依赖较强。

后续优化：

- 完善 migration 版本表和回滚策略。
- 增加数据库索引审计。
- 单聊/群聊消息分区或冷热归档。
- Redis Sentinel/Cluster 或托管 Redis。
- 增加慢查询日志和指标。
- 对 repository 增加更清晰的错误传播。
- 文件/图片迁移到对象存储。

## 13. 面试问答准备

### 13.1 Redis 和 PostgreSQL 怎么分工

回答思路：

```text
PostgreSQL 保存用户、消息、好友、群组等必须长期恢复的业务数据；
Redis 保存 token、session 这类高频、短生命周期、可过期状态。
```

### 13.2 为什么消息不放 Redis

回答思路：

```text
聊天消息是核心业务数据，需要历史查询和可靠恢复，不能只依赖 Redis。
Redis 可以做缓存或队列，但最终消息应该落到持久数据库。
```

### 13.3 ODB 的优缺点

回答思路：

```text
优点是 C++ 对象和数据库表映射清晰，减少手写 SQL；
缺点是工具链复杂，schema 演进和生成代码管理需要额外工程约束。
```

### 13.4 为什么服务启动不自动迁移

回答思路：

```text
多服务场景下如果每个服务启动都自动迁移，可能产生并发迁移和不可控失败。
当前把迁移放到脚本显式执行，职责更清晰。
```

### 13.5 Redis 挂了会影响什么

回答思路：

```text
refresh token、撤销状态和部分 session 状态会受影响；
已持久化的用户、消息、好友、群组数据仍在 PostgreSQL。
生产环境需要 Redis 高可用和降级策略。
```

## 14. 一句话总结

```text
MyChat 的数据层按生命周期分工：PostgreSQL/ODB 保存可恢复的核心业务状态，
Redis 保存 token/session 等短生命周期状态；消息可靠性依赖持久化，实时性由 Push 补充。
```

### B. ODB repository 通用模式

大多数 repository 都是这个结构：

```text
Repository::method
-> odb::transaction t(db_->begin())
-> db_->persist / db_->load / db_->query / db_->update
-> t.commit()
-> catch odb::exception
```

例如用户创建：

```text
UserRepository::create
-> odb::transaction
-> db_->persist(user)
-> commit
```

例如消息写入：

```text
MessageRepository::create
-> odb::transaction
-> db_->persist(msg)
-> commit
```

这说明：

- 事务边界在 repository。
- service 层不直接写 SQL。
- 失败时当前多返回 false/nullopt，错误信息较粗，这是后续可优化点。

### C. 用户数据按代码读

注册写入：

```text
UserService::register_user
-> repo_->account_exists
-> PasswordHasher::hash
-> User user(...)
-> UserRepository::create
```

查询登录：

```text
UserService::login_by_account
-> UserRepository::find_by_account
-> PasswordHasher::verify
-> UserRepository::update_last_login
```

资料更新：

```text
UserRepository::update_profile
-> db_->load<User>(uid)
-> user->nickname/avatar/gender/signature
-> db_->update(*user)
```

代码边界：

- avatar 是文本字段。
- online 字段存在，但生产级在线状态更适合独立在线状态/连接管理。

### D. 消息数据按代码读

发送：

```text
MessageService::send_text_message
-> MessageRepository::create
-> MessageRepository::find_by_id
```

历史：

```text
MessageRepository::find_conversation
-> sender=user_a receiver=user_b
   OR sender=user_b receiver=user_a
-> create_time < before_time
-> ORDER BY create_time
```

离线：

```text
MessageRepository::find_offline
-> receiver_uid == 当前用户
-> status == SENT
-> create_time < before_time
```

状态：

```text
mark_delivered -> status DELIVERED + delivered_time
mark_read -> status READ + read_time
```

这就是“先持久化、Push 失败可补偿”的数据库基础。

### E. 好友/群数据按代码读

好友：

```text
FriendService::send_request
-> FriendRepository::find_relationship
-> FriendRepository::create(PENDING)

FriendService::respond_to_request
-> FriendRepository::find_by_id
-> FriendRepository::update_status
```

群：

```text
GroupService::create_group
-> GroupRepository::create_group_with_owner

GroupService::join_group
-> GroupRepository::find_group_by_id
-> GroupRepository::find_member
-> GroupRepository::add_member

GroupMessageService::send_message
-> GroupMessageRepository::create
```

数据建模重点：

- 好友是用户之间的关系状态。
- 群资料和群成员分表。
- 群消息独立于单聊消息。

### F. Redis token 按代码读

refresh token 写入：

```text
MultiPlatformAuthManager::generate_refresh_token
-> redis.set("refresh_token:" + rt, rtMeta.dump())
-> redis.expire("refresh_token:" + rt, actual_expire_seconds)
-> redis.sadd("user:" + user_id + ":refresh_tokens", rt)
-> redis.expire("user:" + user_id + ":refresh_tokens", actual_expire_seconds)
```

access token 校验撤销：

```text
MultiPlatformAuthManager::verify_access_token
-> verifier.verify(decoded)
-> is_token_revoked(access_token)
```

refresh token 校验：

```text
verify_refresh_token
-> redis.get("refresh_token:" + refresh_token)
-> 检查 revoked/expire/device_id
```

Redis 适合这里的原因直接体现在代码：

- key 有 TTL。
- token 元数据是短生命周期状态。
- 用户 token 集合也是可过期索引。

### G. migration 和运行时按代码读

```text
scripts/dev/prepare_runtime.sh
-> docker compose up -d redis postgres
-> wait_for_container
-> scripts/db/migrate_postgres.sh
```

```text
scripts/dev/run_remote_services_stack.sh
-> prepare_runtime.sh
-> 启动所有远程服务和 Gateway
```

这说明当前 migration 是显式运行步骤，不是每个服务启动时自动执行。
