# MyChat 单聊消息链路分析

日期：2026-06-16

## 1. 文档目标

单聊消息是 IM 项目最核心的业务链路。这份文档重点分析：

- HTTP 单聊发送链路。
- WebSocket 单聊发送链路。
- 消息持久化与发送 ack 的关系。
- Push 在线投递与离线补偿的关系。
- history/offline 查询语义。
- delivered/read 状态的当前实现边界。
- 面试中如何讲清楚“消息不丢”和“实时推送”的取舍。

一句话概括当前设计：

```text
MyChat 的单聊消息以“先持久化、再推送”为核心语义；发送成功表示服务端已保存消息，
在线 Push 是 best-effort 加速路径，失败后通过历史查询和离线拉取补偿。
```

## 2. 单聊链路中的模块职责

| 模块 | 职责 |
| --- | --- |
| Web Client | 发送文本、展示 ack、接收 WebSocket push、拉取历史/离线消息 |
| Gateway HTTP | 提供 `/api/v1/messages/*` 请求响应接口 |
| Gateway WebSocket | 处理 `CMD_SEND_MESSAGE` 二进制消息和 ack |
| MultiPlatformAuthManager | 校验 access token，解析可信 uid |
| MessageClient facade | 屏蔽 local/remote Message Service 差异 |
| Message Service | 校验消息数据、持久化消息、查询历史/离线、标记状态 |
| MessageRepository | 通过 ODB 操作 PostgreSQL 消息表 |
| PushNotifier | 在消息持久化成功后触发在线投递 |
| PushRuntime | 查询在线 session、构造 push payload、fanout、标记 delivered |
| ConnectionManager/WebSocketServer | Gateway 内部持有在线连接和 session |

## 3. 当前接口与协议

### 3.1 HTTP 接口

```text
POST /api/v1/messages/send
GET  /api/v1/messages/history
GET  /api/v1/messages/offline
```

### 3.2 WebSocket 命令

```text
CMD_SEND_MESSAGE + im.message.SendMessageRequest
CMD_PUSH_MESSAGE + im.push.PushRequest
```

### 3.3 gRPC 服务边界

```text

```


## 4. HTTP 单聊发送链路

### 4.1 用户操作

Web 客户端在聊天窗口发送文本消息，可通过 HTTP 发送。

请求：

```text
POST /api/v1/messages/send
Authorization: Bearer <access_token>
```

请求体核心字段：

```json
{
  "receiver_uid": "target-user-uid",
  "content": "hello"
}
```

### 4.2 服务端链路

```text
Web Client
-> Gateway MessageHttpController::handle_send
-> extract_bearer_token
-> MultiPlatformAuthManager::verify_access_token
-> JSON 解析 receiver_uid/content
-> sender_uid = token_user.user_id
-> MessageClient::send_text_message
-> LocalMessageClient 或 RemoteMessageClient
-> MessageService::send_text_message
-> MessageRepository::create
-> PostgreSQL/ODB persist
-> MessageRepository::find_by_id
-> 返回 message
-> PushNotifier::notify_user
-> 返回 HTTP 201 + message
```

### 4.3 关键点

#### sender_uid 来自 token

HTTP body 中不信任 sender_uid。

当前发送者身份来自：

```text
user_info.user_id
```

也就是 Gateway 校验 access token 后解析出的 uid。

这个设计避免客户端伪造发送者。

#### ack 语义

HTTP `201` 返回表示：

```text
消息已经被 Message Service 接收并保存到 PostgreSQL。
```

它不表示：

- 对方一定在线。
- 对方一定收到 WebSocket push。
- 对方已经读过消息。

#### Push 是 best-effort

消息保存成功后，Gateway 调用 PushNotifier。

Push 失败不会回滚消息，因为消息已经持久化，后续可以离线拉取。

## 5. WebSocket 单聊发送链路

### 5.1 用户操作

Web 客户端通过 WebSocket 发送二进制 Protobuf frame：

```text
CMD_SEND_MESSAGE
im.message.SendMessageRequest
```

### 5.2 Gateway 处理链路

```text
WebSocket binary frame
-> MessageParser / UnifiedMessage
-> MessageWsHandler::handle_send
-> 校验是否 WebSocket 消息
-> 校验 cmd_id == CMD_SEND_MESSAGE
-> verify_access_token(header.token)
-> 校验 header.device_id 与 token device_id 一致
-> 校验 protobuf type == im.message.SendMessageRequest
-> 解析 SendMessageRequest
-> receiver_uid = header.to_uid
-> content = body.content
-> sender_uid = token_user.user_id
-> MessageClient::send_text_message
-> MessageService 持久化
-> 构造 SendMessageResponse ack
-> PushNotifier::notify_user
-> 返回 protobuf ack 给发送端
```

### 5.3 WebSocket ack 语义

WebSocket ack 是 `im.message.SendMessageResponse`。

成功 ack 表示：

```text
服务端已经校验 token、解析消息并完成持久化。
```

失败 ack 可能来自：

- cmd_id 错误。
- access token 无效或过期。
- device_id 与 token 不匹配。
- protobuf type 不匹配。
- 请求体解析失败。
- receiver_uid 为空。
- content 为空。
- Message Service 持久化失败。
- ack 编码失败。

### 5.4 为什么 WebSocket 发送也要先落库

即使发送来自 WebSocket，也不应该直接转发给对方。

原因：

- 对方可能已经离线。
- Gateway 发送可能失败。
- 推送成功不代表持久化成功。
- 客户端需要可恢复历史。

所以 WebSocket 发送仍然遵循：

```text
验证身份 -> 解析消息 -> 持久化 -> ack -> best-effort push
```

## 6. Message Service 持久化逻辑

### 6.1 SendRequest

服务层发送请求包含：

- sender_uid。
- receiver_uid。
- content。
- msg_type。
- now_ms。

### 6.2 服务层校验

`MessageService::send_text_message` 会校验：

- sender_uid 不为空。
- receiver_uid 不为空。
- content 不为空。

失败时返回服务层错误：

- `EMPTY_SENDER`
- `EMPTY_RECEIVER`
- `EMPTY_CONTENT`
- `PERSIST_FAILED`

### 6.3 Repository 写入

持久化链路：

```text
MessageService::send_text_message
-> Message msg(...)
-> MessageRepository::create
-> odb::transaction
-> db_->persist(msg)
-> commit
-> MessageRepository::find_by_id
-> 返回 MessageData
```

当前写入成功后会再按 msg_id 读回一次，用于返回带服务端补全字段的消息数据。

### 6.4 当前消息状态

单聊消息当前核心状态：

- `SENT`：已保存，但还没有成功投递到在线 session。
- `DELIVERED`：至少一次在线投递成功，或离线拉取后标记 delivered。
- `READ`：已读标记，当前后端接口存在，UI 和多端同步还可继续完善。

## 7. 在线 Push 链路

### 7.1 触发时机

HTTP 和 WebSocket 发送都在消息持久化成功后触发 Push：

```text
MessageService 持久化成功
-> PushNotifier::notify_user(receiver_uid, msg_id, content, context)
```

### 7.2 本地 Push 模式

```text
Gateway Message entrypoint
-> gateway/push/PushService
-> services/push/PushRuntime
-> ConnectionManager::get_user_sessions(receiver_uid)
-> FanoutPolicy::select_sessions
-> ProtobufCodec::encode CMD_PUSH_MESSAGE + PushRequest
-> WebSocketServer::get_session(session_id)
-> session->send(payload)
-> MessageClient::mark_delivered
```

### 7.3 远程 Push 模式

```text
Gateway Message entrypoint
-> RemotePushNotifier
-> push_server PushGrpcService::NotifyUser
-> PushRuntime
-> GatewayPushDeliveryService callback
-> ListUserSessions
-> SendSessionPayload
-> MarkMessageDelivered
```

### 7.4 为什么远程 Push 需要 callback

WebSocket session 在 Gateway 内存中。

远程 Push Service 不能直接拿到：

- Gateway 的 ConnectionManager。
- Gateway 的 WebSocketServer。
- 某个 session 对象。

因此远程 Push 需要通过 Gateway 暴露的 callback 服务：

```text
GatewayPushDeliveryService
```

由 Gateway 负责：

- 查询用户在线 session。
- 向指定 session 发送 payload。
- 回写 delivered 状态。

这是 MyChat 当前分布式设计里很重要的面试亮点。

### 7.5 fanout policy

当前支持：

- `AllSessionsFanoutPolicy`：所有在线 session。
- `PlatformFilterFanoutPolicy`：指定平台。
- `NewestSessionFanoutPolicy`：最新 session。

默认适合 MVP 的策略是全 session 推送，方便验证多端在线。

## 8. 历史消息查询链路

接口：

```text
GET /api/v1/messages/history?peer_uid=<uid>&before_time=<ts>&limit=<n>
```

链路：

```text
Gateway 验证 token
-> 当前用户 uid = token_user.user_id
-> peer_uid 来自 query
-> MessageClient::get_conversation(user_uid, peer_uid, before_time, limit)
-> MessageRepository::find_conversation
-> 查询 user_a -> user_b 或 user_b -> user_a 的消息
-> ORDER BY create_time
-> 返回 messages
```

当前语义：

- 查询双方会话消息。
- `before_time` 用于分页上界。
- `limit` 控制返回数量。
- 返回按创建时间排序的历史消息。

面试表达：

```text
历史查询不是按客户端传 sender/receiver 查单方向，而是按当前 token 用户和 peer_uid 查询双向会话。
```

## 9. 离线消息拉取链路

接口：

```text
GET /api/v1/messages/offline?before_time=<ts>&limit=<n>
```

链路：

```text
Gateway 验证 token
-> 当前用户 uid = token_user.user_id
-> MessageClient::pull_offline(receiver_uid = 当前 uid)
-> MessageRepository::find_offline
-> 查询 receiver_uid == 当前 uid 且 status == SENT
-> 返回消息列表
-> Gateway 对返回消息逐条 mark_delivered
-> 返回 messages
```

当前离线拉取的核心语义：

```text
离线消息 = 发给当前用户、尚未 delivered 的单聊消息。
```

拉取后标记 delivered，避免后续重复作为离线消息返回。

需要注意：

- 这不是完整的可靠队列实现。
- 当前没有复杂重试表或消息投递日志。
- 但对于 MVP，能表达“在线推送失败后离线补偿”的核心思路。

## 10. delivered/read 状态解释

### 10.1 ack

ack 发生在发送端。

表示：

```text
服务端已接收并保存消息。
```

### 10.2 delivered

delivered 表示：

```text
消息至少被投递到接收者的一个在线 session，或者被接收者离线拉取到。
```

它不表示用户已经读过。

### 10.3 read

read 表示：

```text
用户已读。
```

当前后端有 `MarkRead` 边界和 repository 能力，但前端已读回执、多端同步、会话级已读水位还可以继续完善。

### 10.4 面试中怎么讲

可以这样区分：

```text
ack 是发送者视角的服务端接收确认；
delivered 是服务端视角的投递确认；
read 是接收者视角的阅读确认。
当前 MyChat 已经实现 ack 和 delivered 的核心链路，read 边界存在但产品化还可继续完善。
```

## 11. local/remote Message Service

### 11.1 local 模式

```text
Gateway HTTP/WS
-> LocalMessageClient
-> MessageService in-process
-> MessageRepository
```

优点：

- 本地调试快。
- 单元测试简单。
- 少一层网络故障。

### 11.2 remote 模式

```text
Gateway HTTP/WS
-> RemoteMessageClient
-> im.message.MessageService gRPC
-> message_server
-> MessageGrpcService
-> MessageService
-> MessageRepository
```

优点：

- Message Service 可以独立部署。
- Gateway 不依赖进程内 MessageService。
- 更贴近分布式 IM 服务结构。

### 11.3 remote client 的价值

RemoteMessageClient 做了：

- gRPC stub 调用。
- deadline 控制。
- proto 和 service DTO 转换。
- gRPC 错误映射为服务层错误。

这样 Gateway controller/handler 不需要直接处理 gRPC 细节。

## 12. 当前链路优势

### 12.1 消息不因 Push 失败而丢失

核心原因：

```text
消息先写 PostgreSQL，再尝试在线 Push。
```

Push 失败时：

- 发送端已经拿到服务端保存成功的 ack。
- 接收端后续可通过历史或离线拉取拿到消息。

### 12.2 身份安全边界清晰

发送者 uid 来自 token，而不是客户端随意传入。

WebSocket 还额外校验：

```text
header.device_id == token_user.device_id
```

### 12.3 分布式边界可演示

Message Service 和 Push Service 都可以 remote 化：

- Gateway -> RemoteMessageClient -> message_server。
- Gateway -> RemotePushNotifier -> push_server。
- push_server -> GatewayPushDeliveryService callback。

这能支撑多进程和后续多机器验证。

## 13. 当前不足和后续优化

### 13.1 当前不足

- 当前没有会话级全局递增 seq。
- 当前没有客户端生成 client_msg_id 去重。
- 当前 offline pull 以 `SENT` 状态为主，属于简化补偿模型。
- 当前 delivered 表示至少一个 session 收到，不表示所有设备都收到。
- 当前 read 边界存在，但产品化已读回执还不完整。
- 当前 PushContext 中 direct conversation_id 的表达还可优化，目前更接近“发送者 uid/上下文 id”，后续建议明确为接收方视角的 peer uid 或统一 conversation id。
- 当前没有完整消息撤回、删除、编辑。
- 当前没有富媒体消息的对象存储链路。

### 13.2 后续优化方向

- 引入 `client_msg_id`，支持发送重试去重。
- 引入会话级 `seq`，增强消息顺序和增量同步。
- 引入会话表，统一 direct/group conversation id。
- 增强离线队列和投递日志。
- 完善 read receipt 和多端已读水位。
- 支持消息撤回、编辑、删除。
- 支持图片、文件、语音等富媒体消息。
- PushContext 统一使用标准会话 id，减少前端上下文推断。
- 多 Gateway 下维护 uid -> gateway/session 位置索引。

## 14. 面试问答准备

### 14.1 为什么消息要先落库再推送

回答思路：

```text
因为在线推送不可靠，接收者可能离线，Gateway session 也可能断开。
如果先推送再落库，可能出现客户端收到但服务端不可恢复的问题。
所以 MyChat 以持久化成功作为发送成功语义，Push 只是在线加速路径，失败后通过离线拉取补偿。
```

### 14.2 ack、delivered、read 有什么区别

回答思路：

```text
ack 是发送者收到的服务端保存确认；
delivered 是消息至少投递到接收方一个 session 或被离线拉取；
read 是接收者真正读过。三者处在不同视角，不能混为一谈。
```

### 14.3 如何防止用户伪造 sender_uid

回答思路：

```text
Gateway 不信任客户端传入 sender_uid，HTTP 和 WebSocket 发送都会先校验 access token，
然后用 token 中解析出的 uid 作为 sender_uid。WebSocket 还会校验 device_id。
```

### 14.4 Push 服务为什么不直接操作 WebSocket session

回答思路：

```text
WebSocket 连接是 Gateway 接入层持有的内存状态。远程 Push 服务如果直接操作 session，
会破坏连接所有权边界，也无法跨进程访问 Gateway 内存。因此 Push 通过 Gateway callback 查询 session 并发送 payload。
```

### 14.5 Push 失败会不会导致消息丢失

回答思路：

```text
不会直接导致消息丢失，因为消息在 Push 之前已经持久化。Push 失败只影响实时性，
接收者后续可以通过 offline pull 或 history 获取消息。
```

### 14.6 当前单聊链路还缺什么生产能力

回答思路：

```text
当前是完整 MVP，但生产级还需要 client_msg_id 去重、会话 seq、投递日志、多 Gateway 连接路由、
已读水位、多端同步、压测和监控。
```

## 15. 一句话总结

单聊链路可以总结为：

```text
MyChat 的单聊消息通过 Gateway 完成 HTTP/WebSocket 接入和 token 身份校验，
通过 Message Service 先持久化到 PostgreSQL，再通过 Push Service 做 best-effort 在线投递；
发送 ack、delivered、read 三类状态语义分离，Push 失败由离线拉取和历史查询补偿。
```

### B. HTTP 发送按代码读

```text
MessageHttpController::handle_send
-> extract_bearer_token
-> auth_mgr_->verify_access_token(token, user_info)
-> json::parse(req.body)
-> receiver_uid/content 校验
-> SendRequest send_req
   sender_uid = user_info.user_id
   receiver_uid = body.receiver_uid
   content = body.content
-> msg_client_->send_text_message(send_req)
-> push_notifier_->notify_user(...)
-> HTTP 201 + message
```

关键代码点：

- sender_uid 来自 `user_info.user_id`，不是客户端 JSON。
- `msg_client_->send_text_message` 返回成功后才触发 Push。
- `push_notifier_` 为空时消息仍然发送成功，只是没有在线推送。
- 当前 direct PushContext 的 `conversation_id` 在 HTTP/WS 单聊中传的是发送者 uid，这能让接收端知道对端是谁，但后续更理想是统一成标准 conversation id。

### C. WebSocket 发送按代码读

```text
MessageWsHandler::handle_send
-> msg.is_websocket()
-> msg.get_cmd_id() == CMD_SEND_MESSAGE
-> auth_mgr_->verify_access_token(header.token(), token_user)
-> header.device_id 与 token_user.device_id 比对
-> protobuf type == im.message.SendMessageRequest
-> send_req.ParseFromString
-> receiver_uid = header.to_uid()
-> content = send_req.body().content()
-> store_req.sender_uid = token_user.user_id
-> msg_client_->send_text_message(store_req)
-> 构造 im.message.SendMessageResponse
-> ProtobufCodec::encode
-> push_notifier_->notify_user
-> ProcessorResult(protobuf_response)
```

关键代码点：

- WebSocket 比 HTTP 多了 `cmd_id`、protobuf type、device_id 校验。
- ack 是 `SendMessageResponse`，由 `ProtobufCodec::encode` 返回给发送端。
- 发送成功的核心语义仍然是持久化成功，不是 Push 成功。

### D. Message Service 持久化按代码读

```text
MessageService::send_text_message
-> 校验 sender_uid/receiver_uid/content
-> Message msg(...)
-> repo_->create(msg)
-> repo_->find_by_id(msg.msg_id())
-> to_data
```

Repository：

```text
MessageRepository::create
-> odb::transaction t(db_->begin())
-> db_->persist(msg)
-> t.commit()
```

这说明：

- 消息是 PostgreSQL/ODB 持久化数据。
- Service 层先做最小业务校验。
- Repository 层负责事务和 ODB 操作。

### E. 历史与离线按代码读

历史查询：

```text
MessageHttpController::handle_history
-> token uid = user_info.user_id
-> peer_uid from query
-> msg_client_->get_conversation(user_info.user_id, peer_uid, before_time, limit)
-> MessageRepository::find_conversation
```

`find_conversation` 查询两个方向：

```text
sender=user_a receiver=user_b
OR
sender=user_b receiver=user_a
```

离线拉取：

```text
MessageHttpController::handle_offline
-> msg_client_->pull_offline(user_info.user_id, before_time, limit)
-> MessageRepository::find_offline
-> receiver_uid == 当前用户
-> status == SENT
-> 返回后逐条 mark_delivered
```

这就是“Push 失败或用户离线后可以补偿”的代码基础。

### F. delivered/read 按代码读

delivered：

```text
PushRuntime::notify_user
-> success_count > 0
-> delivery_marker_->mark_delivered(msg_id, now_ms())
```

本地 Gateway 实现：

```text
PushService::mark_delivered
-> msg_client_->mark_delivered
-> MessageService::mark_delivered
-> MessageRepository::mark_delivered
```

离线拉取也会补标 delivered：

```text
MessageHttpController::handle_offline
-> msg_client_->mark_delivered(m.msg_id, delivered_now)
```

read：

```text
MessageService::mark_read
-> MessageRepository::mark_read
```

当前 read 后端边界存在，但前端已读回执和多端水位还不是完整产品化能力。
