# 单聊消息链路分析

日期：2026-06-23

## 1. 文档目标

单聊消息是 IM 项目的核心链路。这份文档按当前统一 packet 架构解释：

- HTTP 单聊发送。
- WebSocket 单聊发送。
- 消息持久化与 ack 的关系。
- Push 在线投递与离线补偿。
- history/offline 查询语义。
- local/remote gRPC 边界。

当前权威链路不是旧的 `MessageHttpController -> MessageClient`，而是：

```text
HTTP / WebSocket
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> message_command_handlers.cpp
-> MessagePacketRequest
-> local MessagePacketDispatcher 或 remote MessageService::ForwardPacket
-> MessagePacketResponse
-> optional push_event
```

一句话概括：

```text
MyChat 单聊消息以“先持久化、再 ack、再 best-effort push”为核心语义；
Gateway 只做鉴权、路由和 packet 转发，消息 payload 解析与持久化在 Message service。
```

## 2. 模块职责

| 模块 | 当前职责 |
| --- | --- |
| Web Client | 发送文本、展示 ack、接收 WS push、拉取历史/离线消息 |
| Gateway HTTP | catch-all 接入、HTTP route -> cmd_id、token 校验、packet 转发 |
| Gateway WebSocket | WS envelope 解码、token/device 校验、packet 转发、ack envelope 编码 |
| Message command handler | 构造 `MessagePacketRequest`，选择 local dispatcher 或 remote adapter |
| MessageWsHandler | WebSocket `CMD_SEND_MESSAGE` 的 Gateway 边界处理 |
| MessagePacketDispatcher | 解析 HTTP JSON 或 WS protobuf payload，调用 MessageService，构造 response/push_event |
| MessageService | 消息校验、持久化、历史/离线查询、delivered/read 状态 |
| MessageRepository | PostgreSQL/ODB 消息表读写 |
| PushNotifier | 消费 message service 返回的 push_event，触发在线投递 |
| ConnectionManager/WebSocketServer | Gateway 内部维护在线 session 和 payload 投递 |

## 3. 当前接口与协议

HTTP 接口：

```text
POST /api/v1/messages/send
GET  /api/v1/messages/history
GET  /api/v1/messages/offline
```

WebSocket 命令：

```text
CMD_SEND_MESSAGE
```

gRPC packet 边界：

```proto
rpc ForwardPacket(MessagePacketRequest) returns (MessagePacketResponse);
```

packet 契约：

```text
MessagePacketRequest {
  IMHeader header;
  string type_name;
  bytes payload;
}

MessagePacketResponse {
  BaseResponse base;
  IMHeader header;
  string type_name;
  bytes payload;
  int32 http_status;
  MessagePushEvent push_event;
}
```

## 4. HTTP 单聊发送链路

用户请求：

```text
POST /api/v1/messages/send
Authorization: Bearer <access_token>

{
  "receiver_uid": "target_uid",
  "content": "hello"
}
```

当前链路：

```text
GatewayServer::init_http_server catch-all
-> MessageParser::parse_http_request_enhanced
-> RouterManager::parse_http_route -> CMD_SEND_MESSAGE
-> MessageProcessor::process_message
-> MessageProcessor::process_message_sync
-> message_command_handlers.cpp CMD_SEND_MESSAGE
-> verify_http_user
-> packet.header.from_uid = token uid
-> packet.payload = raw JSON body
-> MessagePacketDispatcher::handle_send_http 或 remote ForwardPacket
-> MessageService::send_text_message
-> MessageRepository::create
-> MessagePacketResponse(payload JSON + push_event)
-> PushNotifier::notify_user
-> HTTP response
```

关键边界：

- Gateway 不解析 `receiver_uid/content`。
- Gateway 不信任 body 中的 sender。
- Gateway 从 token 中得到可信 sender uid，写入 packet header。
- `MessagePacketDispatcher` 解析 JSON payload 并调用 MessageService。
- Message service 返回 `push_event`，Gateway 只负责转交。

## 5. WebSocket 单聊发送链路

WebSocket frame：

```text
IMHeader(cmd_id=CMD_SEND_MESSAGE, token, to_uid, device_id, platform)
+ type_name = im.message.SendMessageRequest
+ payload = SendMessageRequest bytes
```

当前链路：

```text
WebSocket frame
-> ProtobufCodec::decodeEnvelope
-> MessageParser::parse_websocket_message_enhanced
-> UnifiedMessage
-> Gateway ws_inflight_messages limit
-> ThreadPool::Enqueue
-> MessageProcessor::process_message_sync
-> message_command_handlers.cpp CMD_SEND_MESSAGE
-> MessageWsHandler::handle_send
-> verify_access_token(header.token)
-> 校验 header.device_id 与 token device_id 一致
-> service_header.from_uid = token uid
-> service_header.to_uid = header.to_uid
-> MessagePacketRequest(type_name + protobuf payload)
-> MessagePacketDispatcher::handle_send_ws 或 remote ForwardPacket
-> MessageService::send_text_message
-> MessagePacketResponse(type_name=im.message.SendMessageResponse)
-> ProtobufCodec::encodeEnvelope
-> sender ack
-> optional PushNotifier::notify_user
```

注意：

- Gateway 不解析 `SendMessageRequest` 的 body。
- WebSocket payload protobuf 解析在 `MessagePacketDispatcher` 中完成。
- Gateway 只检查 header 级别的 token、device、receiver 路由信息。

## 6. Ack 语义

HTTP 成功响应或 WS 成功 ack 表示：

```text
消息已经通过 Message service 校验并持久化。
```

它不表示：

- 对方一定在线。
- 对方一定收到 push。
- 对方已经阅读。

为什么这样设计：

- 对方可能离线。
- WebSocket 推送可能失败。
- 消息持久化成功后才能保证历史可恢复。
- push 是加速路径，不是消息可靠性的唯一来源。

## 7. Push 语义

Message service 保存消息后返回：

```text
MessagePushEvent {
  enabled = true;
  receiver_uid;
  msg_id;
  content;
  sender_uid;
  conversation_type = "direct";
  conversation_id = sender_uid;
}
```

Gateway 收到 `push_event` 后：

```text
PushNotifier::notify_user(receiver_uid, msg_id, content, context)
```

local push：

```text
PushRuntime
-> ConnectionManager
-> WebSocketServer
-> receiver sessions
```

remote push：

```text
RemotePushNotifier
-> push_server
-> GatewayPushDeliveryService callback
-> Gateway session lookup / payload send
```

Push 失败不回滚消息。后续靠 history/offline 查询补偿。

## 8. 历史与离线消息

HTTP history：

```text
GET /api/v1/messages/history?peer_uid=<uid>&before_time=<ms>&limit=<n>
-> CMD_MESSAGE_HISTORY
-> MessagePacketDispatcher
-> MessageService::get_conversation
```

HTTP offline：

```text
GET /api/v1/messages/offline?before_time=<ms>&limit=<n>
-> CMD_PULL_MESSAGE
-> MessagePacketDispatcher
-> MessageService::pull_offline
```

当前语义：

- history 查询当前用户与 peer 的会话历史。
- offline 拉取当前用户未投递的一对一消息。
- query 参数先由 `MessageParser` 收进 `UnifiedMessage`，再作为 JSON payload 交给 MessagePacketDispatcher 解析。
- Gateway 不解析 `peer_uid/before_time/limit`。

## 9. Delivered / Read 状态

当前 Message service 支持：

- delivered 标记。
- read 标记。

当前 MVP 的边界要讲清楚：

- delivered/read 能力存在，但还不是生产级完整回执体系。
- 没有完整 client ACK 驱动的端到端送达确认。
- 没有 per-device delivery log。
- 没有严格消息 sequence。

面试表达：

```text
当前 MVP 的可靠性重点是“消息先落库，可通过历史/离线恢复”。
delivered/read 是服务端状态能力，但生产级还需要 client ack、设备级投递记录和会话序列号继续完善。
```

## 10. Local / Remote 差异

local 模式：

```text
message_command_handlers.cpp
-> MessagePacketDispatcher dispatcher(message_service)
-> dispatcher.handle(packet)
```

remote 模式：

```text
message_command_handlers.cpp
-> remote_message_adapter->forward(packet)
-> MessageService::ForwardPacket
-> message_server
-> MessagePacketDispatcher::handle(packet)
```

WebSocket 也是同一个 packet 边界：

```text
MessageWsHandler::handle_send
-> MessagePacketRequest(type_name=im.message.SendMessageRequest)
-> local dispatcher 或 remote ForwardPacket
```

这保证了 HTTP、WS、local、remote 四种组合共享同一套 Message service 业务解析。

## 13. 面试追问

### Q: 为什么 Gateway 不直接解析 SendMessageRequest？

因为消息正文属于 Message service 业务 payload。Gateway 只需要检查 token 和 header 路由字段，解析 protobuf body 会把业务规则重新塞回 Gateway，破坏 packet boundary。

### Q: 为什么要先持久化再 push？

因为 push 只代表在线投递尝试，不能作为可靠存储。先落库才能保证对方离线、推送失败或客户端重连后仍可恢复消息。

### Q: HTTP 和 WS 为什么能共用后端逻辑？

两者最终都构造成 `MessagePacketRequest`。HTTP payload 是 `application/json`，WS payload 是 `im.message.SendMessageRequest` protobuf，具体解析都在 `MessagePacketDispatcher`，所以 service 业务逻辑可以复用。

### Q: 当前还缺什么生产级能力？

缺 client_msg_id 幂等去重、会话 sequence、设备级投递日志、客户端 ACK 驱动的 delivered/read、跨 Gateway 在线路由和更完整的消息类型体系。
