# MyChat 高频面试问题与回答

日期：2026-06-23

## Q1 项目整体类

### Q1.1 简单介绍一下 MyChat？

```text
MyChat 是一个 C++ 分布式 IM 后端项目。它用 Gateway 统一接入 HTTP 和 WebSocket，
后端按 User、Message、Friend、Group、Push 拆分服务边界。当前功能覆盖注册登录、
个人资料、好友、单聊、群聊、在线推送和离线补偿，并配有 React Web 验证客户端。

架构上最核心的是 Gateway 统一 packet 链路：HTTP/WS 请求都会被转换成 UnifiedMessage，
再由 cmd_id 路由到对应服务。Gateway 只负责鉴权、路由、连接管理和 packet 转发，
业务 protobuf 解析和业务判断都在服务侧完成。
```

### Q1.2 这个项目解决了什么问题？

```text
它不是单纯做一个聊天页面，而是验证一个 C++ IM 后端如何从单机开发演进到分布式服务：

1. 客户端只依赖 Gateway，不直接感知后端服务地址。
2. HTTP 和 WebSocket 共享同一套鉴权、路由和服务转发模型。
3. 服务既可以本地进程内运行，也可以通过 gRPC ForwardPacket 独立部署。
4. 消息先持久化再推送，Push 失败后能通过历史/离线拉取补偿。
```

### Q1.3 为什么用 C++ 做 IM 后端？

```text
IM 后端天然涉及网络 IO、连接管理、二进制协议、并发调度、服务边界和性能压测。
C++ 能比较直接地展示这些底层工程能力。这个项目的重点不是“用 C++ 写聊天软件”，
而是把 C++ 项目做成有网关、有服务拆分、有存储、有 gRPC 边界、有压测依据的工程。
```

## Q2 Gateway 与架构类

### Q2.1 为什么需要 Gateway？

```text
如果客户端直接调用 User、Message、Friend、Group、Push，客户端就需要知道每个服务的地址，
还要分别处理 token、错误码和网络状态。WebSocket 连接也会变得难管理。

Gateway 把客户端接入问题统一收口：对外暴露 HTTP/WS，对内通过 cmd_id 和 packet 路由到服务。
这样后端服务的部署方式变化不会影响客户端。
```

### Q2.2 当前 Gateway 主链路是什么？

```text
HTTP / WebSocket
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> GatewayCommandHandlerRegistry
-> GatewayRuntimeRegistry
-> local service packet dispatcher 或 remote ForwardPacket gRPC
```

关键点：

- `MessageParser` 负责把不同协议入口转成统一消息。
- `MessageProcessor` 负责统一调度和执行。
- `GatewayCommandHandlerRegistry` 负责 cmd_id 到 handler 的注册。
- `GatewayRuntimeRegistry` 负责运行时选择本地 endpoint 还是远程 endpoint。
- service packet dispatcher 才解析业务 payload。

### Q2.3 为什么不再使用 controller/client facade 作为主架构？

```text
旧 controller/client facade 模式的问题是：每新增一个服务或接口，Gateway 就要增加 controller、
client、remote client 和大量 DTO/mapper。Gateway 会越来越像业务聚合层，和“只做接入与路由”
的目标冲突。

现在改成统一 packet boundary 后，Gateway 只处理 envelope、cmd_id、auth metadata 和
transport metadata。业务字段不在 Gateway 展开，服务侧自己解析 protobuf。
这让 HTTP、WebSocket、local 和 remote 可以共享同一套服务入口。
```

### Q2.4 Gateway 会不会太重？

```text
Gateway 确实是关键入口，但它的职责边界现在更清晰：

1. 协议接入：HTTP / WebSocket。
2. 鉴权：token 校验和 actor uid 注入。
3. 连接管理：WebSocket session 生命周期。
4. 路由：cmd_id 到服务 endpoint。
5. 转发：packet 转发和响应回写。

它不处理好友申请状态机、群成员权限、消息持久化这些业务逻辑。
```

### Q2.5 local 和 remote 模式如何兼容？

```text
Gateway 外部契约不变，差异在 RuntimeRegistry 选择的 endpoint：

- local：调用进程内服务的 packet dispatcher。
- remote：调用服务的 ForwardPacket gRPC。

两种模式都以 packet 作为边界，因此 Gateway 不需要理解服务内部业务 DTO。
```

## Q3 消息系统类

### Q3.1 为什么消息要先持久化再推送？

```text
如果先推送再落库，推送成功但数据库写入失败，用户会看到一条刷新后消失的消息。
这对 IM 来说比延迟更难接受。

MyChat 选择先落库再推送。持久化成功后返回 ack，Push 是 best-effort。
Push 失败不回滚消息，接收方后续可以通过历史消息或离线拉取补偿。
```

### Q3.2 WebSocket ack、delivered、read 有什么区别？

```text
ack：服务端已收到并持久化发送请求。
delivered：消息至少成功投递到接收方一个在线 session。
read：接收方明确阅读后标记。

ack 是发送侧语义，delivered/read 是接收侧状态。
```

### Q3.3 如何防止客户端伪造 sender_uid？

```text
Gateway 不信任请求体中的 sender_uid。发送者身份来自 access token。
Gateway 鉴权后把 actor uid 作为 metadata 传给服务，服务侧以这个 actor uid 作为当前用户。
这样客户端即使在 payload 中伪造 sender_uid，也不会被当作可信身份。
```

### Q3.4 离线消息怎么实现？

```text
离线消息不是单独的队列表，而是持久化消息中的状态视图。
消息写入后初始是 SENT；在线 Push 成功后标记 DELIVERED。
用户上线或进入会话时拉取未投递/历史消息，服务根据 receiver、状态和时间查询。
```

### Q3.5 当前性能基准如何？

```text
2026-06-23 重构后基准：
- HTTP 500 VU ramp：46296 次请求，约 306.86 req/s，失败率 0，p95 约 3.75ms。
- WebSocket 200 连接：200/200 成功，建连 p95 约 14ms。
- WebSocket 200 用户、每 250ms 一条消息：约 714.4 msg/s，RTT p95 约 35.91ms，错误 0。

这说明当前 MVP 已经具备稳定验证能力，但还不能宣称生产级容量。
后续要继续做多机、多 Gateway、消息队列和分段链路观测。
```

## Q4 好友关系类

### Q4.1 好友申请和好友关系如何建模？

```text
Friend Service 维护好友关系和申请生命周期。当前通过状态区分 PENDING、ACCEPTED、
REJECTED、BLOCKED 等关系。申请被接受后状态流转为 ACCEPTED，不需要把申请记录迁移到另一张表。
```

### Q4.2 好友申请有哪些校验？

```text
服务侧会校验：
1. 不能添加自己。
2. 目标用户必须存在。
3. 不能重复申请。
4. 不能反向重复申请。
5. 只有被申请者可以处理申请。
6. 只有 pending 状态能被处理。
```

### Q4.3 为什么用户搜索不放在 Friend Service？

```text
用户资料属于 User Service，好友关系属于 Friend Service。
添加好友的用户体验是“先搜索用户，再发申请”，但搜索用户本身不是好友关系逻辑。
因此搜索由 User Service 提供，Friend Service 只负责创建和处理关系。
```

## Q5 群聊类

### Q5.1 群消息如何发送？

```text
Group Service 校验成员身份并持久化群消息，然后返回消息结果和 push_events。
Gateway 不展开群成员业务逻辑，只把服务返回的 push_events 交给 Push 服务 fanout。
默认不向发送者重复推送。
```

### Q5.2 大群 fanout 有什么问题？

```text
当前 MVP 适合中小规模验证。大群 fanout 会带来成员查询、在线状态查询、消息投递和状态标记的放大。
后续优化方向包括：
- 群成员缓存。
- 在线成员预过滤。
- Push 异步队列。
- 批量 session 查询和批量投递。
- 热点大群特殊通道。
```

### Q5.3 群资料 UI 中哪些是后续扩展？

```text
当前后端已经支持建群、搜索、加入、退出、成员列表、群消息和历史。
群公告、群昵称、群备注、免打扰、置顶、复杂通知设置等属于后续产品能力，
Web 端可以先作为展示形态存在，但不要在面试中说成后端已经完整实现。
```

## Q6 Push 与分布式类

### Q6.1 为什么 Push 要独立出来？

```text
Push 不只是 Message 的附属逻辑。单聊、群聊、系统通知都可能触发在线投递。
把 PushRuntime 和 FanoutPolicy 独立出来，可以统一多端 fanout、payload 构造、
投递结果处理和未来的异步队列/重试策略。
```

### Q6.2 为什么远程 Push 需要回调 Gateway？

```text
WebSocket session 在 Gateway 进程里。远程 Push 服务不能直接访问 Gateway 内存。
所以 Push 通过 GatewayPushDeliveryService 回调 Gateway：
ListUserSessions 查连接，SendSessionPayload 发送 payload，MarkMessageDelivered 标记投递。
```

### Q6.3 多 Gateway 如何扩展？

```text
当前 MVP 主要验证单 Gateway。多 Gateway 需要 uid -> gateway 实例的全局连接索引。
用户连接 Gateway 后写入 Redis 或服务注册中心，Push 先查用户在哪个 Gateway，再回调对应 Gateway。
连接断开时清理索引，并处理过期和异常断连。
```

## Q7 存储与工程类

### Q7.1 Redis 和 PostgreSQL 如何分工？

```text
Redis 存 token、session、撤销状态等临时状态，特点是高频、可过期、可重建。
PostgreSQL 存用户、消息、好友、群组等持久业务数据，特点是需要事务、查询和长期保存。
```

### Q7.2 为什么用 ODB？

```text
ODB 让 C++ domain model 和 PostgreSQL 表结构建立映射，减少手写 SQL 样板。
代价是工具链更重，生成文件和版本依赖需要管理。当前项目通过 ODB 2.5.0、CMake gate、
迁移脚本和 focused tests 来降低这部分风险。
```

### Q7.3 为什么 proto 放在 common/proto？

```text
common/proto 是唯一权威 IDL 来源。Gateway、服务端、远程 adapter 和测试都使用同一套生成文件。
这样可以避免多个目录里生成文件漂移，也方便通过 generate_proto 统一再生成。
```

## Q8 不足与规划类

### Q8.1 当前项目最大不足是什么？

```text
从生产角度看，主要不足是多 Gateway 连接路由、消息队列化、监控告警、部署编排和富媒体能力还没做。
从面试项目角度看，当前已经是完整可演示 MVP，下一步重点是把架构讲清楚、把压测数据讲明白，
再补少量高价值稳定性验证。
```

### Q8.2 如果继续优化，你优先做什么？

```text
1. 多 Gateway 连接路由：解决横向扩展。
2. Push 异步队列：削峰填谷，降低发送链路尾延迟。
3. 消息序列号：增强会话内顺序语义。
4. 对象存储：支持头像、图片、文件。
5. 更完整压测和链路 tracing：定位瓶颈和给容量规划提供依据。
```
