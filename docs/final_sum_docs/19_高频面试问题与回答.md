## Q2 Gateway 与架构类

### Q2.1 为什么需要一个 Gateway？直接让客户端调后端服务不行吗？

```text
不行。如果客户端直接调后端服务，会面临几个问题：

1. 客户端需要知道 User、Message、Friend、Group、Push 五个服务地址。
2. token 鉴权和 session 管理逻辑要分散到每个服务。
3. WebSocket 连接需要每个服务各自维护。
4. 服务部署方式变化（比如从 local 拆成远程）会影响客户端。

Gateway 把这些问题收拢了：客户端只连 Gateway，鉴权和连接管理集中处理，
后端服务如何部署对客户端透明。代码入口在 gateway/gateway_server/gateway_server.cpp。
```

### Q2.2 Gateway 为什么同时处理 HTTP 和 WebSocket？不分开？

```text
HTTP 和 WebSocket 虽然协议不同，但属于同一层职责——客户端接入。
放在一起的好处：

1. 客户端只需要连一个端口（或两个端口但同一个进程）。
2. token 鉴权逻辑复用，HTTP 和 WS 使用同一套 auth manager。
3. WebSocket session 由 Gateway 管理，Push 服务可以通过本地或远程方式
   使用这些 session。
4. 后端服务拆分不影响 Gateway 的外部协议。

分开反而会增加部署复杂度，且鉴权和 session 管理要各自实现一遍。
```

### Q2.3 local/remote facade 是怎么实现的？

```text
每个服务定义一个抽象的 Client 接口。比如 UserClient 定义 register、login、
getProfile 等方法。然后做两个实现：

- LocalUserClient：直接调用进程内 UserService 的方法。
- RemoteUserClient：构造 gRPC 请求、调用 stub、处理响应和错误。

GatewayServer 初始化时从配置读取 user.mode，根据值 local 或 remote
创建对应的 client 实例。Controller 只依赖 UserClient 接口。

代码入口：
- gateway/http/user_client.cpp（接口定义 + local 实现）
- gateway/http/remote_user_client.cpp（gRPC 实现）
```

### Q2.4 facade 模式的优缺点？

```text
优点：
- Controller 不感知部署方式，切换只改配置。
- 本地调试时不需要启动远程服务。
- 单元测试时可以 mock client 接口。
- gRPC 错误映射集中在 remote client，不污染 controller。

缺点：
- 每增加一个服务需要同时维护 local 和 remote 两套实现。
- 接口设计需要同时兼容 local 和 remote 的语义差异（比如超时、重试策略）。
- 远程模式引入了 proto 转换的样板代码。
```

## Q3 消息系统类

### Q3.1 消息为什么要先落库再推送？

```text
这是消息可靠性的核心取舍。

如果先推送再落库：推送成功但落库失败，用户看到消息但刷新后消息丢了。
如果先落库再推送：落库成功即表示服务端已接收，推送是 best-effort。
推送失败不回滚消息，用户后续可以拉取。

MyChat 选择后者。从 MessageWsHandler 的代码可以看到（gateway/ws/message_ws_handler.cpp），
它先调 MessageClient::send_message 持久化，然后才调 PushNotifier 触发推送。
```

### Q3.2 在线推送失败怎么办？

```text
MyChat 的 Push 是 best-effort 语义。推送失败时：

1. 不重试（当前 MVP 没有重试队列）。
2. 不回滚已持久化的消息。
3. 接收者下次上线会通过离线消息拉取（MessageService::pull_offline）拿到未投递消息。
4. sender 已经收到 ack，不影响 sender 侧体验。

如果后续需要提高推送成功率，可以引入重试队列和回退化推送通道。
```

### Q3.3 离线消息是怎么实现的？

```text
消息落库时记录 receiver_uid、create_time 和 status。status 初始为 SENT。
Push 成功后调用 MarkDelivered 标记为 DELIVERED。

用户上线后调 GET /api/v1/messages/offline，Gateway 转发到 MessageService::pull_offline。
pull_offline 查询 status = SENT 且 receiver_uid = 当前用户的消息，
按时间倒序返回一定数量。客户端收到后展示，并调用 MarkDelivered 标记。

消息表中没有单独的"离线队列"，离线消息本质上是"已持久化但未投递的消息"。
```

### Q3.4 WebSocket ack 和已读回执有什么区别？

```text
WebSocket ack：表示服务端已收到并保存了消息。Gateway 在 MessageWsHandler 中
接收到 WebSocket 消息后，解析 protobuf、校验 token、调 MessageService 持久化，
持久化成功后返回 CMD_SEND_MESSAGE ack。

已读回执：表示接收者看到了消息。目前通过 MessageService::mark_read 实现，
接收者阅读消息后调 HTTP 接口标记阅读状态。

简单来说：ack = 服务端接收，已读 = 用户已看。
当前 ack 已实现，已读标记在 MVP 中可用，但没有完整实现"对方已读"的 UI 展示。
```

### Q3.5 如何防止客户端伪造 sender_uid？ (message_http_controller.cpp::handle_send::93行)

```text
Gateway 不在 WebSocket 或 HTTP 的请求体中信任客户端传入的 sender_uid。
sender 身份来源是 access token 中的 uid：

1. 客户端发送消息时，payload 的 sender_uid 字段在服务端被忽略。
2. Gateway 在验证 token 后，从 token 中提取当前 uid。
3. MessageService::send_message 的 sender_uid 来自 Gateway 的 auth 校验结果，
   不来自客户端请求体。

代码里 token 验证通过 MultiPlatformAuthManager::verify_access_token 完成，
从 JWT 中解析出 user_id。
```

## Q4 好友关系类

### Q4.1 好友申请和好友关系分开存储吗？

```text
同一个表 im_friends 处理。status 字段区分：
PENDING = 申请中，ACCEPTED = 好友，REJECTED = 已拒绝，BLOCKED = 已拉黑。

没有把"申请"和"关系"拆成两张表。一张表加状态字段的好处是：
- 申请被接受后直接改状态，不需要迁移数据。
- 查询"是否已经是好友"只需要查 status = ACCEPTED。
- 重复申请、反向重复申请都在同一张表检查唯一索引。

代码入口：services/odb/friend.hpp。
```

### Q4.2 好友申请的服务端校验有哪些？

```text
FriendService::send_request 中有以下校验：

1. 不能添加自己（requester == target）。
2. 目标用户必须存在（在 UserRepository 中检查）。
3. 不能重复申请（已有 PENDING 状态的记录）。
4. 不能反向重复申请（target 已经向你发过 PENDING 申请）。

respond_to_request 中：
1. 只有 target 用户能处理申请。
2. 申请必须是 PENDING 状态（不能重复处理）。

代码入口：services/friend/friend_service.cpp。
```

## Q5 群聊类

### Q5.1 群消息怎么推送给所有在线成员？

```text
群消息发送链路：

1. 客户端调 POST /api/v1/groups/messages/send 或 WebSocket。
2. GroupMessageService 持久化消息到 im_group_messages。
3. Gateway 从 GroupService::list_members 获取群成员列表。
4. 排除发送者本人后，对每个成员调 PushNotifier::notify_user。
5. PushRuntime 查询每个成员在 Gateway 的在线 session，通过 fanout policy
   决定投递到哪些 session，发送 protobuf 编码的 CMD_PUSH_MESSAGE。

代码入口：
- gateway/http/group_message_http_controller.cpp
- services/group/group_message_service.cpp
```

### Q5.2 大群 fanout 的性能问题怎么考虑？

```text
当前 MVP 没有做大群优化。群消息 fanout 是逐个成员调用 PushNotifier，
对每个成员独立查询 session 和发送。100 人群聊需要 99 次推送操作。

后续优化方向：
- 批量推送：PushRuntime 支持批量 session 查询和发送。
- Push 异步化：消息持久化后立即返回 ack，推送通过后台队列异步完成。
- 在线状态过滤：先筛选在线成员，只对在线成员尝试推送。
- 群成员缓存：减少每次群消息都查数据库的频次。
```

## Q6 分布式与 gRPC 类

### Q6.1 gRPC 在项目中的作用是什么？

```text
gRPC 承担内部服务间调用的契约和传输层：

1. Gateway 调用 User/Message/Friend/Group 服务使用 gRPC（remote 模式）。
2. Push 服务回调 Gateway 使用 gRPC（GatewayPushDeliveryService 的 callback）。
3. 各个服务接口通过 proto 文件定义，作为跨进程契约。

当前 proto 定义在 common/proto/ 目录下，每个服务一个 proto 文件。
gRPC 和 protobuf 一起提供了一套完整的 IDL 和编解码方案。
```

### Q6.2 为什么 Push 服务需要回调 Gateway？

```text
因为 WebSocket 连接由 Gateway 持有，远程 Push 服务进程不能访问 Gateway 的内存。

Push 服务通过 gRPC 调 Gateway 暴露的 GatewayPushDeliveryService：
- ListUserSessions：查用户当前有哪些 WebSocket 连接。
- SendSessionPayload：向指定 session 发送推送 payload。
- MarkMessageDelivered：标记消息已投递。

如果不这样做，要么让 Gateway 和 Push 共享连接状态（耦合），
要么让 Push 也持有 WebSocket 连接（重复管理）——都不够好。

代码入口：
- services/push/push_runtime.cpp（Push 侧）
- gateway/push/gateway_push_delivery_service.cpp（Gateway 侧）
```

### Q6.3 多 Gateway 实例下 Push 怎么找到用户连接？

```text
当前 MVP 假设单 Gateway 实例，没有实现多 Gateway 连接路由。

如果需要多 Gateway，需要维护一个 uid -> gateway 的全局索引。
常见方案：

1. Redis 维护 uid -> gateway_host 映射，用户连上 Gateway 时注册，
   断开时注销。
2. Push 服务先查这个索引，找到目标 Gateway，再调对应 Gateway 的 callback。
3. 如果索引查不到，说明用户离线，不走在线推送。

当前项目没有实现这个索引，属于 MVP 边界。
```

## Q7 存储与数据类

### Q7.1 Redis 和 PostgreSQL 怎么分工？

```text
Redis：token/session 等临时、高频、有过期时间的状态。
- refresh token 元数据（TTL 控制生命周期）。
- token 撤销状态（黑名单）。
- 用户 session 索引（user -> sessions）。

PostgreSQL：所有持久业务数据。
- 用户账号和资料（im_users）。
- 消息记录（im_messages）。
- 好友关系和申请（im_friends）。
- 群资料（im_groups）和群成员（im_group_members）。
- 群消息（im_group_messages）。

选择逻辑：数据是否需要 TTL、是否可以丢失后重建、
是否需要复杂查询和事务。
```

### Q7.2 ODB 好用吗？有什么坑？

```text
ODB 的优点是减少了手写 SQL 的样板代码，C++ 对象和数据库表之间的映射
由生成代码完成。在项目初期构建 repository 层时效率较高。

坑和局限性：
- 工具链复杂：ODB 编译器版本需要严格匹配，构建需要额外编译步骤。
- 编译时间增加：ODB 生成代码量较大。
- 复杂查询不够灵活：涉及多表 join 或特殊 SQL 时仍然需要手写。
- 社区活跃度一般，问题排查时依赖官方文档。

总的来说，ORM 层适合 MVP 阶段的快速迭代，生产阶段可以考虑纯 SQL
配合连接池和查询构建器。
```

### Q7.3 schema migration 为什么不放服务启动里自动执行？

```text
当前 migration 是显式执行的 SQL 脚本，不放在服务启动时自动执行。

理由：
- 服务启动时自动迁移可能导致意外的 schema 变更。
- 多实例部署时多个服务同时 migration 可能导致竞态。
- 迁移失败会导致服务启动失败，回滚更复杂。

显式迁移更适合当前 MVPA 阶段。生产环境可以使用 Flyway 或类似工具
管理迁移版本。
```

### Q8.3 当前压测结果怎么样？

```text
当前压测不是生产 SLA，但已经能支撑 MVP 性能分析。

最新一轮完整复测里：
- HTTP 500 VU 阶梯压测，46230 次请求，失败率 0%，p95 5.81ms。
- WebSocket 200 连接建连失败 0，connect p95 约 15ms。
- 200 用户每 1s 发一条消息，RTT p95 12.60ms，错误 0。
- 200 用户每 500ms 发一条消息，RTT p95 21.92ms，错误 0。
- 200 用户每 250ms 发一条消息时，RTT p95 上升到 2628.05ms，说明进入吞吐拐点。

所以我的表述会比较克制：当前系统在 200 在线用户、0.5-1s 一条消息的场景下实时链路流畅；
但还不能说具备商业 IM 的百万连接能力。
```

### Q8.4 压测中发现过什么性能问题，怎么修的？

```text
两个比较典型：

第一个是 HTTP 高并发 timeout。最开始 500 VU ramp 下大量请求卡到 60s。
我先单独压 /health，发现纯内存接口也超时，说明不是数据库或业务逻辑慢。
再通过禁用连接复用验证是 cpp-httplib 同步 worker 被 keep-alive 空闲连接占用。
最后显式配置线程池、队列上限，并把 keep_alive_max_count 收敛到 1，
HTTP p95 收敛到 5.81ms，失败率 0。

第二个是 WSS 高负载下 RTT 秒级。原链路里每条消息会 std::async，
Gateway 还会再创建 detached thread 等 future。后来改成固定线程池执行，
并加 max_ws_inflight_messages 做初步背压。稳定区间下 200 用户消息 RTT p95 保持在 12-22ms。
```

主要差距：

### Q9.2 如果继续优化，优先级怎么排？

```text
第一梯队（核心体验）：
1. 消息序列号——按会话递增 seq，增强消息顺序语义。
2. 多 Gateway 连接路由——支持多实例水平扩展。
3. 头像和图片对象存储。

第二梯队（功能完善）：
4. 消息撤回和已读回执完整 UI。
5. 群管理（踢人、禁言、转让）。
6. Push 重试队列和退避策略。

第三梯队（工程化）：
7. 性能压测和容量评估。
8. Docker Compose 生产配置。
9. 服务发现集成。

优先级排序依据：用户体验影响 > 功能完整度 > 工程化。
```
