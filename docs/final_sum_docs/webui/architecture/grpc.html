gRPC 服务边界与远程调用

MyChat 用统一 packet 契约把 Gateway 外部 HTTP/WS 协议和内部服务部署方式解耦；local 模式直接调用 service packet dispatcher，remote 模式通过 gRPC ForwardPacket 到独立服务进程，两种模式共享同一套业务解析和业务规则。

§10.1

当前权威模型

✕ 旧模型

Gateway Controller

→ Local*Client / Remote*Client

→ Service

✓ 当前模型

Gateway command handler

→ GatewayRuntimeRegistry

→ local service packet dispatcher

或 → remote service adapter

→ XxxService::ForwardPacket

→ service server packet dispatcher

一句话Gateway 只认识统一 packet，不认识业务 payload；Service 自己解析 payload、执行业务、构造响应。
§10.2

为什么用 ForwardPacket

如果 Gateway 直接调用一堆业务 RPC（SendMessage / GetConversation / CreateGroup / JoinGroup / SendRequest…），Gateway 就需要解析业务 payload、构造业务 request DTO、映射业务 response、为每个服务维护一套 local/remote 业务方法 —— 这会让 Gateway 重新变成业务编排层，违背当前边界。

直接调业务 RPC ✕
  • Gateway 解析业务 payload
  • 构造业务 request DTO
  • 映射业务 response
  • 每服务维护一套 local/remote 业务方法
  • Gateway 重新变成业务编排层
统一 ForwardPacket ✓
  • Gateway 只认识 packet
  • Service 自己解析 payload
  • local 和 remote 共用同一个 dispatcher
  • 外部 HTTP/WS 协议不受部署方式影响
§10.3

统一 packet 模型

HTTP
HTTP request
  → MessageParser → UnifiedMessage
  → command handler
  → XxxPacketRequest(
      type_name="application/json",
      payload=raw json)
  → local dispatcher 或 remote ForwardPacket
  → XxxPacketResponse(
      payload=json, http_status)
WebSocket
WebSocket envelope
  → MessageParser → UnifiedMessage
  → command handler / MessageWsHandler
  → XxxPacketRequest(
      type_name=protobuf type,
      payload=protobuf bytes)
  → local dispatcher 或 remote ForwardPacket
  → XxxPacketResponse(protobuf)
  → Gateway encode envelope
§10.4

GatewayRuntimeRegistry

Gateway 当前的运行时依赖表。handler 只拿一个收敛的 context，比旧的展开式 service/client 成员更清晰，也减少了 handler 注册函数的参数爆炸。

cpp
struct GatewayCommandHandlerContext {
    const GatewayRuntimeRegistry* runtime = nullptr;
};

Remote 链路（以 User 为例）

User remote 链路 · 可交互
特殊边界User service 返回 UserAuthTokenHint，Gateway 根据 hint 生成 access/refresh token —— token 签发仍属于 Gateway 鉴权边界。Message service 返回 MessagePushEvent，Gateway 只消费并转交 PushNotifier。Group service 返回 repeated GroupPushEvent，Gateway 只遍历调用 PushNotifier，不自己算 fanout。
§10.6

Push 边界 · 最典型的分布式取舍

Push 和其他业务服务不同，因为 WebSocket session 在 Gateway 进程内。

Remote Push 链路 · 可交互
为什么需要 callback远程 push_server 无法直接访问 Gateway 内存中的 WebSocket session map，所以它只能通过 Gateway 暴露的 delivery service 回调完成实际投递。
§10.7

Local / Remote 模式对比

Local 模式
local
Gateway command handler
  → RuntimeRegistry local service pointer
  → XxxPacketDispatcher::handle(packet)
  → Service core → Repository/Redis/Push
  • 启动简单、本地调试方便
  • 不需要多个服务进程
  • 适合单机 MVP 验证
Remote 模式
remote
Gateway command handler
  → RemoteXxxServiceAdapter::forward(packet)
  → gRPC ForwardPacket → 独立服务进程
  → XxxPacketDispatcher::handle(packet)
  • 更接近真实分布式服务
  • 分进程部署、验证 gRPC deadline/网络错误
  • Gateway 对外 HTTP/WS 协议不变
  • 代价：启动排查复杂、需 endpoint 配置、多进程日志
一致性保证local 和 remote 都使用同一个 XxxPacketRequest/Response 契约，并最终进入同一个 XxxPacketDispatcher::handle。区别只是 dispatcher 在 Gateway 进程内执行，还是在远程 service 进程内执行。
§10.8

Remote Service Adapter 的职责

adapter 职责 ✓
  • 接收 XxxPacketRequest
  • 设置 gRPC deadline
  • 调用 ForwardPacket
  • 检查 transport status
  • 返回 XxxPacketResponse
adapter 不应该 ✕
  • 暴露 send_text_message / create_group 等业务方法
  • 解析业务 JSON/protobuf payload
  • 构造业务 DTO
  • 做业务错误 mapper 之外的业务逻辑
§10.9

proto 当前状态与优化方向

当前 user/message/friend/group.proto 同时包含三类内容：

packet contract

XxxPacketRequest / Response

历史业务 RPC

Register、SendMessage、CreateGroup 等

业务 DTO

RegisterRequest、MessageBody、GroupInfo 等

长期优化拆出 *_packet.proto 或统一 gateway_packet.proto，让 Gateway 编译期只接触 packet contract，不接触业务 request/response DTO。

§10.11

面试追问 · 点击展开

因为直接调用业务 RPC 会迫使 Gateway 解析 payload、构造业务 DTO、维护 response mapper。当前项目希望 Gateway 只做协议入口和 packet 转发，所以统一使用 ForwardPacket 把业务解析下沉到 service。

local 和 remote 都使用同一个 XxxPacketRequest/XxxPacketResponse 契约，并最终进入同一个 XxxPacketDispatcher::handle。区别只是 dispatcher 在 Gateway 进程内执行，还是在远程 service 进程内执行。

packet contract 和业务 proto 仍混在同一个文件里，Gateway 编译期仍可能接触业务 DTO。后续应拆出独立 packet proto，让 Gateway 只依赖 packet contract。

因为 Push 的实际投递对象是 Gateway 内存里的 WebSocket session。远程 Push 只能调度投递，最终发送 payload 必须通过 Gateway callback 回到 Gateway 完成。
