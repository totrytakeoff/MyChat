# MyChat 核心技术栈与选型分析

日期：2026-06-23

## 1. 技术栈总览

```text
C++20
Boost.Asio / Boost.Beast
HTTP / WebSocket
Protobuf / gRPC
Redis / hiredis
PostgreSQL / ODB
OpenSSL
CMake
GoogleTest / CTest
React / TypeScript / Vite
```

| 技术 | 项目角色 |
| --- | --- |
| C++20 | 后端主体语言 |
| Boost.Asio/Beast | Gateway 网络和 WebSocket 基础 |
| HTTP | 注册、登录、资料、好友、群、历史查询等请求响应接口 |
| WebSocket | 单聊实时发送、ack、在线推送 |
| Protobuf | WebSocket envelope、gRPC IDL、packet contract |
| gRPC | User/Message/Friend/Group/Push 远程服务边界 |
| Redis | refresh token、撤销状态、在线/session 辅助状态 |
| PostgreSQL | 用户、消息、好友、群组持久化 |
| ODB | C++ ORM，降低对象/SQL 映射成本 |
| OpenSSL | PBKDF2、随机数、TLS |
| GoogleTest/CTest | 单测、dispatcher 测试、remote smoke |
| React/Vite | Web 验证客户端 |

## 2. C++ 的价值

MyChat 使用 C++ 不是因为聊天软件只能用 C++，而是为了体现：

- 网络服务开发能力。
- 长连接和对象生命周期管理。
- Protobuf/gRPC/Redis/PostgreSQL 在 C++ 工程中的整合。
- 多模块 CMake 工程组织。
- 面向服务边界的后端架构设计。

面试表达：

```text
我用 C++ 实现 IM 后端，是为了把连接管理、协议编解码、实时推送和服务间通信这些后端基础能力做出来，
而不是只写一个 CRUD 服务。
```

## 3. Gateway 技术选型

Gateway 使用 HTTP + WebSocket：

- HTTP 适合注册、登录、资料、好友、群、历史查询。
- WebSocket 适合实时消息、在线 push、ack。

两者统一进入：

```text
MessageParser
-> UnifiedMessage
-> MessageProcessor
-> GatewayCommandHandlerRegistry
```

这比旧的“每个模块一个 HTTP controller”更贴合当前设计：

- 统一鉴权边界。
- 统一 route/cmd 分发。
- 统一 local/remote packet 转发。
- Gateway 不解析业务 payload。

## 4. Protobuf 与 gRPC

Protobuf 当前有两类用途：

- WebSocket envelope 和消息 payload。
- gRPC 服务 IDL 和 packet contract。

当前 gRPC 主边界是 `ForwardPacket`：

```text
UserService.ForwardPacket(UserPacketRequest)
MessageService.ForwardPacket(MessagePacketRequest)
FriendService.ForwardPacket(FriendPacketRequest)
GroupService.ForwardPacket(GroupPacketRequest)
```

设计价值：

- Gateway 只认识 packet。
- Service 自己解析 payload。
- local 和 remote 共用同一个 packet dispatcher。
- 外部 HTTP/WS 协议不受服务部署方式影响。

待优化：

- 当前 packet contract 和业务 proto 仍混在同一个 `.proto` 文件中。
- 后续应拆出 `*_packet.proto` 或统一 `gateway_packet.proto`。

## 5. Redis 与 PostgreSQL

PostgreSQL/ODB 保存持久业务状态：

- 用户。
- 单聊消息。
- 好友关系。
- 群资料。
- 群成员。
- 群消息。

Redis 保存临时和高频状态：

- refresh token 元数据。
- token 撤销状态。
- token 索引。
- session/在线状态辅助数据。

选型原则：

```text
能恢复、需要事务和长期保存的数据放 PostgreSQL；
有 TTL、高频访问、临时状态的数据放 Redis。
```

## 6. Push 设计

Push 技术点：

- `PushNotifier` 抽象本地/远程推送。
- `PushRuntime` 统一 fanout、payload 构造和投递结果处理。
- `FanoutPolicy` 支持全部 session、平台过滤、最新 session。
- remote push 通过 `GatewayPushDeliveryService` callback 回到 Gateway 投递。

核心原因：

```text
WebSocket session 在 Gateway 内存中，远程 Push 服务不能直接访问。
```

## 7. 测试与压测

测试体系：

- GoogleTest/CTest 覆盖 service core。
- packet dispatcher 覆盖 Gateway/service packet 边界。
- remote adapter 和 gRPC smoke 验证 remote 模式。
- Gateway HTTP/WS smoke 验证入口。
- benchmark 工具验证 HTTP/WSS 稳定性。

重构后压测基准：

- WSS `714.4 msg/s`，p95 `35.91ms`。
- HTTP `306.86 req/s`，失败率 `0.00%`，p95 `3.75ms`。

## 8. Web 客户端

React Web 客户端不是核心技术炫技点，而是验证后端：

- 注册登录。
- 好友申请。
- 单聊实时消息。
- 群聊。
- 个人资料和群资料。
- WebSocket 推送。

它证明后端不是只靠 curl 可用，而能被真实 IM 交互驱动。

## 9. 面试表达

可以这样总结：

```text
MyChat 的技术选型围绕 IM 后端的几个核心问题展开：
Gateway 用 C++/Asio/Beast 处理 HTTP 和 WebSocket；
内部用 Protobuf/gRPC 定义服务边界；
Redis 处理 token 和在线临时状态；
PostgreSQL/ODB 处理持久业务数据；
Push 通过 Gateway callback 解决远程服务无法直接操作 WebSocket session 的问题。
```
