# MyChat 核心技术栈与选型分析

日期：2026-06-16

## 1. 文档目标

这份文档回答三个问题：

- MyChat 当前用了哪些核心技术。
- 每个技术分别解决什么工程问题。
- 面试时如何解释这些选型，而不是简单堆技术名词。

MyChat 的技术栈不是孤立组合，而是围绕 IM 后端的几个核心矛盾展开：

- 客户端接入协议和内部服务协议不同。
- 在线消息需要低延迟，离线消息需要可补偿。
- 用户、好友、群组、消息需要持久化。
- token、session、连接状态需要高频访问和过期控制。
- 项目既要能单机调试，又要能展示分布式服务边界。

## 2. 技术栈总览

当前核心技术栈：

```text
C++20
Boost.Asio / Boost.Beast
HTTP / WebSocket
Protobuf / gRPC
Redis / hiredis
PostgreSQL / ODB
OpenSSL
CMake / vcpkg
GoogleTest / CTest
React / TypeScript / Vite
Docker Compose / local scripts
```

对应职责：

| 技术 | 项目中的角色 | 解决的问题 |
| --- | --- | --- |
| C++20 | 后端主体语言 | 性能、资源控制、服务端工程能力展示 |
| Boost.Asio/Beast | Gateway 网络基础 | HTTP/WebSocket 接入、异步 IO |
| Protobuf | 统一消息结构 | 二进制协议、跨服务数据结构 |
| gRPC | 内部服务调用 | User/Message/Friend/Group/Push 远程边界 |
| Redis | 临时状态缓存 | refresh token、撤销状态、session 类状态 |
| PostgreSQL | 持久业务数据 | 用户、消息、好友、群组 |
| ODB | C++ ORM | 降低手写 SQL 与对象映射成本 |
| OpenSSL | 密码与随机数 | PBKDF2、随机 UID、token 相关安全基础 |
| GoogleTest/CTest | 自动化测试 | 服务逻辑、Gateway 控制器、remote client 验证 |
| React/Vite | Web 验证客户端 | 以真实 IM UI 验证后端闭环 |

## 3. 为什么后端主体使用 C++

### 3.1 当前价值

MyChat 的后端主体使用 C++，主要价值不是“为了聊天软件必须用 C++”，而是为了展示：

- 网络服务开发能力。
- 内存与对象生命周期管理能力。
- 编译期模块拆分与工程组织能力。
- gRPC、Redis、PostgreSQL 等基础设施在 C++ 项目中的整合能力。
- 面向服务边界的工程设计能力。

对于简历项目来说，这比只做一个常规 CRUD Web 服务更能体现后端基础。

### 3.2 适合面试表达

可以这样说：

```text
我选择 C++ 作为后端主体，是因为 IM 项目天然涉及连接管理、消息编解码、实时推送和服务间通信，
这些场景很适合体现 C++ 网络服务开发能力。同时项目通过 Gateway 和服务层拆分，把 C++ 项目做成
可测试、可演进的服务结构，而不是写成一个大 main 函数。
```

### 3.3 不要夸大的点

不要说：

- “C++ 一定比其他语言更适合所有 IM 场景。”
- “当前项目已经做到生产级高并发。”
- “当前网络层已经完成完整压测和容量评估。”

更准确的说法是：

```text
当前项目用 C++ 打通了完整 IM MVP 和分布式服务边界，后续还需要压测、连接路由和生产部署验证。
```

## 4. Gateway 为什么使用 HTTP + WebSocket

### 4.1 HTTP 的职责

HTTP 主要承载请求-响应型业务：

- 注册。
- 登录。
- 用户资料查询和更新。
- 用户搜索。
- 好友申请和处理。
- 群创建、搜索、加入、退出。
- 历史消息查询。

这些接口的特点是：

- 一次请求对应一次明确响应。
- 易于调试。
- 适合 Web 客户端调用。
- 适合用 JSON 表达结果和错误。

### 4.2 WebSocket 的职责

WebSocket 主要承载实时消息：

- 客户端在线发送单聊消息。
- 服务端返回发送 ack。
- 服务端向在线用户推送消息。
- 群消息 fanout 后推送到在线成员。

这些场景需要服务端主动下发，因此不适合只靠 HTTP 轮询。

### 4.3 为什么都放在 Gateway

Gateway 是统一外部入口，HTTP 和 WebSocket 都放在 Gateway 有几个好处：

- 客户端只需要知道一个接入层。
- token 鉴权逻辑集中。
- WebSocket session 由 Gateway 管理。
- 后端服务是否本地调用或远程 gRPC 调用，对客户端透明。
- Push 服务无需暴露给客户端。

面试表达：

```text
HTTP 负责资料、关系、历史查询等请求响应型接口，WebSocket 负责在线实时消息。
二者统一接入 Gateway，是为了集中处理鉴权、连接管理和服务编排，避免前端直接感知后端多个服务。
```

## 5. 为什么内部使用 Protobuf + gRPC

### 5.1 Protobuf 的作用

Protobuf 在 MyChat 中有两类用途：

- WebSocket 二进制消息的结构化载体。
- gRPC 服务间接口的 IDL。

例如：

- `common/proto/user.proto`
- `common/proto/message.proto`
- `common/proto/friend.proto`
- `common/proto/group.proto`
- `common/proto/push.proto`

它的价值：

- 字段结构明确。
- 跨进程、跨语言可扩展。
- 二进制编码比 JSON 更适合高频消息。
- 通过 `.proto` 明确服务边界和数据契约。

### 5.2 gRPC 的作用

gRPC 用于内部服务边界：

- Gateway -> User Service
- Gateway -> Message Service
- Gateway -> Friend Service
- Gateway -> Group Service
- Gateway -> Push Service
- Push Service -> GatewayPushDeliveryService callback

当前项目的关键不是“强行把所有东西拆远程”，而是保留 local/remote 双模式：

```text
Gateway Controller / Handler
-> Gateway Client Facade
-> Local Service 或 Remote gRPC Stub
-> Service Core / Repository / Runtime
```

这样可以兼顾：

- 本地开发效率。
- 单元测试便利。
- 分布式部署演示。
- 外部 HTTP/WebSocket 契约稳定。

### 5.3 面试可讲点

可以重点讲：

- `.proto` 如何定义服务边界。
- Gateway 为什么通过 facade 调用服务。
- local 模式和 remote gRPC 模式如何共享业务语义。
- remote client 如何把 gRPC 错误映射为服务层错误。
- Push 为什么额外需要 Gateway callback。

不要夸大：

- 当前还没有完整服务发现。
- 当前 remote endpoint 主要通过配置和脚本组织。
- 当前不是生产级服务治理平台。

## 6. 为什么使用 Redis

### 6.1 Redis 当前职责

Redis 负责临时、高频、可过期状态：

- refresh token 元数据。
- token 撤销状态。
- 用户 refresh token 索引。
- session/连接相关状态。
- 连接池复用验证。

这些数据不适合全部放 PostgreSQL：

- 有 TTL。
- 访问频繁。
- 状态生命周期短。
- 可由登录态或连接态重新构建。

### 6.2 Redis 与 JWT 的关系

当前认证体系可以理解为：

- access token 使用 JWT，便于 Gateway 本地验证。
- refresh token 存储在 Redis，便于过期、撤销和轮换。
- token 中包含 uid、device_id、platform 等信息。

这带来的好处：

- 常规请求验证 access token 不需要查数据库。
- refresh token 生命周期可以由 Redis TTL 控制。
- 可以支持撤销、设备维度和多平台策略。

### 6.3 面试可讲点

可以这样表达：

```text
我没有把所有登录态都放进 PostgreSQL，而是把 refresh token 和撤销状态放在 Redis。
因为这类状态访问频繁、天然有过期时间，而且和持久业务数据生命周期不同。
```

需要承认的边界：

- 当前 token 安全策略是 MVP 级别。
- 还没有完整登录风控、验证码、限流和设备管理 UI。
- Redis 故障下的降级策略还需要进一步设计。

## 7. 为什么使用 PostgreSQL + ODB

### 7.1 PostgreSQL 当前职责

PostgreSQL 保存持久业务数据：

- 用户账号与资料。
- 单聊消息。
- 好友申请和关系。
- 群资料。
- 群成员。
- 群消息。

这些数据必须可恢复、可查询、可审计，不能只放内存或 Redis。

### 7.2 ODB 的作用

ODB 是 C++ ORM，用于把 C++ 对象映射到 PostgreSQL 表。

当前模型主要位于：

```text
services/odb/user.hpp
services/odb/message.hpp
services/odb/friend.hpp
services/odb/group.hpp
services/odb/group_message.hpp
```

优势：

- 减少重复手写 SQL。
- 让 repository 层以对象形式表达业务数据。
- 便于 C++ 类型和数据库字段对应。

代价：

- ODB 工具链复杂。
- 生成代码需要纳入构建链路。
- 迁移和 schema 管理仍需要明确流程。

### 7.3 面试可讲点

可以说：

```text
PostgreSQL 负责持久业务状态，Redis 负责临时状态。ODB 让 C++ 服务层可以用对象方式操作持久模型，
但我也保留了显式 migration 脚本，不把建表和迁移隐式塞进服务启动流程。
```

不要说：

- “ODB 解决了所有数据库问题。”
- “当前已经有完整生产迁移体系。”

## 8. 为什么使用“先持久化再推送”

IM 消息链路最重要的取舍之一是：

```text
消息发送成功 = 服务端接收并持久化成功
在线推送 = best-effort
离线补偿 = 后续拉取
```

这样设计的原因：

- WebSocket 推送可能失败。
- 对方可能离线。
- Push 服务可能临时不可用。
- Gateway session 可能已经断开。

如果先推送再落库，可能出现客户端看到消息但服务端不可恢复的问题。

如果先落库再推送：

- 消息具备可恢复性。
- 推送失败不影响消息最终可见。
- 离线拉取可以补偿在线投递失败。

面试表达：

```text
MyChat 当前把“持久化成功”作为发送成功的核心语义，Push 是在线加速路径。
推送失败不会回滚消息，而是通过历史查询和离线拉取补偿。
```

## 9. 为什么 Push 独立出来

Push Service 的核心价值不是简单调用 WebSocket send，而是把在线投递抽象成独立能力：

- 统一构造推送 payload。
- 根据 fanout policy 选择投递 session。
- 支持 all sessions / platform filter / newest session。
- 成功投递后标记 delivered。
- 支持本地 Push 和远程 Push。

最关键的架构点：

```text
WebSocket session 归 Gateway 管。
远程 Push 不能直接访问 Gateway 内存。
所以 Push 通过 GatewayPushDeliveryService callback 完成 session 查询和投递。
```

这个设计适合面试展开：

- 为什么不能让 Push 直接持有 WebSocket 连接。
- Gateway callback 解决了什么边界问题。
- 这种方案在多 Gateway 下还需要如何演进。

当前边界：

- 当前 Push 是 best-effort。
- 多 Gateway 下的全局连接路由还需要服务发现或连接索引。
- delivered 标记代表至少一个 session 接收，不等价于用户已读。

## 10. 为什么补 Web 验证客户端

Web 客户端不是为了展示前端能力，而是为了验证后端完整闭环：

- 注册登录。
- 资料修改。
- 好友搜索、申请、接受。
- 单聊实时消息。
- 离线消息补偿。
- 搜群、加群、群聊。
- 红点提醒。
- WebSocket 推送。

相比只用 curl 或单元测试，Web 客户端能验证真实用户流程：

- 两个账号。
- 两个浏览器窗口。
- 在线和离线状态。
- UI 操作是否符合正常 IM 软件逻辑。
- 后端错误是否能转换为用户能理解的提示。

面试表达：

```text
我补 Web 客户端不是把项目转成前端项目，而是为了用真实 IM 交互验证后端链路，
确保 Gateway、服务层、存储层和 WebSocket 推送不是孤立可测，而是端到端闭环。
```

## 11. 技术选型总结

MyChat 的技术选型可以总结为：

- C++：承载后端核心服务和网络工程能力。
- Gateway：统一客户端接入，集中鉴权和连接管理。
- HTTP + WebSocket：分别解决请求响应和实时推送。
- Protobuf + gRPC：定义内部服务契约和远程调用边界。
- Redis：处理 token/session 等临时高频状态。
- PostgreSQL + ODB：处理持久业务数据和 C++ 对象映射。
- PushRuntime + fanout policy：抽象在线投递策略。
- Web 客户端：支撑真实业务流验证和演示。

一句话版本：

```text
MyChat 的技术栈围绕“分布式 IM 后端闭环”组织：Gateway 负责接入，gRPC 划分服务边界，
Redis 管理临时状态，PostgreSQL/ODB 持久化核心业务，WebSocket 和 Push 负责在线实时投递。
```

## 12. 后续技术演进方向

后续可以继续演进：

- 服务发现：让 Gateway 不再只依赖静态 endpoint。
- 多 Gateway 连接路由：维护 uid -> gateway/session 的全局索引。
- 消息序列号：按会话维护递增 seq，增强顺序语义。
- 对象存储：头像、图片、文件迁移到 MinIO/S3。
- 压测：连接数、消息吞吐、数据库写入、Redis QPS。
- 可观测性：trace id、metrics、日志聚合。
- 部署编排：Docker Compose 到 Kubernetes。
- 生产安全：限流、验证码、登录风控、TLS、密钥管理。

### A. Gateway 接入层

核心文件：

```text
gateway/gateway_server/gateway_server.cpp
gateway/http/user_http_controller.cpp
gateway/http/message_http_controller.cpp
gateway/http/friend_http_controller.cpp
gateway/http/group_http_controller.cpp
gateway/http/group_message_http_controller.cpp
gateway/ws/message_ws_handler.cpp
```

阅读重点：

- `register_*_http_routes_on_server` 展示 Gateway 如何把 HTTP 路由挂到 controller。
- `UserHttpController`、`MessageHttpController` 等 controller 负责 HTTP 参数解析、token 校验、调用 client facade。
- `MessageWsHandler::handle_send` 展示 WebSocket 二进制消息如何校验 cmd、解析 protobuf、落库并返回 ack。

面试落点：

```text
Gateway 不是业务数据库层，而是协议接入、鉴权和服务编排层。
```

### B. local/remote facade

核心文件：

```text
gateway/http/user_client.cpp
gateway/http/remote_user_client.cpp
gateway/http/message_client.cpp
gateway/http/remote_message_client.cpp
gateway/http/friend_client.cpp
gateway/http/remote_friend_client.cpp
gateway/http/group_client.cpp
gateway/http/remote_group_client.cpp
gateway/push/remote_push_notifier.cpp
```

阅读重点：

- `Local*Client` 直接调用进程内 service。
- `Remote*Client` 构造 gRPC request，设置 deadline，调用 stub，再把 proto response 转回 service DTO。
- Controller 不直接感知 local/remote，这是当前分布式演进的关键。

面试落点：

```text
facade 让 Gateway 的 HTTP/WS 外部契约不随服务部署方式变化。
```

### C. gRPC 服务边界

核心文件：

```text
common/proto/user.proto
common/proto/message.proto
common/proto/friend.proto
common/proto/group.proto
common/proto/push.proto
services/user/user_grpc_service.cpp
services/message/message_grpc_service.cpp
services/friend/friend_grpc_service.cpp
services/group/group_grpc_service.cpp
services/push/push_grpc_service.cpp
```

阅读重点：

- `.proto` 是服务间契约。
- `*GrpcService` 负责把 proto request 映射到 service core。
- `BaseResponse.error_code` 表达业务错误，gRPC status 表达传输错误。

面试落点：

```text
服务边界不是简单拆目录，而是通过 proto 和 gRPC 明确定义跨进程契约。
```

### D. 领域服务与持久化

核心文件：

```text
services/user/user_service.cpp
services/message/message_service.cpp
services/friend/friend_service.cpp
services/group/group_service.cpp
services/group/group_message_service.cpp
services/*/*_repository.cpp
services/odb/*.hpp
```

阅读重点：

- Service 层做业务校验和状态流转。
- Repository 层封装 ODB 事务、查询、更新。
- ODB model 定义持久对象结构。

面试落点：

```text
业务规则主要在 service，数据库细节主要在 repository。
```

### E. Redis 与认证

核心文件：

```text
gateway/auth/multi_platform_auth.cpp
common/database/redis/redis_mgr.hpp
common/database/redis/redis_mgr.cpp
```

阅读重点：

- `generate_access_token` 生成 JWT。
- `generate_refresh_token` 写 Redis，设置 TTL 和用户 token 集合。
- `verify_access_token` 校验 JWT 签名、issuer、audience、撤销状态。

面试落点：

```text
Redis 承载 refresh token 和撤销状态，不承载核心业务持久数据。
```

### F. Web 验证客户端

核心文件：

```text
clients/web/src/App.tsx

clients/web/src/api/wsClient.ts
```

阅读重点：

- `httpClient.ts` 对应 Gateway HTTP API。
- `wsClient.ts` 管 WebSocket 连接。
- `protobufEnvelope.ts` 负责 WebSocket 发送帧编码。
- `App.tsx` 把登录、好友、单聊、群聊、Push 红点串成用户流程。

面试落点：

```text
Web 客户端不是项目重心，而是用于端到端验证后端链路。
```
