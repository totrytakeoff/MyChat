# 单机 MVP 手工测试清单

日期：2026-06-22

## 1. 测试目标

这份清单用于确认 MyChat 在单机环境下是否具备完整 MVP 可演示能力。

单机测试不是压测，也不是生产部署验收；它关注的是：

- 后端基础依赖能启动。
- Gateway 能同时提供 HTTP 和 WebSocket。
- Web 客户端能完成正常 IM 操作。
- local / remote-all 模式下核心业务链路不退化。
- 出错时前端提示可理解，后端日志可定位。

建议每次进入大改前后都跑一遍这份清单，避免重构时只盯编译而漏掉业务闭环。

## 2. 环境准备

基础依赖：

```bash
docker compose up -d redis postgres
```

后端建议先跑 remote-all，因为它最接近分布式服务形态：

```bash
scripts/dev/run_remote_services_stack.sh
```

Web 客户端：

```bash
scripts/dev/run_web_stack.sh
```

如果只想快速验证 local 模式，可以直接启动 Gateway，使用 `config/dev.json`。

## 3. 冒烟检查

| 检查项 | 预期结果 |
| --- | --- |
| Redis 容器 | healthy 或可连接 |
| PostgreSQL 容器 | healthy 或可连接 |
| User Service | 端口监听，日志无启动异常 |
| Message Service | 端口监听，日志无启动异常 |
| Friend Service | 端口监听，日志无启动异常 |
| Group Service | 端口监听，日志无启动异常 |
| Push Service | 端口监听，Gateway callback 配置正确 |
| Gateway HTTP | `/api/v1/health` 返回 200 |
| Gateway WebSocket | Web 客户端能建立连接 |
| Web Client | 登录/注册页面可访问 |

优先检查 Gateway，因为它是所有客户端请求的第一入口。

## 4. 账号与资料

测试动作：

1. 注册用户 A。
2. 注册用户 B。
3. 用户 A 登录。
4. 用户 B 登录。
5. 用户 A 修改昵称、签名、头像。
6. 刷新页面后重新登录用户 A。

预期结果：

- 注册成功后能直接登录或返回明确成功信息。
- 重复账号注册应返回可理解错误。
- 登录失败不能停留在主页面，应回到登录页或给出明确提示。
- 资料修改后重新查询仍能看到最新数据。
- 头像字段当前支持 URL 或 base64 data URL，后续对象存储不属于当前 MVP 阻塞项。

## 5. 好友链路

测试动作：

1. A 搜索 B 的账号或昵称关键字。
2. 搜索结果展示 B 的用户卡片。
3. A 发起好友申请。
4. B 在待处理申请中看到 A。
5. B 接受申请。
6. A 和 B 的好友列表都出现对方。

预期结果：

- 不应要求用户输入内部 uid。
- 搜索结果应先展示用户信息，再允许申请。
- 接受后双方关系可见。
- 重复申请或已是好友时返回明确提示。

## 6. 单聊链路

测试动作：

1. A 打开与 B 的聊天。
2. B 打开与 A 的聊天。
3. A 发送文本消息。
4. B 在线收到 WebSocket 推送。
5. B 回复消息。
6. A 在线收到 WebSocket 推送。
7. 刷新页面后查看历史消息。

预期结果：

- 发送方能收到 ack。
- 接收方无需手动刷新即可看到新消息。
- 历史消息顺序正确。
- 消息发送使用 token 中的真实 uid，不能相信客户端传入 sender_uid。

## 7. 离线消息

测试动作：

1. B 退出登录或关闭页面。
2. A 给 B 发送多条消息。
3. B 重新登录。
4. B 查看离线消息或历史消息。

预期结果：

- A 发送消息成功，不依赖 B 在线。
- B 上线后能通过离线拉取或历史查询补偿。
- 在线推送失败不应该导致消息丢失。

## 8. 群聊链路

测试动作：

1. A 创建群。
2. B 搜索群。
3. B 查看群资料。
4. B 加入群。
5. A 和 B 都能看到群成员。
6. A 在群里发消息。
7. B 在线收到群消息。
8. B 退出群。

预期结果：

- 加群应先搜索和查看群资料，再执行加入。
- 群资料页展示群信息和成员。
- 群消息 fanout 给在线成员。
- 退群入口应在已加入群的资料页中。

## 10. 通过标准

这轮单机 MVP 通过标准：

- 两个账号能完整完成注册、登录、资料修改、加好友、单聊、离线消息、建群、加群、群聊。
- WebSocket 在线推送不依赖手动刷新。
- Gateway 和各服务日志没有崩溃、死锁、连接泄漏。
- HTTP `/api/v1/health` 和 `/api/v1/stats` 可用。
- 出错提示能让普通用户理解，不只暴露 HTTP 状态码。
