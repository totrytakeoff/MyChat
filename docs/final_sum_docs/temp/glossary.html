● Gateway 核心抽象

Gateway 把 HTTP 请求和 WebSocket envelope 统一转换成的内部消息模型。是 HTTP/WS 殊途同归的关键 —— 不同协议入口汇入同一套命令分发模型。

代码：MessageParser::parse_http_request_enhanced / parse_websocket_message_enhanced

负责把不同协议入口（HTTP catch-all / WS frame）转成 UnifiedMessage，并解析 method/path/token/device/platform/body/query。

代码：gateway/message_processor/message_parser.cpp

保存 cmd_id → handler 映射，将消息处理投递到全局线程池，同步执行对应 handler。不做统一强制 token 校验（因为注册/登录是公开入口）。

代码：message_processor.cpp process_message / process_message_sync

集中注册业务命令（user/message/friend/group）。handler 从 RuntimeRegistry 取依赖，不再持有散落成员。

代码：gateway_command_handler_registry.cpp，注册入口 register_gateway_command_handlers

Gateway 运行时依赖表，保存各服务的 local service instance 或 remote service adapter，以及 AuthManager/PushNotifier/MessageWsHandler。handler 通过它选择 local packet dispatcher 或 remote ForwardPacket。

代码：GatewayRuntimeRegistry，GatewayCommandHandlerContext { const GatewayRuntimeRegistry* runtime }

命令标识符，如 CMD_REGISTER / CMD_LOGIN / CMD_SEND_MESSAGE。MessageProcessor 按 cmd_id 找到对应 handler。HTTP 路由通过配置映射到 cmd_id，WS envelope 头部带 cmd_id。

● Packet Boundary

Gateway 与业务服务之间的统一边界：Gateway 只认识 packet（envelope + cmd_id + payload），不解析业务 payload；Service 自己解析 payload、执行业务、构造响应。这让 HTTP/WS/local/remote 共享同一套服务入口。

统一 packet 契约。Request 含 IMHeader + type_name + payload（HTTP 为 application/json，WS 为 protobuf type）；Response 含 BaseResponse + payload + http_status + push_event。

类型：UserPacketRequest / MessagePacketRequest / FriendPacketRequest / GroupPacketRequest

各服务的统一 gRPC RPC：rpc ForwardPacket(XxxPacketRequest) returns (XxxPacketResponse)。remote 模式下 Gateway 通过它把 packet 发给远程服务进程，服务端用自己的 packet dispatcher 解析。不让 Gateway 直接调业务 RPC，否则 Gateway 要理解业务 DTO。

各业务服务的 packet 分发器，解析 payload（HTTP JSON 或 WS protobuf）、执行业务校验、调用 Service、构造 response 和 push_event。local 和 remote 共用同一个 dispatcher。

代码：services/*/*_packet_dispatcher.cpp

remote 模式下的服务适配器，职责很窄：接收 PacketRequest → 设置 gRPC deadline → 调用 ForwardPacket → 检查 transport status → 返回 PacketResponse。不暴露业务方法、不解析 payload。

● Push 与鉴权

推送调用抽象，本地/远程实现同一接口。Gateway command handler 消费 push_event 后调用它。

统一 fanout、payload 构造、投递结果处理。查询 session → 应用 fanout policy → 构造 CMD_PUSH_MESSAGE → 发送 → mark_delivered。

选择要投递的 session 子集：AllSessionsFanoutPolicy（全部，默认）、PlatformFilterFanoutPolicy（指定平台）、NewestSessionFanoutPolicy（最新连接）。

远程 Push 回调 Gateway 的 gRPC 服务，含三个回调：ListUserSessions（查连接）、SendSessionPayload（发送 payload）、MarkMessageDelivered（标记投递）。存在原因：WebSocket session 在 Gateway 内存中，远程 push_server 无法直接访问。

User service 在登录/注册成功后返回的提示，Gateway 据此生成 access/refresh token。token 签发属于 Gateway 鉴权边界，不在 User service。

Message service 返回 MessagePushEvent（单聊），Group service 返回 repeated GroupPushEvent（群聊 fanout 成员列表）。Gateway 只消费并转交 PushNotifier，不自己计算群成员 fanout。

● 消息语义

核心消息可靠性取舍：消息先落 PostgreSQL → 持久化成功返回 ack → Gateway handoff push_event → Push best-effort 投递。Push 失败不回滚，靠 history/offline 补偿。"发送成功后刷新消失"比"推送稍慢"更不可接受。

ack：服务端已收到并持久化（发送侧语义）。delivered：至少成功投递到一个在线 session 或离线拉取后补标（接收侧）。read：接收方明确阅读后标记。

Push 是加速路径而非可靠来源：没有在线 session 不报致命错误；session 发送失败不回滚消息；远程 Push RPC 失败不影响已持久化消息。可靠性兜底来自持久化 + history/offline 补偿。

local：Gateway 进程内持有服务对象，调用 XxxPacketDispatcher::handle，适合单机 MVP。remote：通过 RemoteXxxServiceAdapter 调 gRPC ForwardPacket 到独立服务进程。两者共享同一 packet 契约和 dispatcher，外部 HTTP/WS 协议不变。
