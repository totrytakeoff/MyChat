§05 · 业务链路篇

注册登录与鉴权链路

MyChat 采用 JWT access token + Redis refresh token 的双 token 多端方案。密码校验属于 User service 业务规则，token 签发与校验属于 Gateway 鉴权边界 —— 这条边界把"账号业务"和"接入层鉴权"清晰分开。

§05.1

注册链路

注册不要求已有 access token；Gateway 不解析 account/password/nickname，这些字段由 UserPacketDispatcher::handle_register 解析；User service 完成账号重复检查、密码哈希、用户落库，只返回 auth_token_hint，Gateway 据此签发 token。

注册链路 · 可交互
token 签发回流
回流
UserPacketResponse(auth_token_hint.required=true)
  → append_tokens_and_respond
  → MultiPlatformAuthManager::generate_tokens
  → HTTP response (profile + access_token + refresh_token)

User service 不直接签发 token，只返回 hint；Gateway 作为鉴权边界统一签发，保证所有业务入口使用同一套多端 token 策略。

§05.3

密码安全 · PBKDF2

PBKDF2-HMAC-SHA256
  • 每个用户独立 salt
  • 存储格式含算法/迭代次数/salt/hash
  • 校验时相同参数重新计算
  • 常量时间比较，防时序泄露
边界划分

密码校验是账号业务规则 → User service

token 签发和校验是接入层鉴权 → Gateway

§05.4

Access Token 与 Refresh Token

Access Token · JWT

短期、轻量、本地可验

uidaccountdevice_idplatformjtiexpires_at
Refresh Token · Redis

长生命周期、可撤销

user_iddevice_idplatformrevokedlast_usedexpire_time
设计目的access token 保持短期、轻量、本地可验；refresh token 生命周期更长，需要服务端可撤销；多端登录态和撤销状态适合放 Redis（天然 TTL）。

§05.7

面试追问 · 点击展开

因为 token 是接入层能力。User service 只证明账号密码校验成功，并返回 auth_token_hint。Gateway 统一签发 token，可以保证所有业务入口使用同一套多端 token 策略、Redis refresh token 管理和 access token 校验逻辑。

当前仅在登录/注册成功后，为响应体注入 access_token 和 refresh_token。这是鉴权边界例外，不应扩展为 Gateway 解析用户业务字段。长期可以进一步把 token 字段作为 packet response 的边界字段，减少 JSON parse/dump。

两者都走同一个 UserPacketRequest/UserPacketResponse 契约。local 直接调用 UserPacketDispatcher::handle，remote 通过 gRPC ForwardPacket 到远端服务后再调用同一个 dispatcher，因此业务解析和规则复用。
