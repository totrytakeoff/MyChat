§06 · 业务链路篇 · 核心

单聊消息链路

单聊消息是 IM 项目的核心链路。MyChat 以 先持久化、再 ack、再 best-effort push 为核心语义；Gateway 只做鉴权、路由和 packet 转发，消息 payload 解析与持久化在 Message service。

HTTP 单聊发送链路

HTTP 发送链路 · 可交互
关键边界Gateway 不解析 receiver_uid/content；不信任 body 中的 sender；从 token 得到可信 sender uid 写入 packet header；Message service 返回 push_event，Gateway 只负责转交。
§06.3

WebSocket 单聊发送链路

WS 发送链路 · 可交互
注意Gateway 不解析 SendMessageRequest 的 body；WebSocket payload protobuf 解析在 MessagePacketDispatcher 中完成；Gateway 只检查 header 级别的 token、device、receiver 路由信息。
§06.4

Ack 语义

ack 成功表示 ✓

消息已经通过 Message service 校验并持久化

ack 不表示 ✕
  • 对方一定在线
  • 对方一定收到 push
  • 对方已经阅读
为什么这样设计对方可能离线；WebSocket 推送可能失败；消息持久化成功后才能保证历史可恢复；push 是加速路径，不是消息可靠性的唯一来源。

§06.8

面试追问 · 点击展开

因为消息正文属于 Message service 业务 payload。Gateway 只需要检查 token 和 header 路由字段，解析 protobuf body 会把业务规则重新塞回 Gateway，破坏 packet boundary。

因为 push 只代表在线投递尝试，不能作为可靠存储。先落库才能保证对方离线、推送失败或客户端重连后仍可恢复消息。

两者最终都构造成 MessagePacketRequest。HTTP payload 是 application/json，WS payload 是 im.message.SendMessageRequest protobuf，具体解析都在 MessagePacketDispatcher，所以 service 业务逻辑可以复用。

缺 client_msg_id 幂等去重、会话 sequence、设备级投递日志、客户端 ACK 驱动的 delivered/read、跨 Gateway 在线路由和更完整的消息类型体系。
