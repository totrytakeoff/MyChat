好友关系链路

MyChat 好友系统以"申请记录 + 状态流转"为核心模型；Gateway 只负责鉴权、可信 uid 注入和 packet 转发，好友业务字段解析和状态机在 Friend service packet dispatcher / FriendService 中完成。

§07.1

搜索用户到好友申请

正常 IM 体验：搜索账号/昵称 → 展示资料卡 → 确认后添加好友。搜索走 User service，好友关系状态机走 Friend service，职责清晰。

§07.2

好友申请状态机

stateDiagram-v2 [*] --> PENDING: send_request PENDING --> ACCEPTED: target 接受 PENDING --> REJECTED: target 拒绝 ACCEPTED --> [*] REJECTED --> [*] note right of PENDING 不允许: · 非 pending 重复处理 · 非目标用户处理 · 同一用户重复发起 · 添加自己 end note
PENDING

申请已发出，等待目标用户处理

ACCEPTED

目标用户已接受，双方成为好友

REJECTED

目标用户已拒绝，申请结束

§07.4

好友列表与待处理申请

GET /api/v1/friends · 好友列表
http
→ CMD_GET_FRIEND_LIST
→ FriendPacketDispatcher::handle_friend_list
→ FriendService::get_friends(from_uid)
GET /api/v1/friends/pending · 待处理
http
→ CMD_GET_FRIEND_REQUESTS
→ FriendPacketDispatcher::handle_pending_friends
→ FriendService::get_pending_requests(from_uid)

返回字段：friend_id / friend_uid / nickname / status / created_at

§07.5

关键业务规则

不能添加自己

requester_uid == target_uid → SELF_REQUEST

目标用户必须存在

FriendService 通过 UserService 校验：user_service_->get_profile_by_uid(target_uid)，不存在返回 TARGET_NOT_FOUND

重复关系拒绝

FriendRepository 检查：find_relationship(requester, target)，已有关系返回 ALREADY_EXISTS

只有目标用户能处理

friend_record.target_uid == current_uid，否则返回 FORBIDDEN

§07.6

面试追问 · 点击展开

因为正常 IM 不应该让用户直接输入内部 uid。用户先通过账号/昵称搜索看到资料卡，再发起申请，体验更接近真实产品；后端上搜索走 User service，好友关系状态机走 Friend service，职责也更清晰。

target_uid 属于好友业务 payload。Gateway 只需要从 token 中拿到 requester uid 并写入 packet header，目标用户字段由 Friend service 解析和校验，避免 Gateway 混入好友业务逻辑。

缺好友备注、分组、黑名单完整接口、申请附言展示、删除好友、好友变更 push 通知、双向关系更细粒度审计等。这些属于后续扩展，不影响当前 MVP 主链路。
