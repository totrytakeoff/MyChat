实时推送与 Push 服务

Push 服务负责在线投递编排，但不拥有 WebSocket 连接；Gateway 持有连接，Push 通过本地适配或远程 Gateway callback 完成 session 查询和 payload 发送。这是项目里最典型的分布式边界设计。

§09.1

Push 触发点

Gateway 不计算单聊/群聊业务字段，不计算群成员 fanout；service packet dispatcher 返回明确的 push event 元数据，Gateway 只转交。

单聊触发
direct
MessagePacketDispatcher 保存消息
  → 返回 MessagePacketResponse.push_event
  → Gateway message_command_handlers
  → PushNotifier::notify_user
群聊触发
group
GroupPacketDispatcher 保存群消息
  → 计算除发送者外的成员 push_events
  → 返回 GroupPacketResponse.push_events
  → Gateway notify_group_push_events
  → PushNotifier::notify_user(each)

§09.4

为什么必须回调 Gateway

sequenceDiagram participant h as Gateway Handler participant rn as RemotePushNotifier participant ps as push_server participant pr as PushRuntime participant cb as GatewayPushDeliveryService participant ws as Gateway WS Session h->>rn: notify_user(uid, msg) rn->>ps: gRPC NotifyUser ps->>pr: fanout + payload 构造 Note over pr,ws: WebSocket session 在 Gateway 内存中
push_server 无法直接访问 pr->>cb: ListUserSessions(uid) cb-->>pr: sessions[] pr->>cb: SendSessionPayload(session, payload) cb->>ws: session lookup + send ws-->>cb: delivered cb->>cb: MarkMessageDelivered
核心原因WebSocket session 是 Gateway 进程内对象。远程 push_server 不能直接访问 Gateway 内存里的 session map，所以 remote push 必须通过 Gateway 暴露的 delivery callback 回到 Gateway 完成实际投递。这是为了在保持 Push 独立服务的同时，不破坏连接所有权。

§09.6

delivered 语义

当前 delivered 表示 ✓

至少一个 session 成功接受 push payload，或离线拉取后补标

不等于 ✕
  • 所有设备都收到
  • 用户已读
  • 端到端强确认
生产级还需要
客户端 ACK设备级投递日志read receipt重试队列
§09.7

best-effort 取舍

Push 当前是 best-effort：没有在线 session 时不报致命错误；session 发送失败不回滚消息；远程 Push RPC 失败不影响已持久化消息；Push 成功只提升实时性。

可靠性兜底链路
先持久化
消息落 PostgreSQL
再 Push
best-effort 在线投递
Push 失败
不回滚，不影响已持久化消息
offline/history 补偿
用户重连后拉取离线消息
