§12

单机 MVP 冒烟检查

单机测试不是压测，关注的是：后端基础依赖能启动、Gateway 能同时提供 HTTP 和 WebSocket、Web 能完成正常 IM 操作、local/remote-all 模式下核心链路不退化。

环境准备

bash
# 基础依赖
docker compose up -d redis postgres

# 推荐 remote-all（最接近分布式形态）
scripts/dev/run_remote_services_stack.sh

# Web 客户端
scripts/dev/run_web_stack.sh

# 或快速验证 local 模式
gateway_server -c config/dev.json

检查项预期结果

核心链路测试动作

账号与资料

注册 A/B → 登录 → 修改昵称签名头像 → 刷新重登验证持久化

好友链路

A 搜索 B → 申请 → B 待处理可见 → 接受 → 双方列表出现对方

单聊链路

A 发消息 → B 在线收 WS push → B 回复 → A 收 push → 刷新看历史

离线消息

B 退出 → A 发多条 → B 重登 → 离线拉取补偿，A 发送不依赖 B 在线

群聊链路

A 建群 → B 搜群 → 查看资料 → 加入 → 双方见成员 → A 发群消息 → B 在线收 → B 退群

关键校验

消息发送使用 token 中真实 uid，不能相信客户端传入 sender_uid

§15

多机器分布式部署验证

面试和演示阶段需要证明它不是"只在单进程里跑得通"。

最小两机器
  • Machine A：Gateway + Web Client
  • Machine B：User/Message/Friend/Group/Push Services
三机器（更真实）
  • Machine A：Gateway
  • Machine B：5 个服务
  • Machine C：Redis / PostgreSQL

验证顺序

① 基础网络
Gateway 机 nc -vz 各服务端口；Push 机 nc -vz Gateway callback 端口
② 服务启动（顺序重要）
先 docker compose 启动 Redis/PG → 再启动 5 个服务 → 最后启动 Gateway（启动时要能初始化 remote client）
③ 功能验证
health → 注册 A/B → 登录 → 搜索 → 申请 → 接受 → 单聊 push → 离线 → 建群加群 → 群聊 push，每步确认 Gateway 日志走 remote
④ Push callback 验证（重点）
A 给在线 B 发单聊 → Push 日志出现 NotifyUser → Gateway callback 日志出现 session lookup/send → B 前端无需刷新收到
面试表达我没有一开始就强行上 Kubernetes，而是先做了 remote-all 多进程和多机器验证。Gateway 对外协议不变，内部通过配置切到 gRPC。这里最关键的是 Push，因为 WebSocket 连接归 Gateway 持有，所以远程 Push 必须 callback Gateway。多机器验证能证明这个边界不是文档设计，而是真实可运行链路。
§16

故障场景与排查手册

排查原则：① 先确认入口（HTTP/WS）② 再确认身份（token/uid/device）③ 再确认链路（Gateway/service/Redis/PG/Push）④ 最后确认性能（队列/连接/线程池）。

定位：/health 纯内存接口也超时，排除业务 DB/Redis；禁用客户端连接复用后延迟恢复，说明 keep-alive 空闲连接占用同步 worker；handler 统计显示业务耗时低，问题在进入 handler 前的 HTTP 接入层。

修复：显式配置 httplib::ThreadPool(64, 1024)；keep-alive timeout 1s；keep-alive max count 1；增加 /stats 输出。

结果：46230 请求失败率 0%，p95 5.81ms。

排查顺序：DNS/IP 可达 → TCP connect → TLS 握手 → WS upgrade → token 有效 → Gateway accept。需区分发压机出口限制、云厂商单源 IP/NAT、TLS/accept 排队、Gateway 应用层 bug。

200u/1s p95=12.60ms 稳定；200u/500ms p95=21.92ms 稳定；200u/250ms p95=2628ms 极限区排队。已排除建连/协议解析/路由失败。尚未精确定位：业务 handler 耗时、PG 持久化、Push 投递、session 写队列、线程池队列深度。下一步必须补分段指标，不要盲改业务代码。

排查：① 前端是否带 Bearer token ② token 是否过期 ③ Redis 中是否被撤销 ④ device_id/platform 是否一致 ⑤ Gateway 是否错误留在主页面。UX 要求：401 应回登录页或弹可理解提示，不应让用户理解 HTTP 状态码。

排查：① WS 是否连接 ② 当前用户 session 是否注册到 Gateway ③ Push 是否被触发 ④ remote Push 是否 callback Gateway ⑤ Gateway 是否找到目标 session ⑥ 前端是否正确合并 push。历史能查到但在线收不到 → 持久化正常，问题集中在 Push/WS。

排查：① 发送者是否在群内 ② 接收者是否在群内 ③ 群成员列表是否正确 ④ group message 是否落库 ⑤ fanout 是否遍历成员 ⑥ Push 是否跳过发送者或错误过滤平台 ⑦ Gateway 是否存在接收者 session。

§标准

通过标准与性能口径

单机 MVP 通过标准
  • 两账号完整完成注册→登录→资料→加好友→单聊→离线→建群→加群→群聊
  • WS 在线推送不依赖手动刷新
  • 日志无崩溃/死锁/连接泄漏
  • /health 和 /stats 可用
  • 出错提示用户可理解
性能口径

双账号验证对应真实用户体感。稳定区间 WSS RTT p95 远低于 200ms 的 Core Web Vitals INP 良好阈值，体感应流畅。极限 200u/250ms 是吞吐拐点，不代表普通聊天场景。

外部参照Google Core Web Vitals INP 良好阈值 200ms 以内；Mattermost 公开模型 30000 并发用户入口约 21 msg/s 但高广播 fanout；ejabberd/Stream 公开百万级连接能力用于说明工业级上限，不应直接对标当前 MVP。
