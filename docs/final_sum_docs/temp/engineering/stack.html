核心技术栈与选型

MyChat 的技术选型围绕 IM 后端的几个核心问题展开：Gateway 用 C++/Asio/Beast 处理 HTTP 和 WebSocket；内部用 Protobuf/gRPC 定义服务边界；Redis 处理 token 和在线临时状态；PostgreSQL/ODB 处理持久业务数据。

§03.2

C++ 的价值

MyChat 使用 C++ 不是因为聊天软件只能用 C++，而是为了体现网络服务开发能力、长连接和对象生命周期管理、Protobuf/gRPC/Redis/PostgreSQL 在 C++ 工程中的整合、多模块 CMake 工程组织、面向服务边界的后端架构设计。

面试表达我用 C++ 实现 IM 后端，是为了把连接管理、协议编解码、实时推送和服务间通信这些后端基础能力做出来，而不是只写一个 CRUD 服务。
§03.3

Gateway 技术选型

Gateway 使用 HTTP + WebSocket：HTTP 适合注册、登录、资料、好友、群、历史查询；WebSocket 适合实时消息、在线 push、ack。两者统一进入 MessageParser → UnifiedMessage → MessageProcessor → CommandHandlerRegistry。

flowchart LR H[HTTP]:::http --> P[MessageParser]:::gw W[WebSocket]:::ws --> P P --> U[UnifiedMessage]:::gw U --> M[MessageProcessor]:::gw M --> R[CommandHandlerRegistry]:::gw R --> S[Service / ForwardPacket]:::svc classDef http fill:#E3EAEF,stroke:#3D5A6C; classDef ws fill:#E6EDE4,stroke:#6B8E6B; classDef gw fill:#F4E3D8,stroke:#D97757; classDef svc fill:#EDE7DE,stroke:#A39B92;
比旧方案的优势统一鉴权边界、统一 route/cmd 分发、统一 local/remote packet 转发、Gateway 不解析业务 payload。
§03.4

Protobuf 与 gRPC

Protobuf 当前有两类用途：WebSocket envelope 和消息 payload；gRPC 服务 IDL 和 packet contract。

gRPC 主边界 · ForwardPacket
proto
UserService.ForwardPacket(UserPacketRequest)
MessageService.ForwardPacket(MessagePacketRequest)
FriendService.ForwardPacket(FriendPacketRequest)
GroupService.ForwardPacket(GroupPacketRequest)
设计价值 ✓
  • Gateway 只认识 packet
  • Service 自己解析 payload
  • local 和 remote 共用同一个 dispatcher
  • 外部 HTTP/WS 协议不受部署方式影响
待优化

packet contract 和业务 proto 仍混在同一个 .proto 文件中。后续应拆出 *_packet.proto 或统一 gateway_packet.proto。

§03.5

Redis 与 PostgreSQL

PostgreSQL/ODB · 持久业务状态
用户单聊消息好友关系群资料群成员群消息
Redis · 临时高频状态
refresh token 元数据token 撤销状态token 索引session/在线状态
选型原则能恢复、需要事务和长期保存的数据放 PostgreSQL；有 TTL、高频访问、临时状态的数据放 Redis。
§03.6

Push 设计

技术点
  • PushNotifier 抽象本地/远程推送
  • PushRuntime 统一 fanout、payload 构造和投递结果处理
  • FanoutPolicy 支持全部 session、平台过滤、最新 session
  • remote push 通过 GatewayPushDeliveryService callback 回到 Gateway 投递
核心原因

WebSocket session 在 Gateway 内存中，远程 Push 服务不能直接访问。

Web 客户端的定位

React Web 客户端不是核心技术炫技点，而是验证后端：注册登录、好友申请、单聊实时消息、群聊、个人资料和群资料、WebSocket 推送。它证明后端不是只靠 curl 可用，而能被真实 IM 交互驱动。
