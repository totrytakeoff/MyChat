§01 · 架构篇

项目整体架构总览

MyChat 是一个 C++ 分布式 IM 后端，核心入口是 Gateway。它统一接入 HTTP 和 WebSocket，把请求解析成统一消息模型后按 cmd_id 分发；后端按服务边界拆分，支持本地 in-process 与 gRPC ForwardPacket 远程两种运行模式。

§01.1

一句话定位

Gateway 统一接入 HTTP 和 WebSocket，把请求解析成统一消息模型后按 cmd_id 分发；后端按 User、Message、Friend、Group、Push 等服务边界拆分，支持本地 in-process 运行，也支持通过 gRPC ForwardPacket 调用远程服务；Redis 管理 token/session 等临时状态，PostgreSQL/ODB 持久化用户、消息、好友和群组数据。

面试压缩版 我做了一个 C++ 分布式 IM 后端，核心是 Gateway 统一接入 HTTP/WebSocket，内部通过 MessageParser、MessageProcessor 和 command handler registry 做统一命令分发。Gateway 只负责协议入口、鉴权、路由、连接管理和 packet 转发，真正的业务解析和业务规则在各 service 的 packet dispatcher 中完成。项目覆盖注册登录、好友、单聊、群聊、在线推送，并支持 local 和 remote gRPC 两种运行模式。

§01.4

最关键的架构变化

之前文档里大量提到的 HTTP Controller → Local*Client / Remote*Client facade → Service 已经不是当前主架构。

✕ 重构前

HTTP Controller

→ Local*Client / Remote*Client facade

→ Service

HTTP 与 WS 是两套割裂入口；Gateway 出现业务字段解析、业务 DTO 构造和 response mapper。

✓ 重构后

HTTP / WebSocket

→ MessageParser → UnifiedMessage

→ MessageProcessor

→ CommandHandlerRegistry

→ RuntimeRegistry

→ local dispatcher 或 remote ForwardPacket

✓ HTTP/WS 统一 · ✓ Gateway 不解析业务 payload · ✓ 边界更干净

交互演示 · 点击切换旧/新链路

重构前 controller/client 重构后统一 packet

HTTP Controller（每模块单独注册 route）

→ Local*Client / Remote*Client facade

→ Service（Gateway 解析业务 payload、构造 DTO、维护 mapper）

→ WebSocket 走另一套入口

✕ HTTP 与 WS 割裂 · ✕ Gateway 混入业务逻辑 · ✕ 新增服务需成套维护 controller/client/DTO

HTTP / WebSocket （统一入口）

→ MessageParser → UnifiedMessage

→ MessageProcessor（按 cmd_id 分发）

→ CommandHandlerRegistry（鉴权 + 身份注入）

→ RuntimeRegistry（选 local / remote）

→ local dispatcher 或 remote ForwardPacket

✓ HTTP/WS 统一 · ✓ Gateway 不解析业务 payload · ✓ 新增服务只需注册 cmd_id

§01.5

模块职责总览

Web Client

后端能力验证入口，不是后台管理系统。提供中文桌面 IM 风格的登录、聊天、联系人、群聊、资料页，验证 Gateway HTTP API 与 WebSocket 二进制 protobuf 收发。

关键边界Web 只访问 Gateway；不直接访问各业务服务；不关心后面是 local 还是 remote。

Gateway

整个系统的统一外部入口。

职责 ✓
  • 暴露 HTTP API 和 WebSocket 入口
  • 转成 UnifiedMessage，按 cmd_id 调 handler
  • 校验 access token，提取可信 uid/device/platform
  • 管理 WS session、心跳、背压
  • packet 转发到 local/remote
  • 消费 auth_token_hint 与 push_event
不负责 ✕
  • 解析注册/好友/群/消息正文等业务 payload
  • 构造业务 DTO
  • 维护业务 response mapper
  • 实现好友状态机、群成员规则
  • 直接操作业务表

五大业务服务

服务核心能力gRPC 边界关键设计
User注册、登录、资料、搜索ForwardPacket(UserPacketRequest)PBKDF2-HMAC-SHA256；返回 auth_token_hint 交 Gateway 签发 token
Message单聊发送、历史、离线、delivered/readForwardPacket(MessagePacketRequest)先持久化再 ack；返回 push_event；push 失败不回滚
Friend申请、处理、好友列表、待处理ForwardPacket(FriendPacketRequest)防重复、防自加、目标存在校验、状态机闭环
Group建群、搜索、加入、退出、群消息ForwardPacket(GroupPacketRequest)群成员 fanout 在 service 内算；返回 push_events 列表
Push在线投递、fanoutNotifyUser + Gateway callback远程 Push 必须回调 Gateway 投递 WS session
面试表达Gateway 是协议接入层、鉴权边界和命令分发层，不是业务处理层。业务 payload 的解析和业务规则都在 service packet dispatcher 内完成。
§01.6

Local / Remote 运行模式

同一套 handler 通过 GatewayRuntimeRegistry 选择 local service 或 remote adapter，外部 HTTP/WS 协议不变。

Local all-in-one Remote gRPC 分布式
Local 模式
local
Gateway command handler
  → GatewayRuntimeRegistry
  → service instance      # 进程内持有
  → XxxPacketDispatcher::handle(packet)

# 优点：启动简单、本地调试方便、单机 MVP
Remote 模式
remote
Gateway command handler
  → GatewayRuntimeRegistry
  → RemoteXxxServiceAdapter
  → XxxService::ForwardPacket  # gRPC
  → service server XxxPacketDispatcher::handle

§01.7

数据与缓存

PostgreSQL + ODB · 持久态
  • 用户账号和资料
  • 单聊消息
  • 好友申请和好友关系
  • 群资料和群成员
  • 群消息
Redis · 临时态
  • access/refresh token 相关状态
  • Gateway connection/session 映射
  • 在线状态辅助信息
取舍原则持久化状态放 PostgreSQL；临时态、在线态、token 态放 Redis；Gateway 不直接成为业务数据持久化层。
