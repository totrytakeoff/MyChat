§04 · 架构篇 · 核心

Gateway 统一接入与服务编排

理解项目必须先从 Gateway 入手。它的核心定位是 协议入口 + 鉴权边界 + 连接管理 + cmd 路由 + packet 转发，不再是旧的 HTTP controller 集合，也不再通过 Local/RemoteClient facade 作为主链路。

§04.5

MessageProcessor 的职责

职责很窄：保存 cmd_id→handler 映射、投递线程池、同步执行 handler、处理通用错误。它不做统一强制 token 校验。

为什么不在通用层强制验签注册/登录是公开入口，不能强制验签；HTTP 和 WS 鉴权语义不同；handler 需要按命令语义决定是否鉴权，并把 token uid 写入 packet header。所以鉴权位置在 command handler / WS handler 中。

§04.7

Command Handler 的边界

handler 可以做 ✓
  • 检查 service/adapter 是否存在
  • 校验 HTTP/WS token
  • 从 token 提取可信 uid/device/platform
  • 把可信身份写入 packet header
  • 构造 XxxPacketRequest
  • 调 local dispatcher 或 remote adapter
  • 消费 auth_token_hint 生成 token
  • 消费 push_event 转交 PushNotifier
handler 不应该做 ✕
  • 解析注册/好友/群/消息正文等业务 payload
  • 构造业务 DTO
  • 查询业务 repository
  • 实现好友/群成员/消息状态业务规则
  • 自己计算群 fanout 成员
  • 维护业务 response mapper
一句话Gateway handler 是 packet 转发边界，不是业务 service。

Remote Push 时序：为什么必须回调 Gateway

sequenceDiagram participant h as Gateway Handler participant rn as RemotePushNotifier participant ps as push_server participant pr as PushRuntime participant cb as GatewayPushDeliveryService participant ws as Gateway WS Session h->>rn: notify_user(uid, msg) rn->>ps: gRPC NotifyUser ps->>pr: fanout + payload 构造 Note over pr,ws: WebSocket session 在 Gateway 内存中
push_server 无法直接访问 pr->>cb: ListUserSessions(uid) cb-->>pr: sessions[] pr->>cb: SendSessionPayload(session, payload) cb->>ws: session lookup + send ws-->>cb: delivered cb->>cb: MarkMessageDelivered
核心原因WebSocket session 属于 Gateway 进程内存；远程 push_server 不能直接操作 Gateway 的 session map，所以 remote push 只能通过 Gateway 暴露的 delivery callback 回到 Gateway 投递。这是项目里很值得讲的分布式边界设计。

§04.11

面试时怎么讲

主表述 MyChat 的 Gateway 不是传统意义上的一堆 HTTP controller。我把 HTTP 和 WebSocket 都统一成 UnifiedMessage，然后通过 MessageProcessor 按 cmd_id 分发。Gateway handler 只做鉴权、身份注入、packet 转发和 push event 转交，真正的业务 payload 解析在各 service 的 packet dispatcher 中完成。这样 Gateway 的职责很清楚：它是协议入口和服务路由层，不是业务处理层。同时，GatewayRuntimeRegistry 让同一套 handler 可以选择 local service 或 remote ForwardPacket gRPC，所以项目可以从 all-in-one 开发模式平滑切到多进程分布式服务模式。
被追问"为什么要重构掉 controller/client" 因为原来的 controller/client 链路让 HTTP 和 WS 走了两套模型，而且 Gateway 中出现了业务字段解析、业务 DTO 构造和 response mapper。这违背了我最初对 Gateway 的设计：Gateway 只做协议入口、鉴权、路由和转发。重构后 HTTP/WS 都回到 MessageParser + MessageProcessor + command handler registry，业务解析下沉到 service packet dispatcher，Gateway 边界更干净，也更利于讲清分布式服务边界。
仍需注意的问题
  • packet contract 和业务 proto 仍混在同一 proto 文件，长期应拆出 *_packet.proto
  • RuntimeRegistry 仍是显式字段，可进一步收敛为泛化 endpoint map
  • /api/v1/stats 目前是文本输出，后续应改为 JSON
  • CLI/config 优先级需修复，避免压测端口被配置覆盖
