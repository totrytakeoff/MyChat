§11 · 架构篇

数据持久化与缓存设计

MyChat 使用 PostgreSQL/ODB 保存可恢复的业务状态，用 Redis 保存高频、临时、可过期状态；两者按数据生命周期和一致性要求分工，而不是混用。

§11.2

PostgreSQL / ODB 设计

为什么用 PostgreSQL
  • 用户和关系天然是结构化数据
  • 消息历史需要可靠保存
  • 好友和群成员需要关系查询
  • 可扩展索引、事务、分页、统计
为什么用 ODB
  • C++ 对象和数据库表映射
  • repository 围绕 C++ 对象写逻辑
  • 减少重复手写 SQL
  • 生成 schema 和对象访问代码
ODB 价值 ✓

对象模型与表映射清晰，repository 隔离 ODB 访问，事务边界在 repository。

ODB 代价 ✕

工具链复杂、生成物需纳入构建、schema 演进需单独管理、新人理解成本高。

面试表达我用 ODB 是为了在 C++ 后端中保持对象模型和数据库表的映射，但没有把数据库迁移完全交给 ODB 隐式处理，而是保留显式 migration 脚本。

ODB repository 通用模式

pattern
Repository::method
  → odb::transaction t(db_->begin())
  → db_->persist / load / query / update
  → t.commit()
  → catch odb::exception

# 事务边界在 repository；service 层不直接写 SQL

§11.7

Redis 设计

为什么 token 状态适合 Redis

  • refresh token 天然有过期时间
  • 撤销状态只需要保留到 token 过期
  • 访问频繁，不适合每次查 PostgreSQL
  • Redis TTL 可以自动清理过期状态
refresh token 写入
MultiPlatformAuthManager::generate_refresh_token
  → redis.set("refresh_token:" + rt, rtMeta.dump())
  → redis.expire("refresh_token:" + rt, expire_seconds)
  → redis.sadd("user:" + user_id + ":refresh_tokens", rt)
  → redis.expire("user:" + user_id + ":refresh_tokens", expire_seconds)
Redis 连接池 · hiredis
pool size 可配置wait timeout 可配置连接异常重连+一次重试RAII 管理借还
面试表达Redis 不是用来替代数据库，而是用来承载 token/session 这类高频短生命周期状态。连接池避免每次操作都创建 TCP 连接。
§11.8

Migration 与运行时准备

运行时准备序列 · 可交互
策略服务启动不自动迁移，由脚本显式准备运行时环境。优点：迁移职责清晰、服务启动更可控、多服务部署时避免多个服务同时抢迁移。
§11.9

数据一致性取舍

消息

先持久化，再 Push。Push 失败不回滚；离线拉取补偿；delivered/read 是后续状态更新。

群聊

群消息先持久化，再对成员 fanout。群 fanout 不是事务的一部分。

好友

好友申请状态更新是单条关系状态流转，当前不是复杂分布式事务。

Token

access token 是 JWT，refresh token 在 Redis。Redis 和 PostgreSQL 之间不做强事务绑定。

§11.10

面试问答 · 点击展开

PostgreSQL 保存用户、消息、好友、群组等必须长期恢复的业务数据；Redis 保存 token、session 这类高频、短生命周期、可过期状态。

聊天消息是核心业务数据，需要历史查询和可靠恢复，不能只依赖 Redis。Redis 可以做缓存或队列，但最终消息应该落到持久数据库。

优点是 C++ 对象和数据库表映射清晰，减少手写 SQL；缺点是工具链复杂，schema 演进和生成代码管理需要额外工程约束。

多服务场景下如果每个服务启动都自动迁移，可能产生并发迁移和不可控失败。当前把迁移放到脚本显式执行，职责更清晰。

refresh token、撤销状态和部分 session 状态会受影响；已持久化的用户、消息、好友、群组数据仍在 PostgreSQL。生产环境需要 Redis 高可用和降级策略。
