# MyChat 项目整体架构总览

日期：2026-06-15

## 1. 项目一句话定位

MyChat 是一个基于 C++ 的分布式 IM 后端项目，采用 Gateway 统一接入，后端按 User、Message、Friend、Group、Push 等服务拆分，通过 gRPC 保留远程服务边界，使用 Redis 管理 token/session 等临时状态，使用 PostgreSQL/ODB 持久化用户、消息、好友和群组数据，并提供 React Web 客户端用于完整业务流程验证。

面试中可以压缩成：

```text
我做了一个 C++ 分布式 IM 后端，核心是 Gateway + 多服务边界 + gRPC + Redis + PostgreSQL/ODB + WebSocket 实时推送，并配套 Web 客户端验证注册登录、好友、单聊、群聊和在线推送流程。
```

## 2. 项目当前阶段

## 4. 模块职责总览

### 4.1 Web Client

Web 客户端是后端能力验证入口，不是后台管理系统。

当前职责：

- 提供正常 IM 软件视角的登录、聊天、联系人、群聊、资料页面。
- 验证 Gateway HTTP 接口。
- 验证 WebSocket 二进制消息收发。
- 用双账号/多窗口验证好友、单聊、群聊和在线推送。
- 调试信息放入 debug drawer，不干扰主用户流程。

关键点：

- Web 只访问 Gateway。
- Web 不直接调用 User/Message/Friend/Group/Push 服务。
- Web 不关心 Gateway 后面是本地服务还是远程 gRPC 服务。

### 4.2 Gateway

Gateway 是整个系统的统一外部入口。

当前职责：

- 暴露 HTTP API。
- 暴露 WebSocket 入口。
- 验证 access token。
- 从 token 提取可信 uid。
- 管理 WebSocket session。
- 注册业务 HTTP controller。
- 将请求转发到各服务的 local/remote facade。
- 在 WebSocket 单聊发送后触发 Push。

Gateway 不负责：

- 用户资料持久化规则。
- 消息持久化规则。
- 好友关系状态机。
- 群成员领域规则。
- Push fanout 策略本身。

面试表达：

```text
Gateway 是协议接入层和服务编排层，负责客户端协议、鉴权和连接管理；真正的业务规则下沉到服务模块中。
```

### 4.3 User Service

User Service 负责账号和用户资料。

当前能力：

- 注册。
- 登录凭证校验。
- 用户信息查询。
- 用户搜索。
- 昵称、签名、性别、头像更新。
- gRPC 服务边界和独立 `user_server`。

技术点：

- 密码使用 PBKDF2-HMAC-SHA256。
- 用户数据使用 PostgreSQL/ODB 持久化。
- token 由 Gateway 负责发放和校验。
- 资料更新使用 token uid 作为当前用户身份，不信任客户端传 uid。

### 4.4 Message Service

Message Service 负责单聊消息数据和消息状态。

当前能力：

- 单聊发送。
- 会话历史。
- 离线消息拉取。
- delivered/read 标记。
- gRPC 服务边界和独立 `message_server`。

关键设计：

- 消息先持久化，再推送。
- WebSocket ack 表示服务端接收并保存成功。
- 在线推送失败不回滚消息。
- 离线用户后续可通过 offline pull 获取消息。

### 4.5 Friend Service

Friend Service 负责好友关系和好友申请。

当前能力：

- 发送好友申请。
- 接受/拒绝好友申请。
- 好友列表。
- 待处理申请列表。
- gRPC 服务边界和独立 `friend_server`。

关键规则：

- 不能添加自己。
- 目标用户不存在时拒绝。
- 重复申请和反向重复申请拒绝。
- 只有目标用户能处理申请。
- 非 pending 状态不能重复处理。

### 4.6 Group Service

Group Service 负责群资料、成员关系和群消息。

当前能力：

- 创建群。
- 搜索群。
- 查看群资料。
- 加入群。
- 退出群。
- 我的群列表。
- 群成员列表。
- 群消息发送。
- 群消息历史。
- gRPC 服务边界和独立 `group_server`。

关键点：

- 群资料、成员关系、群消息分别持久化。
- 群消息 fanout 默认不向发送者重复推送。
- 群公告、群备注、群通知设置等 UI 形态目前主要是未来扩展方向，不应夸大为完整后端能力。

### 4.7 Push Service

Push Service 负责在线投递和 fanout。

当前能力：

- `PushNotifier` 服务边界。
- `PushRuntime` 统一 fanout、payload 构造、投递结果处理。
- `AllSessionsFanoutPolicy`。
- `PlatformFilterFanoutPolicy`。
- `NewestSessionFanoutPolicy`。
- 远程 `push_server`。
- `GatewayPushDeliveryService` callback。

最关键的架构点：

```text
WebSocket session 由 Gateway 持有。
远程 Push 服务不能直接操作 Gateway 内存中的 session。
因此远程 Push 需要通过 Gateway callback 查询 session、发送 payload、标记 delivered。
```

这是本项目很适合面试讲的一个分布式取舍点。

## 5. Local/Remote 服务边界

MyChat 不是一开始就强制所有服务远程部署，而是采用可演进策略：

```text
Gateway Controller / Handler
-> Gateway Client Facade
-> Local Service 或 Remote gRPC Stub
-> Service Core
```

好处：

- 本地开发快。
- 默认 local 模式更容易调试。
- remote 模式保留真实分布式服务边界。
- Gateway 对外 HTTP/WS 契约不随部署方式变化。
- 同一套业务逻辑能被 local 和 gRPC 复用。

当前支持 local/remote 的服务：

- User
- Message
- Friend
- Group
- Push

典型配置项：

```text
user.mode
message.mode
friend.mode
group.mode
push.mode
```

## 6. 核心业务链路

### 6.1 注册登录链路

```text
Web Client
-> POST /api/v1/auth/register 或 /api/v1/auth/login
-> Gateway UserHttpController
-> UserClient local/remote facade
-> UserService
-> UserRepository
-> PostgreSQL/ODB
-> Gateway AuthManager 生成 token
-> 返回 profile + token
```

面试重点：

- Gateway 为什么负责 token。
- User Service 为什么只负责账号资料。
- token uid 为什么是后续请求身份来源。

### 6.2 单聊消息链路

```text
WebSocket CMD_SEND_MESSAGE
-> Gateway MessageWsHandler
-> 校验 cmd_id / protobuf type / token
-> MessageClient local/remote facade
-> MessageService 持久化消息
-> 返回 SendMessageResponse ack
-> PushNotifier
-> 在线接收者收到 CMD_PUSH_MESSAGE
```

面试重点：

- 消息先落库再推送。
- ack 不等于已读。
- Push 失败不回滚，离线拉取兜底。

### 6.3 好友链路

```text
搜索用户
-> 展示用户资料
-> 发送好友申请
-> 对方查看 pending
-> 接受/拒绝
-> 好友列表更新
```

面试重点：

- 好友申请和好友关系分开建模。
- 服务端处理重复、反向重复、自加好友等边界。

### 6.4 群聊链路

```text
创建/搜索/加入群
-> 查询群资料和成员
-> 发送群消息
-> GroupMessageService 持久化
-> 查询成员
-> Push fanout 给其他在线成员
```

面试重点：

- 群成员关系是权限基础。
- 群消息 fanout 和单聊 push 的差异。
- 大群 fanout 后续如何优化。

### 6.5 远程 Push 链路

```text
Gateway
-> RemotePushNotifier
-> push_server
-> PushRuntime
-> GatewayPushDeliveryService callback
-> Gateway 查询 session
-> Gateway 发送 WebSocket payload
```

面试重点：

- 为什么远程 Push 要回调 Gateway。
- 多 Gateway 实例下需要连接位置路由。
- fanout policy 的扩展空间。

## 7. 数据与缓存设计

当前职责划分：

```text
Redis:
  token/session/连接相关的临时状态

PostgreSQL:
  用户、消息、好友、群组等持久业务数据

ODB:
  C++ 对象与 PostgreSQL 表映射
```

设计理由：

- token/session 需要 TTL、快速读写，适合 Redis。
- 消息、好友、群组是长期业务数据，适合 PostgreSQL。
- ODB 降低 C++ 业务对象和关系表之间的重复转换成本。
- schema migration 显式执行，避免服务启动时隐式修改数据库。

## 8. 当前 MVP 完成度

当前可以认为完整 MVP 已经成立：

- 用户可以注册登录。
- 用户可以修改个人资料。
- 用户可以搜索并添加好友。
- 用户可以进行单聊。
- 用户可以创建、搜索、加入群并群聊。
- 在线消息可以通过 WebSocket 推送。
- 离线单聊消息可拉取。
- 后端服务可以 local 模式运行，也具备 remote gRPC 边界。
- Web 客户端可以实践验证主要业务流。

但它还不是生产级 IM：

- 没有富媒体消息。
- 没有消息撤回、编辑、删除。
- 没有完整群管理权限体系。
- 没有服务发现、生产部署编排和压测报告。
- 没有完整对象存储和文件上传链路。
- 没有多 Gateway 连接路由的生产级方案。

这些应作为后续扩展规划，而不是当前 MVP 的阻塞项。

## 9. 项目亮点

适合简历和面试强调的亮点：

- C++ 后端实现 IM 核心链路，不只是 CRUD 项目。
- Gateway 统一 HTTP/WebSocket 接入，职责清晰。
- local/remote facade 让项目能从本地开发平滑演进到 gRPC 分布式服务。
- User、Message、Friend、Group、Push 都有明确服务边界。
- 消息链路采用先持久化再 best-effort 推送，具备离线补偿思路。
- PushRuntime 抽象 fanout policy，支持多端推送策略扩展。
- 远程 Push 通过 Gateway callback 解决 WebSocket session 归属问题。
- Redis 与 PostgreSQL 职责分离明确。
- Web 验证客户端让后端能力可被真实业务流程验证。

## 10. 面试开场话术

可以这样介绍：

```text
MyChat 是我做的一个 C++ 分布式 IM 后端项目。它的核心架构是 Gateway 统一接入 HTTP 和 WebSocket，后端按 User、Message、Friend、Group、Push 拆成服务边界。Gateway 对外保持稳定协议，对内通过 local/remote facade 调用本地服务或远程 gRPC 服务。

目前项目已经实现注册登录、资料更新、好友申请、单聊、群聊、离线消息和在线推送。消息链路上，我采用先持久化再推送的方式，Push 是 best-effort，失败后可以通过离线拉取补偿。比较有特点的是 Push 服务拆成了 PushNotifier/PushRuntime/fanout policy，并且远程 Push 通过 Gateway callback 操作 Gateway 持有的 WebSocket session。

存储上，Redis 负责 token/session 这类临时状态，PostgreSQL + ODB 负责用户、消息、好友、群组等持久数据。项目还配了一个 React Web 客户端，用来真实验证双账号加好友、单聊、群聊和实时推送。
```
