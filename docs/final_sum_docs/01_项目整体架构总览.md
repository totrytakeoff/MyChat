# MyChat 项目整体架构总览

日期：2026-06-23

## 1. 项目一句话定位

MyChat 是一个 C++ 分布式 IM 后端项目，核心入口是 Gateway。Gateway 统一接入 HTTP 和 WebSocket，把请求解析成统一消息模型后按 `cmd_id` 分发；后端按 User、Message、Friend、Group、Push 等服务边界拆分，支持本地 in-process 运行，也支持通过 gRPC `ForwardPacket` 调用远程服务；Redis 管理 token/session 等临时状态，PostgreSQL/ODB 持久化用户、消息、好友和群组数据，React Web 客户端用于完整业务验证。

面试中可以压缩成：

```text
我做了一个 C++ 分布式 IM 后端，核心是 Gateway 统一接入 HTTP/WebSocket，
内部通过 MessageParser、MessageProcessor 和 command handler registry 做统一命令分发。
Gateway 只负责协议入口、鉴权、路由、连接管理和 packet 转发，
真正的业务解析和业务规则在各 service 的 packet dispatcher 中完成。
项目覆盖注册登录、好友、单聊、群聊、在线推送，并支持 local 和 remote gRPC 两种运行模式。
```

## 4. 最关键的架构变化

之前文档里大量提到的：

```text
HTTP Controller
-> Local*Client / Remote*Client facade
-> Service
```

已经不是当前主架构。

当前 Gateway 统一链路是：

```text
HTTP / WebSocket
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> GatewayCommandHandlerRegistry
-> GatewayRuntimeRegistry
-> local service packet dispatcher 或 remote ForwardPacket gRPC
```

这次重构的核心收益：

- HTTP 和 WebSocket 不再是两套割裂入口。
- Gateway 不再按业务模块维护 controller 主入口。
- Gateway 不再解析 user/message/friend/group 的业务 payload 字段。
- local/remote 不再通过一堆薄 `Client` 包装层暴露给 handler，而是由 `GatewayRuntimeRegistry` 统一管理 service 或 remote adapter。
- 各业务 service 拿到统一 packet 后，在自己的 packet dispatcher 中解析 payload、执行业务校验、构造响应。

## 5. 模块职责总览

### 5.1 Web Client

Web 客户端是后端能力验证入口，不是后台管理系统。

当前职责：

- 提供中文桌面 IM 风格的登录、聊天、联系人、群聊、资料页。
- 验证 Gateway HTTP API。
- 验证 WebSocket 二进制 protobuf envelope 收发。
- 用双账号/多窗口验证好友、单聊、群聊和在线推送。
- 调试信息隐藏在验证辅助区域，不作为普通用户主流程。

关键边界：

- Web 只访问 Gateway。
- Web 不直接访问 User/Message/Friend/Group/Push 服务。
- Web 不关心 Gateway 后面是 local service 还是 remote gRPC service。

### 5.2 Gateway

Gateway 是整个系统的统一外部入口。

当前职责：

- 暴露 HTTP API 和 WebSocket 入口。
- 将 HTTP route 或 WS envelope 转成 `UnifiedMessage`。
- 按 `cmd_id` 调用注册的 command handler。
- 校验 access token，从 token 中提取可信 uid/device/platform。
- 管理 WebSocket session、连接绑定、断连清理、心跳和背压。
- 将 packet 转给本地 service packet dispatcher 或远程 `ForwardPacket` gRPC。
- 消费 service 返回的边界元数据，例如 `auth_token_hint` 和 `push_event`。
- 把 push event 转交给 PushNotifier 或 Gateway 本地推送路径。

Gateway 不负责：

- 解析用户注册字段、好友申请字段、群资料字段、消息正文等业务 payload。
- 构造业务 DTO。
- 维护业务 response mapper。
- 实现好友状态机、群成员规则、消息持久化规则。
- 直接操作业务表。

面试表达：

```text
Gateway 是协议接入层、鉴权边界和命令分发层，不是业务处理层。
业务 payload 的解析和业务规则都在 service packet dispatcher 内完成。
```

核心代码入口：

- `gateway/main.cpp`
- `gateway/gateway_server/gateway_server.cpp`
- `gateway/message_processor/message_parser.cpp`
- `gateway/message_processor/message_processor.cpp`
- `gateway/command_handlers/gateway_command_handler_registry.cpp`
- `gateway/command_handlers/*_command_handlers.cpp`
- `gateway/ws/message_ws_handler.cpp`
- `gateway/service_adapters/remote_*_service_adapter.cpp`

### 5.3 User Service

User Service 负责账号和用户资料。

当前能力：

- 注册。
- 登录凭证校验。
- 用户信息查询。
- 用户搜索。
- 昵称、签名、性别、头像更新。
- `ForwardPacket(UserPacketRequest) returns (UserPacketResponse)` gRPC 边界。
- 独立 `user_server`。

关键点：

- 密码使用 PBKDF2-HMAC-SHA256。
- 用户数据使用 PostgreSQL/ODB 持久化。
- Gateway 负责 token 签发和校验。
- User service 在登录/注册成功后返回 `auth_token_hint`，Gateway 基于 hint 生成 access/refresh token。

### 5.4 Message Service

Message Service 负责单聊消息数据和消息状态。

当前能力：

- 单聊发送。
- 会话历史。
- 离线消息拉取。
- delivered/read 标记。
- HTTP JSON payload 和 WS protobuf payload 的 packet dispatch。
- `ForwardPacket(MessagePacketRequest) returns (MessagePacketResponse)` gRPC 边界。
- 独立 `message_server`。

关键设计：

- 消息先持久化，再返回 ack。
- 在线推送通过 `push_event` 交给 Gateway/Push 链路。
- 在线推送失败不回滚消息。
- 离线用户后续通过 pull offline 获取消息。

### 5.5 Friend Service

Friend Service 负责好友关系和好友申请。

当前能力：

- 发送好友申请。
- 接受/拒绝好友申请。
- 好友列表。
- 待处理申请列表。
- `ForwardPacket(FriendPacketRequest) returns (FriendPacketResponse)` gRPC 边界。
- 独立 `friend_server`。

关键规则：

- 不能添加自己。
- 目标用户不存在时拒绝。
- 重复申请和反向重复申请拒绝。
- 只有目标用户能处理申请。
- 非 pending 状态不能重复处理。

### 5.6 Group Service

Group Service 负责群资料、群成员关系和群消息。

当前能力：

- 创建群。
- 搜索群。
- 查看群资料。
- 加入群。
- 退出群。
- 我的群列表。
- 群成员列表。
- 群消息发送。
- 群消息历史。
- `ForwardPacket(GroupPacketRequest) returns (GroupPacketResponse)` gRPC 边界。
- 独立 `group_server`。

关键点：

- 群资料、成员关系、群消息分别持久化。
- 群存在校验、成员校验、群资料组合、群消息 fanout 成员计算在 group service packet dispatcher 中完成。
- Gateway 只消费 group service 返回的 push events，不自己计算群成员 fanout。

### 5.7 Push Service

Push Service 负责在线投递和 fanout。

当前能力：

- `PushNotifier` 抽象。
- `PushRuntime` 统一 fanout、payload 构造、投递结果处理。
- `AllSessionsFanoutPolicy`。
- `PlatformFilterFanoutPolicy`。
- `NewestSessionFanoutPolicy`。
- 远程 `push_server`。
- `GatewayPushDeliveryService` callback。

最关键的分布式边界：

```text
WebSocket session 由 Gateway 持有。
远程 Push 服务不能直接操作 Gateway 进程内 session。
因此 remote push 必须通过 Gateway callback 回到 Gateway 完成 session lookup 和 payload send。
```

## 6. Local / Remote 运行模式

当前项目支持两种服务调用形态：

```text
local 模式：
Gateway command handler
-> GatewayRuntimeRegistry
-> service instance
-> XxxPacketDispatcher::handle(packet)

remote 模式：
Gateway command handler
-> GatewayRuntimeRegistry
-> RemoteXxxServiceAdapter
-> XxxService::ForwardPacket gRPC
-> service server packet dispatcher
```

这里不要再把它叫成旧的 `Local*Client / Remote*Client facade`。当前更准确的说法是：

- local service：Gateway 进程内持有服务对象。
- remote service adapter：Gateway 通过 gRPC `ForwardPacket` 调用远端服务。
- runtime registry：统一保存当前可用的是 local service 还是 remote adapter。

当前支持 local/remote 的服务：

- User
- Message
- Friend
- Group
- Push

常见配置：

- `config/dev.json`：本地 all-in-one 验证。
- `config/dev.remote-push.json`：Push 远程化验证。
- `config/dev.remote-all.json`：User/Message/Friend/Group/Push 全远程验证。
- `config/benchmark.json`：压测配置。

## 7. 数据与缓存

PostgreSQL + ODB：

- 用户账号和资料。
- 单聊消息。
- 好友申请和好友关系。
- 群资料和群成员。
- 群消息。

Redis：

- access/refresh token 相关状态。
- Gateway connection/session 映射。
- 在线状态辅助信息。

设计取舍：

- 持久化状态放 PostgreSQL。
- 临时态、在线态、token 态放 Redis。
- Gateway 不直接成为业务数据持久化层。

## 10. 项目亮点

适合在简历和面试中重点讲：

- C++ 实现 Gateway 统一接入层，兼容 HTTP 和 WebSocket。
- `MessageParser + MessageProcessor + GatewayCommandHandlerRegistry` 统一消息模型。
- Gateway 和业务 service 的 packet boundary 清晰，Gateway 不解析业务 payload。
- local/remote 运行模式可切换，支持从 all-in-one 演进到 gRPC 分布式服务。
- Push remote callback 处理 Gateway session 归属问题。
- PostgreSQL/ODB + Redis 的持久态/临时态拆分。
- Web 客户端不是摆设，而是用于真实业务链路验证。
- 通过压测定位并修复 WSS 线程模型、连接生命周期、HTTP timeout、Gateway 架构漂移问题。
