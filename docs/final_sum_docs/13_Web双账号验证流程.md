# Web 双账号验证流程

日期：2026-06-22

## 1. 目标

Web 双账号验证用于从真实用户视角验证 MyChat，而不是只看接口。

建议准备两个浏览器窗口：

- 窗口 A：用户 A。
- 窗口 B：用户 B。

这个流程重点验证：

- 前端是否符合正常聊天软件使用习惯。
- 后端 HTTP / WebSocket 链路是否闭环。
- token、好友、单聊、群聊、推送是否能被用户操作真实触发。

## 2. 启动

推荐使用一键脚本启动：

```bash
scripts/dev/run_web_stack.sh
```

如果后端未启动，先启动 remote-all：

```bash
scripts/dev/run_remote_services_stack.sh
```

浏览器访问：

```text
http://127.0.0.1:5173/
```

## 3. 登录注册验证

窗口 A：

1. 注册账号 `alice_test`。
2. 登录。
3. 打开左上角个人头像。
4. 修改昵称、签名、头像。

窗口 B：

1. 注册账号 `bob_test`。
2. 登录。
3. 修改昵称和头像。

预期：

- 登录页不应弹出“登录状态已过期”后仍停留主页面。
- token 过期或 401 时应回到登录页。
- 用户不需要输入 device_id、platform、endpoint 这类调试字段。

## 4. 好友验证

窗口 A：

1. 切换到联系人 tab。
2. 搜索 `bob` 或 B 的账号。
3. 查看搜索结果用户卡片。
4. 发起好友申请。

窗口 B：

1. 联系人 tab 中查看待处理申请。
2. 查看 A 的资料。
3. 接受好友申请。

预期：

- 中间列表在联系人 tab 展示好友和申请，不应继续展示消息列表。
- 右侧主区域展示选中用户资料，不重复展示联系人列表。
- 用户界面展示账号、昵称、头像，不直接暴露内部 uid。

## 5. 单聊验证

窗口 A：

1. 从好友列表打开 B。
2. 发送消息。

窗口 B：

1. 不刷新页面。
2. 应收到消息和红点提醒。
3. 回复 A。

预期：

- 发送支持 Enter 或 Ctrl+Enter，具体行为由设置页决定。
- 收消息依赖 WebSocket push，不需要手动刷新。
- 会话列表红点和未读计数正确变化。
- 聊天顶部状态栏只显示当前选中的聊天对象，不应在切换到联系人/群聊 tab 后仍显示旧聊天对象状态。

## 6. 离线验证

窗口 B：

1. 退出登录或关闭页面。

窗口 A：

1. 连续发送 2-3 条消息。

窗口 B：

1. 重新登录。
2. 打开与 A 的聊天。

预期：

- B 能看到离线期间消息。
- 如果在线 push 不存在，历史/离线拉取要补偿。
- A 的发送状态不应因为 B 离线而失败。

## 7. 群聊验证

窗口 A：

1. 创建群。
2. 打开群资料。
3. 确认群名、简介、成员列表。

窗口 B：

1. 切换到群聊 tab。
2. 搜索群名。
3. 先查看群资料。
4. 点击加入群。

双方：

1. 打开群聊。
2. A 发送群消息。
3. B 在线收到群消息。
4. B 打开群资料并退出群。

预期：

- 群聊 tab 中间栏展示群列表，不应继续显示消息列表。
- 未加入群时，搜索结果页展示群资料和加入按钮。
- 已加入群时，群资料页展示成员和退出入口。
- 群消息在线 fanout 正常。

## 8. WebSocket 推送观察点

前端可观察：

- 不刷新也能收到单聊消息。
- 不刷新也能收到群聊消息。
- 红点和未读计数更新。

后端可观察：

```text
GET /api/v1/stats
```

关键字段：

- `ws.current_sessions`
- `processed websocket message count`
- `parse.decode failed count`
- `parse.routing failed count`
- `ws.inflight_messages`

如果前端必须刷新才看到消息，优先排查：

1. WebSocket 是否连接成功。
2. token 是否过期。
3. Push 是否调用 Gateway callback。
4. Gateway 是否持有目标用户 session。
5. 前端是否正确订阅和合并 push 消息。

## 9. 当前性能口径

双账号验证不是压测，但它对应真实用户体感。

当前压测基线显示：

- 200 用户 / 1s 一条消息，RTT p95 约 12.60ms。
- 200 用户 / 500ms 一条消息，RTT p95 约 21.92ms。
- 这些值远低于 200ms 的常见交互响应阈值，说明当前稳定区间内用户体感应是流畅的。
- 200 用户 / 250ms 一条消息时 RTT p95 上升到 2628.05ms，说明这是极限压力区，不代表普通聊天场景。
