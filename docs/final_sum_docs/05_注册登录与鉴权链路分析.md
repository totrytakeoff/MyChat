# MyChat 注册登录与鉴权链路分析

日期：2026-06-16

## 1. 文档目标

这份文档从业务链路角度分析 MyChat 的注册、登录、token 鉴权和资料访问逻辑。

目标不是罗列接口，而是讲清楚：

- 用户从 Web 客户端发起注册/登录后，请求经过哪些模块。
- Gateway、User Service、Redis、PostgreSQL 分别承担什么职责。
- access token 和 refresh token 当前如何协作。
- 为什么后续业务接口不能信任客户端传入的 uid。
- 面试中如何解释这个链路的设计取舍。

## 2. 链路总览

认证链路可以分成四段：

```text
账号数据链路：
Web Client -> Gateway HTTP -> UserClient facade -> User Service -> UserRepository -> PostgreSQL/ODB

token 生成链路：
Gateway HTTP -> MultiPlatformAuthManager -> JWT access token + Redis refresh token

token 验证链路：
Gateway Controller/Handler -> Authorization Bearer token -> verify_access_token -> UserTokenInfo

资料访问链路：
token uid -> UserClient -> User Service -> PostgreSQL/ODB -> profile
```

最重要的职责划分：

| 模块 | 职责 |
| --- | --- |
| Web Client | 收集账号密码、保存 token、请求 Gateway |
| Gateway | HTTP 接入、参数校验、token 生成与验证、错误响应 |
| MultiPlatformAuthManager | JWT access token、Redis refresh token、多平台 token 配置 |
| UserClient facade | 屏蔽 local/remote User Service 差异 |
| User Service | 注册、登录校验、资料查询和更新 |
| UserRepository | PostgreSQL/ODB 数据访问 |
| Redis | refresh token 元数据、撤销状态、token 索引 |
| PostgreSQL | 用户账号、密码哈希、用户资料 |

## 3. 注册链路

### 3.1 用户视角

用户在 Web 客户端输入：

- 账号。
- 密码。
- 昵称。

Web 客户端调用：

```text
POST /api/v1/auth/register
```

### 3.2 服务端链路

完整链路：

```text
Web Client
-> Gateway UserHttpController::handle_register
-> JSON 解析和必填字段校验
-> UserClient::register_user
-> LocalUserClient 或 RemoteUserClient
-> UserService::register_user
-> UserRepository::account_exists
-> PasswordHasher::hash
-> UserRepository::create
-> PostgreSQL/ODB
-> Gateway MultiPlatformAuthManager::generate_tokens
-> 返回 profile + access_token + refresh_token
```

### 3.3 User Service 内部逻辑

`UserService::register_user` 主要做：

- 检查 account 不为空。
- 检查 password 不为空。
- 检查账号是否重复。
- 生成随机 uid。
- 对密码做 PBKDF2-HMAC-SHA256 哈希。
- nickname 为空时默认使用 account。
- 创建 User 持久化对象。
- 写入 PostgreSQL。
- 返回 UserProfile。

### 3.4 注册失败场景

当前典型失败：

| 场景 | 服务层错误 | HTTP 表现 |
| --- | --- | --- |
| JSON 非法 | Invalid JSON body | 400 |
| 缺少 account/password | Missing required fields | 400 |
| 账号重复 | DUPLICATE_ACCOUNT | 409 |
| 持久化失败 | PERSIST_FAILED | 500 |
| 远程 User 不可用 | REMOTE_USER_UNAVAILABLE | 500 或映射错误 |

### 3.5 面试可讲点

可以强调：

```text
注册成功不是只写用户表，还会立即生成登录态，让 Web 客户端能直接进入主界面。
账号和密码校验由 User Service 完成，token 发放由 Gateway 完成，这样账号数据和接入态管理职责是分开的。
```

## 4. 登录链路

### 4.1 用户视角

用户输入：

- 账号。
- 密码。

Web 客户端调用：

```text
POST /api/v1/auth/login
```

### 4.2 服务端链路

完整链路：

```text
Web Client
-> Gateway UserHttpController::handle_login
-> JSON 解析和必填字段校验
-> UserClient::login_by_account
-> LocalUserClient 或 RemoteUserClient
-> UserService::login_by_account
-> UserRepository::find_by_account
-> PasswordHasher::verify
-> UserRepository::update_last_login
-> Gateway MultiPlatformAuthManager::generate_tokens
-> 返回 profile + access_token + refresh_token
```

### 4.3 密码校验

当前密码使用：

```text
PBKDF2-HMAC-SHA256
```

存储格式包含：

```text
算法名 + 迭代次数 + salt + hash
```

校验时：

- 解析存储的算法、迭代次数、salt、hash。
- 用用户输入密码和同样参数重新计算 hash。
- 使用 `CRYPTO_memcmp` 做常量时间比较。

这比明文密码或简单 hash 更合理：

- 每个用户有独立 salt。
- PBKDF2 有迭代成本。
- 比较过程避免简单时序泄露。

### 4.4 登录失败场景

| 场景 | 服务层错误 | HTTP 表现 |
| --- | --- | --- |
| JSON 非法 | Invalid JSON body | 400 |
| 缺少 account/password | Missing required fields | 400 |
| 账号不存在 | INVALID_ACCOUNT | 401 |
| 密码错误 | WRONG_PASSWORD | 401 |
| 数据库异常 | 其他错误 | 500 |

面向用户时，不应区分“账号不存在”和“密码错误”，避免泄露账号存在性。

当前 Gateway 已把这两类错误统一为：

```text
Invalid account or password
```

## 5. token 生成链路

### 5.1 access token

access token 使用 JWT。

当前 payload 关键字段：

- subject：用户 uid。
- username：账号。
- device_id：设备标识。
- platform：平台，如 web。
- type：access。
- issuer：mychat-gateway。
- audience：mychat-client。
- expires_at：过期时间。
- jti：token 唯一 ID。

作用：

- Gateway 后续可以本地验证 token。
- 业务接口可以从 token 中拿到可信 uid。
- 不需要每次请求都查 PostgreSQL。

### 5.2 refresh token

refresh token 是随机字符串，元数据写入 Redis。

Redis 中保存的信息包含：

- user_id。
- username。
- device_id。
- platform。
- expire_time。
- create_time。
- jti。
- type。
- revoked。
- last_used。

同时设置 TTL，并加入用户 token 集合。

作用：

- 支持 access token 过期后的刷新。
- 支持 token 撤销和轮换。
- 支持按用户、设备、平台管理登录态。

### 5.3 为什么 access token 和 refresh token 分开

原因：

- access token 用于高频接口鉴权，要求验证轻量。
- refresh token 生命周期更长，需要可撤销、可轮换。
- Redis 更适合保存 refresh token 这类可过期状态。
- JWT access token 不适合把所有状态都存进去。

面试表达：

```text
access token 用 JWT 承担短期访问凭证，Gateway 可以本地验证；
refresh token 存 Redis，承担长期登录态和撤销、过期控制。
这样读接口不需要频繁查库，同时保留了服务端控制登录态的能力。
```

## 6. 业务接口鉴权链路

### 6.1 典型接口

需要鉴权的接口例如：

```text
GET  /api/v1/auth/info
POST /api/v1/auth/profile
POST /api/v1/messages/send
GET  /api/v1/messages/history
POST /api/v1/friends/request
POST /api/v1/groups/join
```

### 6.2 鉴权流程

```text
HTTP Request
-> Authorization: Bearer <access_token>
-> Gateway extract_bearer_token
-> MultiPlatformAuthManager::verify_access_token
-> 校验签名、issuer、audience、过期时间
-> 解析 UserTokenInfo
-> 从 user_info.user_id 得到当前 uid
-> 执行业务逻辑
```

### 6.3 为什么不信任客户端 uid

这是认证链路中最重要的安全边界。

客户端提交的 JSON 里即使包含：

```json
{
  "sender_uid": "xxx",
  "uid": "yyy"
}
```

服务端也不应该信任。

正确做法是：

```text
当前用户身份 = access token 中解析出的 uid
```

原因：

- 客户端参数可以被伪造。
- token 是 Gateway 签发并校验过的。
- 资料更新、发消息、处理好友申请、加群都应该基于 token uid。

当前项目中，资料更新链路已经遵循：

```text
POST /api/v1/auth/profile
-> token 解析 user_info.user_id
-> update.uid = user_info.user_id
```

## 7. 资料查询和更新链路

### 7.1 查询当前用户资料

接口：

```text
GET /api/v1/auth/info
```

链路：

```text
Gateway 提取 token
-> verify_access_token
-> user_info.user_id
-> UserClient::get_profile_by_uid
-> User Service / gRPC
-> UserRepository::find_by_uid
-> 返回 profile
```

### 7.2 更新个人资料

接口：

```text
POST /api/v1/auth/profile
```

当前可更新：

- nickname。
- avatar。
- gender。
- signature。

链路：

```text
Gateway 提取 token
-> verify_access_token
-> 查询当前 profile
-> 组合 UpdateProfileRequest
-> uid 使用 token uid
-> UserService::update_profile
-> UserRepository::update_profile
-> 返回更新后 profile
```

### 7.3 头像当前策略

当前头像字段是文本：

- 可以存 URL。
- Web 客户端也支持 base64 data URL。

这是 MVP 取舍：

- 能支持头像展示和资料编辑验证。
- 不需要提前引入文件服务和对象存储。

生产演进方向：

- 上传头像到 MinIO/S3。
- 数据库只保存对象 key 或 CDN URL。
- 支持图片压缩、鉴权访问和过期 URL。

## 8. local/remote User Service 差异

### 8.1 local 模式

```text
Gateway UserHttpController
-> LocalUserClient
-> UserService in-process
-> UserRepository
```

优点：

- 本地调试简单。
- 单元测试容易。
- 少一层网络依赖。

### 8.2 remote 模式

```text
Gateway UserHttpController
-> RemoteUserClient
-> im.user.UserService gRPC stub
-> user_server
-> UserGrpcService
-> UserService
-> UserRepository
```

优点：

- 服务可独立部署。
- Gateway 不直接链接 User 进程内对象。
- 更接近真实分布式架构。

### 8.3 facade 的价值

Gateway 面向的是 `UserClient` 抽象，而不是直接面向 gRPC stub。

这样：

- HTTP controller 不需要关心 local/remote。
- remote client 可以做错误码映射。
- local 和 remote 可以共用服务语义。
- 后续切换部署方式不影响 Web 客户端。

## 9. 当前链路的优势

### 9.1 职责清晰

- Gateway 管接入和 token。
- User Service 管账号与资料。
- Redis 管 refresh token 和撤销状态。
- PostgreSQL 管持久用户数据。

### 9.2 安全边界明确

- 密码不明文存储。
- access token 有签名和过期。
- refresh token 可服务端控制。
- 当前身份以 token uid 为准。

### 9.3 支持分布式演进

- User Service 有 gRPC 边界。
- Gateway 可用 remote client。
- Web 客户端不感知后端部署方式。

## 10. 当前不足和后续优化

当前不足：

- 登录失败限流还未做。
- 验证码和二次验证还未做。
- refresh token 刷新接口和前端自动刷新链路还可以继续完善。
- secret key 管理还偏开发环境。
- token 撤销策略还没有完整 UI。
- 设备管理、踢下线、多端登录策略还需要继续完善。

后续优化：

- 登录失败次数限制和 Redis 滑动窗口。
- 验证码或邮箱/短信验证。
- refresh token rotation 完整化。
- token 黑名单和设备维度撤销。
- HTTPS/TLS 和密钥配置外置。
- 用户安全日志。
- 多端设备管理页面。

## 11. 面试问答准备

### 11.1 为什么 token 放 Gateway 而不是 User Service

回答思路：

```text
Gateway 是统一接入层，所有客户端请求都先到 Gateway，因此 token 校验和发放放在 Gateway 能集中处理。
User Service 专注账号和资料数据，不需要关心 HTTP/WebSocket 接入细节。
这样服务职责更清晰，也方便后续 Friend、Message、Group 等接口统一复用 Gateway 的鉴权结果。
```

### 11.2 为什么业务接口不能信任客户端传 uid

回答思路：

```text
客户端传参可以被伪造，真正可信的身份应该来自服务端签发并校验过的 token。
所以发消息、改资料、处理好友申请时，当前操作者都应该从 token 中解析 uid。
```

### 11.3 密码为什么用 PBKDF2

回答思路：

```text
密码不能明文保存，也不能只做简单 hash。PBKDF2 使用 salt 和迭代计算，提高离线破解成本。
当前实现还用常量时间比较避免简单时序泄露。
```

### 11.4 access token 和 refresh token 为什么分开

回答思路：

```text
access token 短期、轻量，用于高频接口鉴权；refresh token 长期、可撤销，存 Redis 便于 TTL 和服务端控制。
这样既减少每次请求查库，又保留了登录态管理能力。
```

### 11.5 如果 Redis 挂了会怎样

回答思路：

```text
access token 的本地 JWT 校验理论上可以继续工作一段时间，但 refresh token、撤销状态和部分会话状态会受影响。
当前 MVP 还没有完整 Redis 降级方案，生产环境需要 Redis 高可用、超时控制和故障降级策略。
```

## 12. 一句话总结

注册登录链路可以总结为：

```text
MyChat 用 Gateway 统一处理认证接入和 token 生命周期，用 User Service 管账号与资料，
用 Redis 保存 refresh token 等临时状态，用 PostgreSQL/ODB 保存用户持久数据，
并且所有后续业务操作都以 token 中解析出的 uid 作为可信身份来源。
```

## 附录：源码导读

这一节用于把上面的链路说明落到真实代码。阅读时建议按“入口 -> Gateway -> facade -> service -> repository/token 副作用”的顺序走。

### A. 源码阅读地图

| 层级 | 文件 | 重点函数/对象 |

| token 管理 | `gateway/auth/multi_platform_auth.cpp` | `generate_tokens`、`generate_access_token`、`generate_refresh_token`、`verify_access_token` |
| facade local | `gateway/http/user_client.cpp` | `LocalUserClient::*` |
| facade remote | `gateway/http/remote_user_client.cpp` | `RemoteUserClient::register_user`、`login_by_account`、`update_profile` |
| gRPC server | `services/user/user_grpc_service.cpp` | `UserGrpcService::Register`、`Login`、`GetUserInfo`、`UpdateUserInfo` |
| 领域逻辑 | `services/user/user_service.cpp` | `register_user`、`login_by_account`、`update_profile` |
| 密码哈希 | `services/user/password_hasher.cpp` | `hash`、`verify` |
| 持久化 | `services/user/user_repository.cpp` | `create`、`find_by_account`、`find_by_uid`、`update_profile` |

### B. 注册链路按代码读

从 Gateway 入口看：

```text
UserHttpController::handle_register
-> json::parse(req.body)
-> body.value("account"/"password"/"nickname"/"device_id"/"platform")
-> RegisterRequest reg_req
-> user_client_->register_user(reg_req)
-> auth_mgr_->generate_tokens(...)
-> response_body["profile/access_token/refresh_token"]
```

关键代码点：

- `handle_register` 中先解析 JSON，缺少 `account/password` 时直接 `400`。
- `reg_req.nickname = nickname.empty() ? account : nickname` 说明昵称有默认值。
- `device_id` 和 `platform` 不进 User Service，而是用于 token 生成。
- 注册成功后 Gateway 立即生成 access/refresh token，因此 Web 端注册后可直接进入登录态。

继续追到 User Service：

```text
LocalUserClient::register_user
-> UserService::register_user
-> repo_->account_exists
-> GenerateUid
-> hasher_->hash
-> User user(...)
-> repo_->create
-> to_profile
```

关键代码点：

- `GenerateUid` 使用 `RAND_bytes` 生成 UUID 风格 uid。
- `repo_->account_exists` 通过 `find_by_account` 判断唯一性。
- `hasher_->hash` 写入的是 `pbkdf2_sha256$iterations$salt$hash`，不是明文密码。
- `repo_->create` 用 ODB transaction 持久化用户。

如果是远程模式：

```text
RemoteUserClient::register_user
-> im::user::RegisterRequest
-> grpc ClientContext + apply_deadline
-> UserService::Stub::Register
-> UserGrpcService::Register
-> UserService::register_user
```

这里要注意两层错误：

- `status.ok()` 是 gRPC 传输是否成功。
- `rpc_response.base().error_code()` 是业务是否成功。

### C. 登录链路按代码读

Gateway 入口：

```text
UserHttpController::handle_login
-> json::parse
-> account/password/device_id/platform
-> user_client_->login_by_account(account, password, now_ms())
-> auth_mgr_->generate_tokens(...)
-> 返回 profile + tokens
```

User Service：

```text
UserService::login_by_account
-> repo_->find_by_account
-> hasher_->verify(password, user.password_hash())
-> repo_->update_last_login
-> to_profile
```

密码校验继续追：

```text
PasswordHasher::verify
-> 按 '$' 拆 stored_hash
-> 校验算法名 pbkdf2_sha256
-> 解析 iterations/salt/expected_hash
-> PKCS5_PBKDF2_HMAC 重新计算
-> CRYPTO_memcmp 常量时间比较
```

面试时可以直接落到这段代码：

```text
密码不是简单 hash；每个用户有 salt，有迭代次数，并使用常量时间比较。
```

### D. token 生成与 Redis 副作用

access token：

```text
MultiPlatformAuthManager::generate_access_token
-> PlatformTokenStrategy::get_platform_token_config
-> uuid_generate 生成 jti
-> jwt::create()
   .set_issuer("mychat-gateway")
   .set_subject(user_id)
   .set_audience("mychat-client")
   .set_payload_claim("username"/"device_id"/"platform"/"type")
   .sign(hs256)
```

refresh token：

```text
MultiPlatformAuthManager::generate_refresh_token
-> rt_generate()
-> 构造 rtMeta JSON
-> redis.set("refresh_token:" + rt, rtMeta.dump())
-> redis.expire("refresh_token:" + rt, actual_expire_seconds)
-> redis.sadd("user:" + user_id + ":refresh_tokens", rt)
-> redis.expire("user:" + user_id + ":refresh_tokens", actual_expire_seconds)
```

这里体现 Redis 的职责：

- refresh token 元数据可过期。
- 用户 token 集合也可过期。
- token 状态不是写 PostgreSQL。

### E. token 校验与可信身份

典型入口：

```text
UserHttpController::handle_profile
UserHttpController::handle_update_profile
其他 controller 的 handle_* 鉴权段
```

统一代码模式：

```text
extract_bearer_token(req)
-> auth_mgr_->verify_access_token(token, user_info)
-> user_info.user_id
```

`verify_access_token` 内部：

```text
jwt::verify()
  .allow_algorithm(hs256)
  .with_issuer("mychat-gateway")
  .with_audience("mychat-client")
-> jwt::decode(access_token)
-> verifier.verify(decoded)
-> is_token_revoked(access_token)
-> user_info.user_id = decoded.get_subject()
-> user_info.device_id/platform/username
```

因此后续业务代码应以 `user_info.user_id` 为准。比如资料更新：

```text
UpdateProfileRequest update;
update.uid = user_info.user_id;
```

这就是“不信任客户端 uid”的代码落点。

### F. 资料更新按代码读

Gateway：

```text
UserHttpController::handle_update_profile
-> verify_access_token
-> current = user_client_->get_profile_by_uid(user_info.user_id)
-> update.uid = user_info.user_id
-> update.nickname/avatar/signature/gender
-> user_client_->update_profile(update)
```

Service：

```text
UserService::update_profile
-> 校验 uid 非空
-> 校验 nickname 非空
-> repo_->update_profile(uid, nickname, avatar, gender, signature)
```

Repository：

```text
UserRepository::update_profile
-> db_->load<User>(uid)
-> user->nickname/avatar/gender/signature
-> db_->update(*user)
-> commit
```

代码边界：

- 当前头像就是 `avatar` 文本字段。
- URL/base64 data URL 都是前端和字段层面的 MVP 方案。
- 没有对象存储上传服务。

### G. 读代码时要注意的边界

- `refresh_access_token`、`revoke_refresh_token` 等能力在 auth manager 中有实现，但当前 Gateway HTTP 暴露的主流程主要是注册/登录返回 token、业务接口校验 access token。
- `RegisterRequest` proto 支持 phone/email，但当前 Web/Gateway 主路径使用 account/password。
- `UserInfo` proto 字段比较丰富，当前 service/profile 实际使用的是 uid、account、nickname、avatar、gender、signature、create_time、last_login、online 这组 MVP 字段。
- Redis 初始化在 `MultiPlatformAuthManager` 构造中触发；Redis 不健康会导致 auth manager 初始化失败。
