# 注册登录与鉴权链路分析

日期：2026-06-23

## 1. 文档目标

这份文档按当前代码解释 MyChat 的注册、登录、token 鉴权和用户资料链路。

当前权威链路不是旧的 `UserHttpController -> UserClient`，而是：

```text
HTTP
-> Gateway MessageParser
-> UnifiedMessage
-> MessageProcessor
-> user_command_handlers.cpp
-> UserPacketRequest
-> local UserPacketDispatcher 或 remote UserService::ForwardPacket
-> UserPacketResponse
-> Gateway 注入 access_token / refresh_token
```

核心职责：

| 模块 | 当前职责 |
| --- | --- |
| Web Client | 收集账号密码、保存 token、请求 Gateway |
| Gateway | HTTP 入口、route -> cmd_id、鉴权、可信 uid 注入、token 签发 |
| User command handler | 构造 `UserPacketRequest`，选择 local dispatcher 或 remote adapter |
| UserPacketDispatcher | 解析 JSON payload、调用 UserService、构造 JSON payload/HTTP status/auth hint |
| UserService | 注册、登录校验、资料查询、资料更新、用户搜索 |
| UserRepository | PostgreSQL/ODB 数据访问 |
| MultiPlatformAuthManager | JWT access token、Redis refresh token、多端登录态 |
| Redis | refresh token、撤销状态、token 索引 |
| PostgreSQL | 用户账号、密码哈希、资料信息 |

## 2. 注册链路

用户请求：

```text
POST /api/v1/auth/register
Content-Type: application/json

{
  "account": "alice",
  "password": "123456",
  "nickname": "Alice",
  "device_id": "web-device",
  "platform": "web"
}
```

服务端链路：

```text
GatewayServer::init_http_server catch-all
-> MessageParser::parse_http_request_enhanced
-> RouterManager::parse_http_route -> CMD_REGISTER
-> MessageProcessor::process_message
-> MessageProcessor::process_message_sync
-> register_user_command_handlers 中的 CMD_REGISTER handler
-> handle_user_packet_http
-> make_user_packet
-> UserPacketDispatcher::handle_register 或 RemoteUserServiceAdapter::forward
-> UserService::register_user
-> UserRepository / PostgreSQL
-> UserPacketResponse(auth_token_hint.required=true)
-> append_tokens_and_respond
-> MultiPlatformAuthManager::generate_tokens
-> HTTP response
```

关键点：

- 注册不要求已有 access token。
- Gateway 不解析 account/password/nickname，这些字段由 `UserPacketDispatcher::handle_register` 解析。
- User service 完成账号重复检查、密码哈希、用户落库。
- User service 不直接签发 token，只返回 `UserAuthTokenHint`。
- Gateway 基于 `auth_token_hint` 生成 access/refresh token，并注入 JSON 响应。

## 3. 登录链路

用户请求：

```text
POST /api/v1/auth/login
Content-Type: application/json

{
  "account": "alice",
  "password": "123456",
  "device_id": "web-device",
  "platform": "web"
}
```

服务端链路：

```text
HTTP -> MessageParser -> CMD_LOGIN
-> user_command_handlers.cpp
-> UserPacketRequest(application/json)
-> UserPacketDispatcher::handle_login 或 remote ForwardPacket
-> UserService::login_by_account
-> UserRepository::find_by_account
-> PasswordHasher::verify
-> UserRepository::update_last_login
-> UserPacketResponse(auth_token_hint.required=true)
-> Gateway generate_tokens
-> 返回 profile + access_token + refresh_token
```

登录失败场景：

| 场景 | 处理位置 | HTTP 表现 |
| --- | --- | --- |
| JSON 非法 | UserPacketDispatcher | 400 |
| 缺少 account/password | UserPacketDispatcher/UserService | 400 |
| 账号不存在 | UserService | 401 |
| 密码错误 | UserService | 401 |
| 数据库异常 | UserService/Repository | 500 |
| Gateway 未初始化 User runtime | Gateway handler | 500 |

面试表达：

```text
登录链路里账号密码属于用户业务 payload，Gateway 不解析；
Gateway 只负责把 JSON 原样装进 UserPacketRequest，然后由 User service 解析和校验。
登录成功后，User service 返回 auth hint，Gateway 作为鉴权边界统一签发 token。
```

## 4. 密码安全

当前密码处理在 User service 内完成：

- 使用 PBKDF2-HMAC-SHA256。
- 每个用户独立 salt。
- 存储格式包含算法、迭代次数、salt、hash。
- 校验时使用相同参数重新计算 hash。
- 比较使用常量时间比较，避免简单时序泄露。

这体现了一个清晰边界：

```text
密码校验是账号业务规则，属于 User service；
token 签发和 token 校验是接入层鉴权规则，属于 Gateway。
```

## 5. Access Token 与 Refresh Token

### 5.1 Access Token

access token 是短期 JWT，用于高频业务接口鉴权。

关键字段：

- uid。
- account。
- device_id。
- platform。
- token type。
- issuer。
- audience。
- expires_at。
- jti。

Gateway 后续业务接口通过 `verify_access_token` 得到 `UserTokenInfo`，再把可信身份写入 packet header。

### 5.2 Refresh Token

refresh token 是随机字符串，元数据写入 Redis。

Redis 元数据包含：

- user_id。
- username/account。
- device_id。
- platform。
- create_time。
- expire_time。
- jti。
- revoked。
- last_used。

设计目的：

- access token 保持短期、轻量、本地可验。
- refresh token 生命周期更长，需要服务端可撤销。
- 多端登录态和撤销状态适合放 Redis。

## 6. 鉴权链路

注册/登录以外的 user 命令需要 access token。

Gateway 鉴权链路：

```text
Authorization: Bearer <access_token>
-> verify_http_user
-> MultiPlatformAuthManager::verify_access_token
-> UserTokenInfo(uid/device/platform)
-> packet.header.from_uid = token uid
-> packet.header.device_id = token device_id
-> packet.header.platform = token platform
-> service packet dispatcher
```

为什么不能信任客户端传 uid：

- 客户端 body/header 都可以伪造。
- 只有 token 经过 Gateway 验签。
- 后端写操作必须使用 token 派生身份。

面试表达：

```text
MyChat 的写操作不信任客户端传入的 uid。
Gateway 从 access token 里解析出可信 uid，再写入 packet header，
业务 service 只使用这个 header 身份作为当前操作者。
```

## 7. 资料查询、资料更新、用户搜索

当前命令：

| 功能 | cmd_id | Gateway handler | Service dispatcher |
| --- | --- | --- | --- |
| 注册 | `CMD_REGISTER` | `user_command_handlers.cpp` | `UserPacketDispatcher::handle_register` |
| 登录 | `CMD_LOGIN` | `user_command_handlers.cpp` | `UserPacketDispatcher::handle_login` |
| 用户资料 | `CMD_GET_USER_INFO` | `user_command_handlers.cpp` | `UserPacketDispatcher::handle_profile` |
| 更新资料 | `CMD_UPDATE_USER_INFO` | `user_command_handlers.cpp` | `UserPacketDispatcher::handle_update_profile` |
| 搜索用户 | `CMD_SEARCH_USER` | `user_command_handlers.cpp` | `UserPacketDispatcher::handle_search_user` |

其中：

- Gateway 负责鉴权和 header 身份注入。
- UserPacketDispatcher 负责解析 JSON 中的 account/uid/keyword/nickname/avatar/gender/signature 等字段。
- UserService 负责真正的资料查询、资料更新和搜索规则。

## 8. Local / Remote 差异

local 模式：

```text
user_command_handlers.cpp
-> im::service::user::UserPacketDispatcher dispatcher(user_service)
-> dispatcher.handle(packet)
```

remote 模式：

```text
user_command_handlers.cpp
-> remote_user_adapter->forward(packet)
-> UserService::ForwardPacket(UserPacketRequest)
-> user_server
-> UserPacketDispatcher::handle(packet)
```

外部 HTTP 协议不变，Gateway handler 不关心下游到底是本地对象还是远程进程。

当前不要再用旧说法 `LocalUserClient / RemoteUserClient` 来解释主链路。它们已经不是当前代码中的主抽象。

## 10. 面试追问

### Q: 为什么 token 由 Gateway 生成，而不是 User service 生成？

因为 token 是接入层能力。User service 只证明账号密码校验成功，并返回 `auth_token_hint`。Gateway 统一签发 token，可以保证所有业务入口使用同一套多端 token 策略、Redis refresh token 管理和 access token 校验逻辑。

### Q: Gateway 为什么还要解析 User service 返回的 JSON？

当前仅在登录/注册成功后，为响应体注入 `access_token` 和 `refresh_token`。这是鉴权边界例外，不应扩展为 Gateway 解析用户业务字段。长期可以进一步把 token 字段作为 packet response 的边界字段，减少 JSON parse/dump。

### Q: local 和 remote 怎么保证行为一致？

两者都走同一个 `UserPacketRequest/UserPacketResponse` 契约。local 直接调用 `UserPacketDispatcher::handle`，remote 通过 gRPC `ForwardPacket` 到远端服务后再调用同一个 dispatcher，因此业务解析和规则复用。
