# 后端接口与服务测试清单

日期：2026-06-22

## 1. 目标

这份文档用于把后端验证拆成可执行的接口和服务层检查。

重点不是追求测试数量，而是确认每条核心链路至少有一类验证：

- HTTP 接口验证。
- WebSocket 协议验证。
- gRPC remote 模式验证。
- 数据库状态验证。
- Redis token/session 状态验证。
- 压测基线验证。

## 2. 编译与基础测试

推荐先编译核心目标：

```bash
cmake --build build/remote-push-odb --target gateway_server -j2
```

HTTP 相关测试目标：

```bash
cmake --build build/remote-push-odb --target \
  test_gateway_user_http \
  test_gateway_message_http \
  test_gateway_friend_http \
  test_gateway_group_http \
  test_gateway_group_message_http \
  -j2
```

注意：部分测试共用数据库，遇到并行死锁时优先顺序执行，不要直接判断业务代码有问题。

## 4. WebSocket 验证

核心命令：

- `CMD_SEND_MESSAGE`
- `CMD_HEARTBEAT`
- 服务端通知 / ack frame

验证点：

- 建连时 token 有效。
- 发送消息时 device_id 与 token 上下文一致。
- 客户端传入 sender_uid 不被信任。
- ack 表示服务端已处理，不等于对方已读。
- 对方在线时收到 push。
- 对方离线时消息仍落库，后续可拉取。

压测工具：

```bash
test/benchmark/bench_ws
```

当前可引用结果：

| 场景 | 结果 |
| --- | --- |
| 200 WSS 建连 | 失败 0，connect p95 约 15ms |
| 200 用户 / 1s | RTT p95 12.60ms，错误 0 |
| 200 用户 / 500ms | RTT p95 21.92ms，错误 0 |
| 200 用户 / 250ms | RTT p95 2628.05ms，错误 1，进入极限区 |

## 5. gRPC remote 模式验证

remote-all 模式下至少确认：

- Gateway 能连到 User gRPC。
- Gateway 能连到 Message gRPC。
- Gateway 能连到 Friend gRPC。
- Gateway 能连到 Group gRPC。
- Gateway 能调用 remote PushNotifier。
- Push 服务能反向调用 GatewayPushDeliveryService。

建议检查配置：

- `config/dev.remote-all.json`
- `config/dev.remote-push.json`
- 各服务 listen address。
- Gateway callback address 是否能被 Push 服务访问。

## 6. Redis 验证

关注状态：

- access token / refresh token。
- token revoke 状态。
- session / device / platform 状态。
- 在线连接映射。

测试动作：

- 登录后查询 token 是否写入。
- 登出后 token 是否失效。
- 同平台重复登录是否触发挤号策略。
- token 过期后前端是否回登录页。

## 7. PostgreSQL 验证

关注表：

- user
- message
- friend
- group
- group_member
- group_message

测试动作：

- 注册后 user 表新增。
- 单聊发送后 message 表新增。
- 好友申请和接受后 friend 状态正确。
- 创建群后 group 和 group_member 正确。
- 群消息发送后 group_message 正确。

## 8. 性能验证

当前压测基准来自：

```text
docs/benchmark/benchmark_report_20260622_full_retest/
```

后端性能结论：

- HTTP 接入层修复后，500 VU ramp 下 46230 请求，失败率 0%，p95 5.81ms。
- WSS 建连 200 并发稳定。
- WSS 消息在 200 用户 / 500ms 以内保持低延迟。
- 200 用户 / 250ms 是当前明显吞吐拐点，需要后续分段指标定位。

后续如果要继续定位极限性能，优先补：

- Gateway 业务线程池队列深度。
- WSS 单连接 pending write 数。
- 消息持久化耗时。
- Push 投递耗时。
- ack 写回耗时。
