# MyChat 架构追问与技术取舍

日期：2026-06-23

## 1. 文档定位

这份文档用于回答面试中的连续追问。重点不是背概念，而是说明 MyChat 在 Gateway、
服务边界、Push、存储和性能之间做了哪些具体取舍。

当前权威架构以统一 packet 链路为准：

```text
HTTP / WebSocket
-> MessageParser
-> UnifiedMessage
-> MessageProcessor
-> GatewayCommandHandlerRegistry
-> GatewayRuntimeRegistry
-> local service packet dispatcher 或 remote ForwardPacket gRPC
```

旧的 HTTP controller / local remote client facade 只能作为重构前的历史方案讲，不应再作为当前主架构。

## 2. Gateway 统一接入 vs 拆成 HTTP Gateway 和 WS Gateway

### 追问链

```text
为什么 Gateway 同时处理 HTTP 和 WebSocket？
-> Gateway 会不会太重？
-> 多 Gateway 如何扩展？
```

### 为什么合在一个 Gateway

HTTP 和 WebSocket 协议不同，但在 IM 系统中都属于客户端接入层。合在同一个 Gateway 的收益是：

- 鉴权逻辑统一。
- WebSocket session 与用户身份绑定更直接。
- Push 投递能回到同一个连接管理边界。
- 客户端只需要感知一个统一入口。

如果拆成两个入口，token 校验、用户状态、连接状态和错误模型都需要跨进程同步，MVP 阶段复杂度更高。

### Gateway 会不会太重

Gateway 重不重取决于职责是否外溢。当前 Gateway 只保留五类职责：

- 协议接入。
- 鉴权边界。
- cmd 路由。
- 连接/session 管理。
- packet 转发和响应回写。

它不做好友关系状态机、群成员权限、消息落库、用户搜索等业务逻辑。业务 payload 也不在 Gateway 展开。

### 多 Gateway 如何扩展

当前 MVP 主要验证单 Gateway。多 Gateway 需要补全：

- `uid -> gateway instance` 的全局连接索引。
- Push 根据连接索引回调对应 Gateway。
- Gateway session 上线/断线时注册与清理。
- 连接索引 TTL 和异常断连修正。

这个扩展方向和当前 Push callback 设计兼容，因为实际 WebSocket 发送仍然由连接所在 Gateway 执行。

## 3. 统一 Packet Boundary vs Controller/Facade

### 追问链

```text
为什么不保留 controller/client facade？
-> 统一 packet 会不会太抽象？
-> Gateway 完全不解析业务 payload 是否可行？
```

### 为什么移除旧 controller/facade 主链路

旧方案的问题是新增服务需要成套维护：

- HTTP controller。
- local client。
- remote client。
- DTO 转换。
- Gateway 内业务字段解析。

这会让 Gateway 逐渐变成业务聚合层。对于 IM 网关来说，Gateway 的核心价值应该是接入、鉴权、路由和连接，而不是理解每个业务服务的字段。

### 统一 packet 的收益

统一 packet 后，Gateway 的主流程稳定：

```text
读 envelope / metadata
-> 校验 token
-> 根据 cmd_id 找 endpoint
-> 转发 packet
-> 写回 response / handoff push_event
```

业务 protobuf 解析下沉到服务侧 `*_packet_dispatcher.cpp`。这样 HTTP 和 WebSocket 入口可以共享同一套服务转发逻辑，local 和 remote 也可以共享同一套 packet 契约。

### 代价

统一 packet 的代价是：

- cmd_id 规划必须稳定。
- packet envelope 要设计清楚。
- 服务侧 dispatcher 需要维护业务 payload 解析。
- 客户端和服务端都要遵守统一错误模型。

但这些代价比 Gateway 到处散落业务 DTO 更可控，也更符合分布式服务边界。

## 4. local 模式 vs remote gRPC 模式

### 追问链

```text
为什么还保留 local 模式？
-> remote 模式如何做到外部接口不变？
-> 为什么用 ForwardPacket 而不是每个接口都让 Gateway 调业务 RPC？
```

### local 模式的意义

local 模式用于快速开发和单机验证。服务逻辑仍然在各自模块中，只是通过进程内 packet dispatcher 调用，不代表边界消失。

### remote 模式的意义

remote 模式用于验证真实分布式部署。GatewayRuntimeRegistry 选择 remote endpoint 后，通过服务的 ForwardPacket gRPC 把同一份 packet 发给远程服务。服务端仍然用自己的 packet dispatcher 解析业务 payload。

### 为什么是 ForwardPacket

如果 Gateway 直接调用每个业务 RPC，它仍然要理解业务请求类型。ForwardPacket 的价值是让 Gateway 与服务之间保持统一 envelope 边界：

- Gateway 知道 cmd_id。
- Gateway 不知道业务 DTO。
- 服务知道自己支持哪些 cmd。
- 新增业务接口主要扩展服务 dispatcher 和 cmd 注册。

## 5. 先持久化再推送 vs 先推送再持久化

### 当前取舍

MyChat 选择先持久化再推送，因为 IM 中“发送成功后刷新消失”比“实时推送稍慢”更不可接受。

```text
客户端发送
-> 服务持久化
-> 返回 ack
-> Gateway handoff push_event
-> Push best-effort 投递
```

### Push 失败怎么办

Push 失败不回滚已持久化消息。接收方可以通过历史消息或离线拉取补偿。当前语义更接近：

- 持久化成功：服务端已接收。
- Push 成功：实时性更好。
- Push 失败：退化为拉取。

### 是否需要 MQ

生产级 IM 通常会引入消息队列，让持久化后的 push/fanout 异步化。当前 MVP 暂时直接调用 Push，是为了减少部署复杂度。后续引入 MQ 后可以把链路改成：

```text
Message/Group 服务落库
-> 写入 MQ
-> Push Worker 消费
-> Gateway callback 投递
```

收益是削峰填谷、失败重试和更低发送链路尾延迟。

## 6. Push 独立服务 vs 合在 Message/Group

Push 独立出来的理由：

- 单聊、群聊、系统通知都需要投递。
- 多端 fanout 策略可以集中管理。
- 远程部署时可以独立扩展。
- Gateway 只负责实际 session 发送，不负责投递策略。

远程 Push 的复杂点在于 Gateway callback。这个复杂度是 WebSocket session 所有权带来的：连接在 Gateway，发送就必须回到 Gateway。

## 7. Redis vs PostgreSQL

Redis 用于临时状态：

- access token 撤销。
- refresh token 元数据。
- session/在线状态。
- 可过期、可重建的数据。

PostgreSQL 用于持久业务状态：

- 用户。
- 消息。
- 好友关系。
- 群资料和群成员。

判断标准是：是否需要长期保存、事务语义、复杂查询和可靠恢复。

## 8. ODB vs 手写 SQL

ODB 的收益：

- C++ domain model 与表结构映射更自然。
- 减少 CRUD 样板。
- Repository 层代码更贴近业务模型。

ODB 的成本：

- 工具链重。
- 生成文件需要管理。
- 版本依赖更敏感。

当前项目用 CMake gate、ODB 2.5.0 构建脚本、迁移脚本和 focused tests 控制成本。

## 9. 性能数据如何讲

当前压测数据适合作为 MVP 稳定性基准，不应包装成生产极限：

- HTTP 500 VU ramp：失败率 0，p95 约 3.75ms。
- WSS 200 连接：200/200 成功，建连 p95 约 14ms。
- WSS 200 用户每 250ms 一条消息：约 714 msg/s，RTT p95 约 35.91ms。

可讲结论：

```text
重构后链路稳定性明显提升，HTTP 已经不再出现旧压测中的大面积失败。
WebSocket 在 200 并发用户下可以稳定完成在线消息收发。
后续瓶颈定位要继续拆分观测数据库写入、Push fanout、WebSocket 发送和网络环境。
```

## 10. 面试中的取舍表达模板

回答架构追问时尽量按这个结构：

```text
先说目标：这个设计解决什么问题。
再说方案：当前代码如何做。
再说代价：它带来什么复杂度。
最后说演进：生产级会怎么继续优化。
```

示例：

```text
Gateway 统一 packet boundary 的目标是避免网关理解业务 DTO。
当前通过 MessageParser、UnifiedMessage、CommandHandlerRegistry 和 RuntimeRegistry 做到
HTTP/WS/local/remote 的统一转发。
代价是 cmd_id 和 envelope 契约要设计稳定。
后续如果服务数量更多，可以把 RuntimeRegistry 与服务发现、动态配置结合起来。
```
