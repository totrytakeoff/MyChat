# MyChat 架构追问与技术取舍

日期：2026-06-17

## 1. 文档定位

这份文档补充 `19_高频面试问题与回答.md` 中不适合展开的深度追问。
解决的问题是：当面试官针对架构设计和技术选型做连续追问时，你有连贯的思考路径，
而不是零散地回答"这里这么做的"。

每个话题按"追问链"组织：面试官可能从 A 追问到 B 再追问到 C。

## 2. Gateway 统一接入 vs 独立拆分

### 追问链

```text
为什么 Gateway 要同时处理 HTTP 和 WS？
-> 那 Gateway 本身会不会太重？
-> Gateway 崩溃了怎么办？
-> 多 Gateway 怎么路由？
```

### 为什么 HTTP 和 WS 放在一起

面试官可能问"HTTP 和 WebSocket 是不同协议，为什么不拆成两个独立服务？"

```text
HTTP 和 WebSocket 虽然在协议层不同，但都属于"客户端接入"这一层职责。
拆成两个独立服务意味着：

1. 鉴权逻辑要复制两份——每增加一个接入点都要重复实现 token 校验。
2. session 管理要跨服务同步——HTTP 无 session，WS 有 session，
   但 Push 服务需要知道某个用户是否在线。
3. 客户端配置更复杂——需要配置两个地址。

合在一起的好处：
- 鉴权和 session 管理自然共享。
- Push 服务通过本进程就能触及 session。
- 客户端只关心 Gateway 一个入口。

代码上 Gateway 内部也对两者做了隔离：
HTTP 由 UserHttpController / MessageHttpController 等处理，
WebSocket 由 MessageWsHandler 处理，两者在 GatewayServer 内注册到不同端口。
```

### Gateway 会不会太重

```text
当前 Gateway 的职责边界是清晰的：它做协议接入、鉴权、连接管理和请求转发，
不做业务逻辑。业务逻辑在 Service 层。

如果 Gateway 变得太重，通常是业务逻辑泄漏到了 Gateway controller 中。
我们的做法是在 code review 中确保 controller 只做：参数解析、token 校验、
调 client facade、返回结果。业务判断、状态机、持久化都在 service 侧。

当前项目里 gateway/http/ 下的 controller 代码量都不大，职责明确。
```

### Gateway 单点故障

```text
当前 MVP 阶段是单 Gateway 实例。生产环境下多 Gateway 实例是必须的。

多 Gateway 需要解决的三个问题：
1. session 不共享：用户连 Gateway A，Push 如果只查 Gateway B 就找不到。
   需要全局连接索引（如 Redis 维护 uid -> gateway 映射）。
2. Push callback 路由：Push 服务需要先查 uid 在哪个 Gateway，再调对应 callback。
3. Gateway 无状态化：把 session 信息从进程内存搬到 Redis，让多实例共享。

当前项目没有实现这些，但 Push 的 Gateway callback 设计已经预留了
多 Gateway 场景的接口——只是缺少路由索引层。
```

## 3. local/remote 双模式 vs 直接 gRPC

### 追问链

```text
为什么不让 controller 直接调 gRPC？
-> facade 额外增加了什么复杂度？
-> 当初是不是先做 local，后加的 remote？
```

### 为什么需要 facade 层

```text
如果 controller 直接调 gRPC：

1. controller 里充满 proto 构造和解析代码。
2. 本地调试和单元测试需要启动 gRPC 服务。
3. gRPC 传输错误（超时、断开）和业务错误混在一起。
4. 切换到 local 模式要改 controller 代码。

facade 层说白了就是"面向接口编程"。UserClient 是一组纯 C++ 方法签名，
LocalUserClient 和 RemoteUserClient 分别实现。controller 只关心
UserClient 接口。

这是 GoF 策略模式的工程化应用，不是什么玄学。
```

### facade 带来的额外成本

```text
每增加一个服务接口，需要：
1. 在 Client 接口中声明。
2. 在 LocalClient 中实现（通常是透传调用）。
3. 在 RemoteClient 中实现（构造 proto、调 gRPC、映射错误）。

remote client 中还有 proto 到 service DTO 的转换，
这部分是重复的样板代码。

如果 remote 成为永久方案，可以考虑生成 pattern——通过 proto 自动生成
client 适配代码。当前手动维护模式适合 MVP 阶段。
```

### 演进过程

```text
项目确实是从 local 模式开始的——所有服务在 Gateway 进程中直接调用。
后来拆出 gRPC 边界时，没有改 controller，而是加了 Remote*Client 实现。

这就是 facade 的价值：从 local 到 remote 的演进，对 controller 零影响。
```

## 4. 先持久化再推送 vs 先推送再持久化

### 追问链

```text
为什么不是先推送再持久化？
-> 但如果 Push 失败，消息就没法实时到达了？
-> 有没有考虑过消息队列？
```

### 当前取舍

```text
先持久化再推送的核心理由是：消息可恢复性优先于实时性。

假设场景：先推送后落库，推送成功返回，但落库时数据库故障。
用户看到了消息，刷新后消息消失——这是不可接受的。

先落库后推送：落库成功后返回 ack，数据库故障此时已经不会导致消息丢失。
推送失败只是实时性下降，消息最终通过离线拉取可见。

这个取舍使"发送成功"的语义更可靠，代价是推送延迟有所增加。
```

### Push 失败的影响

```text
Push 失败时用户不会丢失消息，但收取方式退化为被动拉取：

1. 用户下次进入聊天页面时，通过 HTTP 调 history 接口拿到消息。
2. 用户重新上线时，通过 offline 接口拉取未投递消息。
3. 如果 Push 是瞬时的（网络抖动），用户刷新页面后消息就会出现。

真正有问题的情况是：Push 持续失败，用户依赖刷新才能看到新消息。
这可以在后续通过 Push 重试队列和降级推送通道改善。
```

### 消息队列方案

```text
消息队列（如 Kafka / RabbitMQ）可以在生产环境进一步解耦：

当前链路：Gateway -> MessageService 持久化 -> PushNotifier -> Gateway session。

引入 MQ 后：
1. Gateway -> MessageService 持久化 -> 写入 MQ（持久化后立即返回 ack）。
2. 独立的 Push Worker 消费 MQ，执行 Push runtime。
3. 推送失败的消息可以重新入队重试。

MQ 的好处：削峰填谷、推送异步化、失败重试内置。
当前 MVP 没有引入 MQ 是因为增加了部署复杂度，暂时不值得。

面试时可以表达：你知道 MQ 方案，但当前 MVP 阶段直接调用足够，扩展开销需要评估。
```

## 5. Push 服务独立 vs 合并在 Gateway

### 追问链

```text
Push 为什么独立成服务？
-> 独立后有没有增加复杂度？
-> 单聊 push 和群聊 fanout 为什么用同一个 PushRuntime？
```

### 独立 Push 的理由

```text
独立的原因不是简单的"职责拆分"，而是有三个工程驱动：

1. Fanout 逻辑集中：群聊消息、系统通知都涉及推送，集中到 PushRuntime
   避免每个业务模块重复实现 session 查询、payload 构造、delivered 标记。

2. 推送策略统一：AllSessionsFanoutPolicy、PlatformFilterFanoutPolicy、
   NewestSessionFanoutPolicy 三种策略可以在 Runtime 层统一配置，
   不需要每个调用侧单独实现。

3. 分布式演进：远程 Push 服务可以在独立进程中扩展，不需要 Gateway
   关心 Push 的部署方式。

代码入口：services/push/push_runtime.cpp。
```

### 独立 Push 增加的复杂度

```text
主要增加了远程模式下的回调复杂度：

远程模式：
1. Gateway 不直接调 PushRuntime，而是通过 RemotePushNotifier 调 push_server。
2. push_server 的 PushRuntime 需要查询 Gateway 的 session 才能发送。
3. 所以 push_server 作为 gRPC 客户端调 Gateway 的 GatewayPushDeliveryService。

这个 callback 是增加的复杂度。但好处是让 PushRuntime 可以在远程模式下
保持与本地模式相同的 fanout 逻辑，只有"如何发送"的实现不同。

本地模式：调 process-internal PushService。
远程模式：RemotePushNotifier -> push_server -> Gateway callback。

回调绕不绕？绕。但这是分布式边界的本质——连接在哪儿，发送就得在哪儿。
```

### PushRuntime 统一处理单聊和群聊

```text
单聊 push 和群聊 fanout 底层都是"需要通知某用户"。
PushRuntime::notify_user 的 context 中通过 conversation_type 区分。

PushContext 结构：
- sender_uid：谁发的
- conversation_type："direct" 或 "group"
- conversation_id：会话 ID

PushRuntime 构造 payload 时把 context 信息编入推送消息，
客户端收到 CMD_PUSH_MESSAGE 后根据 conversation_type 展示到对应区域。

统一的 PushRuntime 避免了两套推送逻辑，fanout policy 也能统一作用于所有推送场景。
```

## 6. 存储选型：Redis vs PostgreSQL vs ODB

### 追问链

```text
为什么不用 Redis 存消息？
-> 为什么不用 MongoDB？
-> ODB 为什么不用 SQLAlchemy 那种方案？
```

### Redis 不存消息

```text
Redis 是内存数据库，存消息有两个问题：

1. 容量：消息会持续增长，内存容量有限，不适合做全量存储。
2. 持久化弱：Redis RDB/AOF 虽然可以落地，但不是为业务查询设计的。
   需要按会话查历史、按时间排序、按状态筛选——这些是 SQL 的强项。

Redis 在我们的项目中只放临时状态：token、session 索引、撤销列表。
这些数据有 TTL，丢失后用户重新登录即可恢复。
```

### 为什么不是 MongoDB

```text
MongoDB 在 IM 场景也有应用（特别是消息存储），但当前项目选 PostgreSQL 的原因是：

1. 项目当前的业务数据（用户、好友、群组）是结构化关系数据，
   PostgreSQL 的关系模型和事务能力天然适合。
2. 消息数据虽然有文档特点（按会话聚合），但 MVP 阶段消息查询模式简单
   （按 receiver 拉取、按时间排序），PostgreSQL 完全够用。
3. 多了一种数据库就多一套运维和依赖。MVP 阶段优先减少基础设施复杂度。

如果后续消息量级很大（数亿条），可以考虑消息专用存储或分库分表方案。
PostgreSQL 的分区表也是可选项。
```

### ODB vs 其他 ORM

```text
C++ 的 ORM 选择本身就不多。ODB 的优势是：

1. 编译期代码生成，运行时无反射开销。
2. 支持多数据库后端（PostgreSQL、MySQL、SQLite）。
3. 注解直接在 C++ 头文件里，模型定义和业务代码在一起。

和 Python 的 SQLAlchemy 没法比——C++ 没有运行时反射，ORM 设计必然不同。

ODB 的痛点：
1. 工具链复杂，编译器版本要匹配。
2. 生成代码量大，编译慢。
3. 复杂查询还是得手写 SQL。

如果重来，可能考虑 socia 或直接手写 SQL + 连接池。
当前 ODB 方案适合 MVP，但不一定是生产最优解。
```

## 7. Token 认证体系设计

### 追问链

```text
为什么用双 token（access + refresh）？
-> 为什么 access token 用 JWT？
-> JWT 有撤销问题，你们怎么解决的？

```

### 双 token 设计

```text
双 token 的核心考虑是安全性和体验的平衡：

access token：短期（Web 端 1 小时），每次请求携带鉴权。
refresh token：长期（7 天），仅用于获取新的 access token。

access token 如果泄漏，有效期短，危害范围有限。
refresh token 存储在 Redis，可以主动撤销，客户端需要 refresh 时检查。

平台差异化策略：通过 PlatformTokenStrategy 读取 JSON 配置，
不同平台可以设置不同的过期时间和多设备策略。
```

### JWT 的撤销处理

```text
JWT 本身无法撤销——签发后就有效，除非等到过期。

我们的做法是维护"撤销列表"（放在 Redis）：
1. 用户登出或管理员踢人时，token 的 jti 被加入 Redis 撤销集合。
2. verify_access_token 时先检查 JWT 签名是否有效，
   再检查 token 是否在撤销列表中。
3. 撤销列表的 key 设置了 TTL 清理，避免无限增长。

这本质上是用 Redis 存储了额外的撤销状态，
好在 token 数量是可管理的（每个用户最多若干设备 × 各一个 access token）。
```

## 8. 测试与质量保障

### 追问链

```text
项目怎么测的？
-> 单元测试够吗？
-> 怎么测 WebSocket 推送？
```

### 当前测试策略

```text
三层测试：

1. 单元测试（GoogleTest）：覆盖 service 业务逻辑层。
   比如测试 FriendService 的重复申请拒绝、自加好友拒绝等边界。

2. Gateway 组件测试：测试 HTTP controller 在 mock client 下的行为。
   remote client 的 proto 转换逻辑通过 mock gRPC stub 验证。

3. Web 手工端到端测试：开两个浏览器窗口，模拟两个用户走完整链路。

代码在 test/ 目录。
```

### 测试覆盖的不足

```text
当前没有：

1. WebSocket 自动化测试——需要维护长连接状态和异步消息匹配。
2. 并发竞态测试——多个客户端同时请求的冲突场景。
3. 压力测试和稳定性测试。
4. 服务容错测试（kill -9 其中一个服务进程）。

后续建议：
1. 对消息链路加 WebSocket 集成测试，验证先落库再推送的原子性。
2. 用 Docker Compose 编排 remote-all 模式后，加服务级故障注入测试。
3. 将现有 k6 / bench_ws 压测输出结构化，形成固定性能门槛。
4. 给 WSS 极限场景补分段耗时和队列深度，定位 200 用户 / 250ms 下的 RTT 拐点。
```

## 9. 项目演进的思考

### 追问链

```text
这个项目如果做到生产级，最大的改动是什么？
```

```text
如果从当前 MVP 演进到生产级，最大改动不是新增功能，而是基础设施：

1. 连接管理：从单进程内存 session 改为 Redis 共享 session +
   全局连接路由。这是架构层面的变化。

2. 服务治理：从配置静态 endpoint 改为服务发现（etcd / consul / K8s），
   Gateway 动态感知后端服务地址变化。

3. 消息可靠性：从 best-effort push 改为 MQ 削峰 + 重试队列 +
   退避推送 + 推送回执确认。

4. 可观测性：trace id 串联 Gateway -> Service -> DB 调用链。
   业务指标（消息 TTL、推送成功率、在线率）暴露到 Prometheus。

5. 基础设施：Docker Compose -> K8s，日志 -> ELK，配置 -> 配置中心。

这些改动单个都不复杂，但整合起来需要系统化推进。
当前 MVP 的价值是证明了架构方向是正确的——Gateway 接入、服务边界、
Push callback 这些设计在面向生产演进时是扩展而不是重构。
```
