# MyChat 项目总结分析与面试准备总规划

日期：2026-06-15

## 1. 阶段目标

MyChat 当前已经进入“完整 MVP 审视 + 简历面试准备”阶段。后续工作的核心不再是无限补功能，而是把已经完成的分布式 IM 后端项目梳理成一套可以运行、可以演示、可以讲清楚、可以经受面试追问的工程作品。

本阶段目标分为四类：

- 项目总结：梳理 MyChat 的整体架构、模块职责、技术栈、核心链路和现有完成度。
- 技术分析：把 Gateway、gRPC 服务边界、Redis、PostgreSQL/ODB、WebSocket、Push、Web 验证客户端等关键技术点讲清楚。
- 测试验证：围绕现有 MVP 做单机验证、多窗口验证，后续再做多机器分布式验证。
- 面试准备：沉淀简历表达、项目亮点、技术难点、取舍理由、常见追问和回答思路。

所有本阶段新增总结文档统一放在：

```text
docs/final_sum_docs
```

## 2. 当前项目定位

MyChat 可以定位为：

```text
一个基于 C++ 的分布式 IM 后端项目，包含 Gateway HTTP/WebSocket 接入层、
User/Message/Friend/Group/Push 等服务边界、gRPC 内部远程调用、
Redis 会话与 token 状态、PostgreSQL/ODB 持久化、实时消息推送，
并配套 React Web 客户端用于完整业务流验证。
```

当前适合放到简历上的重点不是“做了一个聊天界面”，而是：

- 用 C++ 实现 IM 后端核心链路。
- 从单体本地调用逐步演进到可远程部署的 gRPC 服务边界。
- Gateway 对外稳定暴露 HTTP/WebSocket，对内通过 local/remote facade 屏蔽服务部署方式。
- 消息先持久化再推送，支持在线推送和离线拉取。
- Push 服务抽象了 fanout 策略，并通过 Gateway callback 处理 Gateway 持有 WebSocket 连接的问题。
- 使用 Redis 处理 token/session 等高频临时状态，使用 PostgreSQL/ODB 处理用户、消息、好友、群组等持久业务状态。
- 有可运行的 Web 验证客户端和一组后端测试/脚本支撑项目演示。

## 3. 当前 MVP 能力边界

### 3.1 已完成能力

账号与个人资料：

- 注册。
- 登录。
- token 鉴权。
- 当前用户信息查询。
- 昵称、签名、性别、头像更新。
- 头像当前以 URL 或 base64 data URL 存入头像文本字段。

单聊：

- HTTP 发送消息。
- WebSocket 发送消息与 ack。
- 单聊历史查询。
- 离线消息拉取。
- 在线用户实时推送。

好友：

- 用户搜索。
- 好友申请。
- 待处理申请列表。
- 接受/拒绝好友申请。
- 好友列表。

群聊：

- 创建群。
- 搜索群。
- 查看群资料。
- 加入群。
- 退出群。
- 群成员列表。
- 群消息发送。
- 群消息历史。
- 群消息在线 fanout。

分布式服务边界：

- User Service gRPC 边界与独立进程。
- Message Service gRPC 边界与独立进程。
- Friend Service gRPC 边界与独立进程。
- Group Service gRPC 边界与独立进程。
- Push Service gRPC 边界与独立进程。
- Gateway local/remote facade。
- remote-all 本地运行脚本。

Web 验证客户端：

- 中文 IM 风格登录/注册页面。
- 聊天、联系人、群聊、设置等主流程。
- 两账号多窗口验证。
- 好友搜索后申请。
- 群搜索后加入。
- 消息红点提示。
- 在线推送验证。
- 个人资料和群资料页面。
- 调试面板保留后端验证信息，但不干扰正常 IM 主流程。

- Hosted CI 恢复。
- 压测与容量评估。

## 6. 总体梳理路线

建议按下面这条路线推进：

```text
项目定位
-> 总体架构
-> Gateway 统一接入
-> local/remote 服务切换
-> 模块职责
-> 核心业务链路
-> gRPC 分布式边界
-> 数据与缓存
-> Push 实时推送
-> Web 验证客户端
-> 单机测试
-> 多机测试
-> 简历表达
-> 面试问答
```

每一步都要回答三个问题：

- 用户视角：这个功能解决什么问题？
- 工程视角：代码里怎么实现？
- 面试视角：这里有什么值得讲的技术点和取舍？

## 7. 面试重点技术点

后续需要重点准备以下问题。

架构设计：

- 为什么采用 Gateway 统一接入？
- Gateway 为什么同时处理 HTTP 和 WebSocket？
- 为什么 Web 客户端只访问 Gateway？
- 如何从本地服务调用演进到 gRPC 远程服务？
- local/remote facade 解决了什么工程问题？

消息系统：

- 消息为什么要先落库再推送？
- 在线推送失败怎么办？
- 离线消息如何处理？
- 如何防止客户端伪造 sender_uid？
- 群消息 fanout 怎么做？

Push 服务：

- PushRuntime 和 Gateway PushService 为什么拆开？
- 为什么远程 Push 需要 Gateway callback？
- 多 Gateway 实例下怎么找到用户连接在哪台机器？
- fanout policy 有什么扩展空间？

存储与缓存：

- Redis 存什么，PostgreSQL 存什么？
- token/session 为什么适合 Redis？
- ODB 的优点和问题是什么？
- schema migration 为什么不放进服务启动里自动执行？

分布式与工程化：

- gRPC proto 如何管理？
- 服务接口如何保持兼容？
- 如何做多机器部署？
- 如何处理服务不可用？
- 当前测试覆盖了哪些层次？

项目边界：

- 当前项目离生产 IM 还差什么？
- 为什么没有立刻做富媒体、撤回、群管理？
- 如果继续优化，优先级怎么排？

## 8. 当前结论

MyChat 当前已经具备作为简历项目的基本完整度：

- 有明确业务场景：分布式 IM。
- 有核心用户功能：账号、好友、单聊、群聊、实时推送。
- 有后端工程深度：C++、Gateway、gRPC、Redis、PostgreSQL/ODB、WebSocket。
- 有分布式演进价值：local/remote facade、独立服务进程、remote-all 运行模式。
- 有验证入口：Web 客户端和测试脚本。

接下来最重要的是把这些能力讲清楚、测明白、包装准确。后续不应该把生产级 IM 的所有扩展都当成当前阶段阻塞项，而应该把它们整理为“项目不足与后续优化规划”，在面试中主动说明。
