# 多机器分布式部署验证方案

日期：2026-06-22

## 1. 目标

MyChat 当前已经具备 remote-all 服务形态，但面试和演示阶段需要证明它不是“只在单进程里跑得通”。

多机器验证目标：

- Gateway 与后端服务可跨机器通信。
- User / Message / Friend / Group / Push 可独立进程部署。
- Push 服务能通过 Gateway callback 投递在线消息。
- Redis / PostgreSQL 可作为共享基础设施。
- Web 客户端不感知后端部署变化。

## 2. 推荐部署形态

最小两机器：

| 机器 | 角色 |
| --- | --- |
| Machine A | Gateway + Web Client |
| Machine B | User / Message / Friend / Group / Push Services |

更接近真实分布式的三机器：

| 机器 | 角色 |
| --- | --- |
| Machine A | Gateway |
| Machine B | User / Message / Friend / Group / Push Services |
| Machine C | Redis / PostgreSQL |

如果机器资源有限，也可以先把 Redis/PostgreSQL 与服务进程放在同一台机器上，但要在文档里说明这是 MVP 部署验证，不是生产 HA。

## 3. 配置要点

Gateway 配置：

- `user.mode=remote`
- `message.mode=remote`
- `friend.mode=remote`
- `group.mode=remote`
- `push.mode=remote`
- remote endpoint 指向服务机器 IP。
- GatewayPushDeliveryService listen address 必须能被 Push Service 访问。

服务配置：

- 各 gRPC server listen `0.0.0.0:<port>` 或明确内网 IP。
- PostgreSQL DSN 指向共享数据库。
- Redis 地址指向共享 Redis。
- 防火墙放通服务端口。

## 4. 验证顺序

### 4.1 基础网络

在 Gateway 机器上验证：

```bash
nc -vz <service-host> <user-port>
nc -vz <service-host> <message-port>
nc -vz <service-host> <friend-port>
nc -vz <service-host> <group-port>
nc -vz <service-host> <push-port>
```

在 Push 机器上验证：

```bash
nc -vz <gateway-host> <gateway-callback-port>
```

### 4.2 服务启动

先启动基础设施：

```bash
docker compose up -d redis postgres
```

再启动后端服务：

```bash
user_server
message_server
friend_server
group_server
push_server
```

最后启动 Gateway：

```bash
gateway_server -c config/dev.remote-all.json -w 10001 -H 10002
```

顺序很重要：Gateway 启动时要能初始化 remote client；Push callback 也要求 Gateway 暴露可访问地址。

## 5. 功能验证

按下面顺序跑：

1. `GET /api/v1/health`
2. 注册 A/B。
3. 登录 A/B。
4. A 搜索 B。
5. A 申请好友。
6. B 接受好友。
7. A/B 单聊在线推送。
8. B 离线，A 发送消息，B 重登后拉取。
9. A 创建群，B 搜索并加入。
10. 群聊消息在线推送。

每一步都确认 Gateway 日志能看到 remote client 调用，而不是意外回退 local。

## 6. Push callback 验证

Push remote 模式是多机器验证的重点。

链路：

```text
Gateway
-> RemotePushNotifier
-> Push Service
-> GatewayPushDeliveryService
-> WebSocket session lookup
-> send payload
```

验证方法：

- A/B 都在线。
- A 给 B 发单聊。
- Push Service 日志应出现 NotifyUser。
- Gateway callback 日志应出现 session lookup / payload send。
- B 前端无需刷新收到消息。

如果失败，优先排查：

1. Push Service 是否能访问 Gateway callback 地址。
2. Gateway 是否正确注册当前用户 session。
3. token 解析出的 uid 是否与 session 绑定一致。
4. 防火墙或安全组是否放通 callback 端口。

## 7. 性能验证口径

多机器验证不要求一上来复现完整压测，但至少要跑基础基线：

- HTTP `/api/v1/health`。
- HTTP `/api/v1/auth/info`。
- WSS 10/50/100/200 建连。
- WSS 200 用户 / 1s 消息。

当前单机/云服务器完整复测基线：

| 指标 | 当前值 |
| --- | ---: |
| HTTP 500 VU ramp | 失败率 0%，p95 5.81ms |
| WSS 200 建连 | 失败 0，connect p95 约 15ms |
| WSS 200 用户 / 1s | RTT p95 12.60ms |
| WSS 200 用户 / 500ms | RTT p95 21.92ms |
| WSS 200 用户 / 250ms | RTT p95 2628.05ms，极限区 |

多机器结果如果明显差于该基线，优先怀疑：

- 公网路径和跨机网络延迟。
- 安全组/NAT/单源 IP 限制。
- 服务端口连接池不足。
- Push callback 走公网回环导致延迟放大。

## 8. 面试表达

可以这样讲：

```text
我没有一开始就强行上 Kubernetes，而是先做了 remote-all 多进程和多机器验证。
Gateway 对外协议不变，内部通过配置把 User/Message/Friend/Group/Push 切到 gRPC。
这里最关键的是 Push，因为 WebSocket 连接归 Gateway 持有，所以远程 Push 必须 callback Gateway。
多机器验证能证明这个边界不是文档设计，而是真实可运行链路。
```
