# 故障场景与排查手册

日期：2026-06-22

## 1. 目标

这份文档用于面试和自测时解释：MyChat 出问题时怎么定位，而不是只说“看日志”。

排查原则：

1. 先确认入口：HTTP 还是 WebSocket。
2. 再确认身份：token、uid、device、platform。
3. 再确认链路：Gateway、service、Redis、PostgreSQL、Push。
4. 最后确认性能：队列、连接、线程池、持久化、推送。

## 2. HTTP 大量超时

历史问题：

修复前 HTTP ramp 在高并发下出现大量 60s timeout，失败率曾达到 56% 左右。

定位过程：

- `/health` 纯内存接口也超时，排除业务数据库和 Redis。
- 禁用客户端连接复用后延迟恢复，说明 keep-alive 空闲连接占用同步 worker。
- Gateway handler 统计显示实际业务处理耗时很低，问题发生在进入 handler 前的 HTTP 接入层。

当前修复：

- 显式配置 `httplib::ThreadPool(64, 1024)`。
- 设置 keep-alive timeout 为 1s。
- 设置 keep-alive max count 为 1。
- 增加 `/api/v1/stats` 输出 HTTP 状态码和 route handler 耗时。

修复后结果：

| 指标 | 当前值 |
| --- | ---: |
| 请求数 | 46230 |
| 失败率 | 0% |
| 平均延迟 | 3.52ms |
| p95 | 5.81ms |
| p99 | 7.85ms |
| 最大延迟 | 19.88ms |

面试表达：

```text
这个问题不是接口逻辑慢，而是同步 HTTP server 的 worker 被 keep-alive 连接长期占用。
我通过只压 /health 排除了业务链路，再通过禁用连接复用和调整 keep-alive 参数验证判断。
```

## 3. WebSocket 建连失败

排查顺序：

1. DNS / IP 是否可达。
2. TCP 是否能 connect。
3. TLS 握手是否成功。
4. WebSocket upgrade 是否成功。
5. token 是否有效。
6. Gateway accept 是否失败。

当前稳定数据：

- 200 WSS 建连。
- 连接失败 0。
- connect p95 约 15ms。
- Gateway `accept_fail=0`。

历史上 PVE 发压路径出现过固定阈值失败，但 SUT loopback 和 `lzr-host` 公网复测没有复现。因此不能简单归咎 Gateway，必须区分：

- 发压机出口限制。
- 云厂商单源 IP/NAT 策略。
- TLS/accept 排队。
- Gateway 应用层 bug。

## 4. WebSocket 消息延迟突增

当前拐点：

| 场景 | RTT p95 | 判断 |
| --- | ---: | --- |
| 200 用户 / 1s | 12.60ms | 稳定 |
| 200 用户 / 500ms | 21.92ms | 稳定 |
| 200 用户 / 250ms | 2628.05ms | 极限区，出现排队 |

已能排除：

- 不是建连失败：connect 仍稳定。
- 不是协议解析失败：`parse.decode failed=0`。
- 不是路由配置失败：`parse.routing failed=0`。

尚未精确定位：

- 业务 handler 耗时。
- PostgreSQL 持久化耗时。
- Push 投递耗时。
- session 写队列长度。
- 线程池队列深度。
- bench 客户端接收能力。

下一步必须补的指标：

- 收包到解析完成。
- 解析到业务处理完成。
- 业务处理到持久化完成。
- 持久化到 push 投递。
- push 到 session enqueue。
- session enqueue 到写完成。

不要在没有分段指标时盲改业务代码。

## 5. 登录后 401 或状态过期

排查顺序：

1. 前端是否带 `Authorization: Bearer <token>`。
2. token 是否过期。
3. Redis 中 token 状态是否被撤销。
4. device_id/platform 是否和登录时一致。
5. Gateway 是否错误地把 401 留在主页面而不是回登录页。

用户体验要求：

- 401 应直接回登录页或弹出可理解提示。
- 不应让用户理解 HTTP 状态码。
- 登录页点击登录不应提示“登录状态已过期”。

## 6. 消息需要手动刷新

排查顺序：

1. WebSocket 是否连接。
2. 当前用户 session 是否注册到 Gateway。
3. Push 是否被触发。
4. remote Push 是否 callback Gateway。
5. Gateway 是否找到目标 session。
6. 前端是否正确合并 push 消息。

如果历史消息能查到但在线收不到，说明持久化链路正常，问题集中在 Push / WebSocket。

## 7. 群消息收不到

排查顺序：

1. 发送者是否在群内。
2. 接收者是否在群内。
3. 群成员列表是否正确。
4. group message 是否落库。
5. fanout 是否遍历成员。
6. Push 是否跳过发送者或错误过滤平台。
7. Gateway 是否存在接收者 session。

当前群聊是 MVP 形态，群主转让、管理员、禁言、踢人不作为当前阻塞项。

## 8. 性能基准怎么讲

当前最适合面试使用的说法：

```text
MyChat 当前不是百万连接级商业 IM，但已经不是只能跑 demo。
HTTP 侧 500 VU ramp 下 46230 次请求失败率 0%，p95 约 5.81ms。
WebSocket 在 200 在线用户、每人 0.5-1s 一条消息时，RTT p95 约 12-22ms，
用户体感属于非常流畅。极限场景 200 用户每 250ms 一条时 p95 上升到 2.6s，
说明当前消息处理/持久化/推送链路在约 700 msg/s 入口压力附近出现排队。
```

外部参照：

- Google Core Web Vitals 把 INP 良好阈值定义为 200ms 以内，MyChat 稳定区间 WSS RTT p95 低于该体感阈值。
- Mattermost 公开压测模型中，30000 并发用户、每人 12 小时 30 条消息，入口平均约 21 msg/s，但有很高广播 fanout。
- ejabberd / Stream 这类成熟 IM 或商业服务公开过百万级连接能力，它们用于说明工业级上限，不应用来直接对标当前 MVP。
